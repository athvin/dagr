# 005 · T0.10 — Stability policy and criteria partition

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** system-level (Stability, System-level acceptance)
> **Branch:** `adr/t0.10-stability-policy-and-criteria-partition` · **Depends on:** — · **Blocks:** T7, T4, T39, T48, T65

## Why / context
dagr sells compile-time confidence and honest, replayable artifacts, so the two most expensive things to break are the authoring API and recorded artifacts. This ticket writes the project's binding stability contract — MSRV, semver, additive-only schema evolution, the fixture-corpus plan, and fingerprint-algorithm versioning — and, just as importantly, produces the single source of truth that classifies every acceptance criterion in arch.md as `machine`, `human`, or `disclaimer`. It is governed by arch.md's **Stability**, **Platform support**, **Performance envelope**, and **System-level acceptance** sections, and it locks system-level criterion 8: the criteria matrix that CI enforces. Nothing here is code; it is decisions recorded as a versioned ADR plus a checked-in criteria matrix that later tickets consume. It reshapes T7 (CI matrix scaffold), T4 (schema versioning), T39 (publish schemas), T48 (artifact compatibility CI), and T65 (system acceptance gate), which is why it must land before any of them.

## Objective
Produce a committed ADR and a checked-in criteria matrix that decide and record, unambiguously, the project's stability policy and criterion classification. Concretely:

- **MSRV:** state the pinned minimum supported Rust version, where it is pinned (workspace) and documented (README), and the rule that raising it is a minor version bump called out in release notes.
- **Semantic-versioning policy:** state the authoring-API semver contract (within a major, pipelines keep compiling), and enumerate what constitutes a major event — breaking the authoring API, replacing or major-bumping tokio, breaking recorded-artifact compatibility.
- **Schema-evolution rules:** record the additive-only-within-a-version rule for all three schemas (event stream, graph artifact, run artifact), the reader contract (ignore unknown fields, default missing ones), and how the folding function declares which stream schema versions it reads.
- **Fixture-corpus plan:** define the corpus location in-repo, its contents (one artifact per released schema version per artifact kind, plus the ten-thousand-attempt scale artifact), when it is seeded (M3) and frozen, and the standing CI obligation to parse every prior-version fixture forever after.
- **Fingerprint-algorithm versioning:** record that the structural fingerprint and policy hash each carry a versioned algorithm identifier, that cross-toolchain stability is a tested guarantee, and that an algorithm-version mismatch reads as "cannot compare" (distinct from "topology differs"), never as false difference.
- **Criteria partition:** classify every acceptance criterion in arch.md — all of C1–C28 and the eight system-level criteria — as exactly one of `machine`, `human`, or `disclaimer`, with platform-conditional criteria (limit detection, signal handling, flush behavior) named as such. Record this as the checked-in criteria matrix that seeds the T7 CI coverage matrix.

## Decision

### 1. MSRV — a single pinned Rust version

The **MSRV is Rust 1.95.0**. It is **pinned in the workspace**
(`Cargo.toml`, `[workspace.package].rust-version = "1.95.0"`, inherited by every
member crate) and **documented in the README** ("MSRV" section), and matches the
channel in `rust-toolchain.toml` verbatim; all three state the same number with
no drift. **Raising the MSRV is a minor version bump**, called out in the release
notes. A single concrete minimum is named — there is exactly one MSRV, not a
range — so a consumer, CI, and the trybuild/UI-snapshot harness (T8) all pin the
identical toolchain.

### 2. Semantic versioning — the authoring-API contract and a closed set of major events

The **authoring API follows semantic versioning**: **within a major version,
unchanged pipelines keep compiling.** A change is a **major-version event** if
and only if it is one of this **closed set**:

- **Breaking the authoring API** — any change that makes an unchanged pipeline
  stop compiling or change meaning.
- **Replacing or major-bumping tokio** — tokio is a supported public dependency
  (C13); a tokio major bump or swap is a dagr major-version event (Stability).
- **Breaking recorded-artifact compatibility** — a non-additive schema change
  (see §3) or any change that makes a prior-version artifact unreadable by
  current tooling is a major event, because breaking recorded artifacts is, per
  the Stability section, *worse* than breaking the API.

Anything outside this set (additive schema evolution, an MSRV raise, internal
refactors, a tokio minor/patch bump) is at most a minor or patch event. The list
is closed: if a proposed change is not on it, it is not a major event.

### 3. Schema evolution — additive-only, reader-tolerant, per schema

All **three schemas** — the **event stream** (C19), the **graph artifact**
(C20), and the **run artifact** (C22) — each carry a schema version, and evolve
by one rule: **within a version, only additive changes are permitted.** The
**reader contract** is: **readers ignore unknown fields and default missing
ones**, so a newer reader accepts an older artifact and an older reader accepts a
newer one within the same version. The **folding function declares which stream
schema versions it reads** (C22) — the crashed-run fold is a standalone function
and CLI verb, so it names its supported input versions explicitly rather than
guessing. **A non-additive change requires a version bump** (a new schema
version), which is a recorded-artifact-compatibility event per §2. This ADR
fixes only the versioning and additive-only rules; the concrete serialization
format, schema language, and validation crate are **T4**, and the schema
documents themselves are **T39**.

### 4. Fixture corpus — location, contents, seeding, and the forever-parse obligation

The **fixture corpus lives in-repo** under `tests/fixtures/corpus/` (a stable,
version-controlled path; the exact sub-layout by artifact kind and schema version
is T48's to populate). Its **contents**: **one frozen artifact per released schema version per artifact kind**
(event stream, graph artifact, run artifact),
**plus the ten-thousand-attempt scale artifact** the Performance envelope
requires to keep artifact/stream tooling honest at scale. It is **seeded at M3**
(when the artifacts first exist — C20/C22 land in M3) and each entry is **frozen**
once committed: a released schema version's fixture is never edited afterward. The
**standing CI obligation**, consumed by **T48**: **every prior-schema-version
fixture must remain parseable by current tooling, forever after** — the
compatibility job parses the whole corpus on every change and fails if any frozen
fixture no longer parses. This ticket records the plan and the obligation only;
generating, freezing, and wiring the corpus is **T48**.

### 5. Fingerprint-algorithm versioning — "cannot compare" is not "topology differs"

The **structural fingerprint and the policy hash each carry a versioned algorithm
identifier** (C21). **Cross-toolchain stability is a tested guarantee**: two
builds of unchanged source, on different machines and different toolchains,
produce the same fingerprint, and this is testable because every fingerprint
input is author-declared (stable names, not `std::any::type_name`). An
**algorithm-version mismatch reads as "cannot compare"** — a *distinct* outcome
from **"topology differs,"** which resume reports separately: a mismatch means the
two fingerprints were computed by different algorithms and are simply not
comparable, so it is **never surfaced as a false difference**. This ADR versions
the algorithm and fixes the "cannot compare" semantics; the fingerprint's
composition, canonicalization, and stable-name trait are **T0.7**.

### 6. Platform posture and the platform-conditional criteria

The platform posture is arch.md's **Platform support** verbatim: **tier-1 Linux
containers** (everything works; cgroup v2/v1 limit detection; the full suite
including fault-injection and signal tests runs in CI here); **dev-supported
macOS** (compiles and runs; pool sizing falls back to host resources; documented
divergences only — no cgroups, different fsync semantics; a CI job runs the core
suite); **Windows explicitly unsupported in v1** (signal and process-model
differences make untested promises dishonest; revisit on demand).

The three **platform-conditional acceptance criteria** are **named as such**:
**limit detection (C12)**, **signal handling (C16)**, and **flush/fsync behavior
(C19)**. In the matrix they carry a `platform-conditional` flag so the **T70**
platform-matrix CI job gates them per platform, rather than treating them as
unconditionally machine on every platform. They remain machine — they are tested
— but their pass/fail is platform-dependent.

### 7. Criteria partition — machine / human / disclaimer, one class per criterion

Every acceptance criterion in arch.md — **all of C1–C28 and the eight
system-level criteria** — is classified as **exactly one** of **machine**,
**human**, or **disclaimer**, recorded in the checked-in matrix at
[`docs/criteria-matrix.md`](../criteria-matrix.md). The three classes:

- **machine** — verifiable by an unambiguous automated test; T7's CI coverage
  matrix maps each to a passing test and fails CI on any unmapped machine
  criterion.
- **human** — a judgment (readability, documentation quality, a time goal) no
  test can settle; on the version-controlled release checklist.
- **disclaimer** — something the tool deliberately claims nothing about; carried
  present-but-unclassified-as-testable so it is never silently absent.

The classification honours arch.md's own bracketed labels:

- **Criterion 1** is **machine** (quickstart compiles/runs in CI) **plus human**
  (the thirty-minute walkthrough) — recorded as one row (SL1) carrying both
  parts.
- **Criterion 4** splits into **SL4a** *structural* (machine), **SL4b**
  *interpretive* (machine), and **SL4c** external-systems (**disclaimer**).
- **Criterion 8** splits into **SL8machine** (coverage-of-machine-criteria,
  machine) and **SL8human** (human-classed criteria on the release checklist,
  human).
- The **human-classed set** includes at minimum **diagram readability (C24)**,
  **documentation-at-point-of-use (C21)**, **the thirty-minute walkthrough**
  (SL1's human facet, echoed by SL8human), and **C1's types-readable-from-the-
  declaration** judgment.
- **SL4c** is the **single disclaimer**, present and marked `disclaimer`.

**Granularity decision.** The matrix lists one row per *criterion id*. For the
two system-level criteria arch.md itself splits (4 and 8), the classified ids are
the sub-parts (SL4a/SL4b/SL4c; SL8machine/SL8human), because each sub-part
carries a different class — there is no bare classified `SL4` or `SL8`. **C1 and
C21** keep a single id each: each bundles mechanical sub-criteria *plus* the one
judgment-shaped criterion arch.md criterion 8 names as human (C1's
readable-from-declaration; C21's internal-logic-limitation documented at the
point of use), so each is classed **human** with a matrix note recording that its
remaining mechanical sub-criteria are still covered by automated tests. The class
is the criterion's *governing* class per criterion 8, not a claim that nothing
about C1 or C21 is testable. This is the defensible resolution of the only
genuine judgement calls in the partition; every other criterion is unambiguously
one class.

### 8. Downstream hand-off — what each reshaped ticket consumes

Recorded explicitly so no downstream ticket re-derives a stability or
classification decision:

- **T7 (CI matrix scaffold)** consumes the **classification** (which criteria are
  machine, and so must map to a passing test; which are human, and so map to the
  release checklist) and the **platform-conditional flags** (C12/C16/C19), so its
  matrix-enforcement script knows which criteria a per-platform job gates. T7
  builds the *tooling*; this ticket supplies the *content*.
- **T4 (schema versioning) and T39 (publish schemas)** consume the
  **schema-version-field** requirement and the **additive-only-within-a-version**
  rule with the reader contract (§3), so they encode versioning and evolution
  without re-deciding the policy — T4 chooses only the format/crate, T39 only the
  documents.
- **T48 (artifact-compatibility CI)** consumes the **fixture-corpus plan** (§4) —
  the in-repo location, one-artifact-per-released-schema-version-per-kind, the
  ten-thousand-attempt artifact, M3 seeding — and the **forever-parse
  obligation** it must enforce.
- **T65 (system acceptance gate)** consumes the **machine/human/disclaimer
  partition** (§7): the gate asserts every machine criterion maps to a passing
  test and every human criterion to the release checklist, with SL4c carried as
  the disclaimer.

## Consequences

- **T7, T4, T39, T48, and T65 are unblocked with no stability or classification
  choice left open.** Each builds against §§1–8 and re-decides none of them; the
  matrix at [`docs/criteria-matrix.md`](../criteria-matrix.md) is the row-by-row
  input to T7's coverage matrix and T65's gate.
- **The criteria matrix is now the single source of truth** for every criterion's
  class. A criterion absent from it fails CI (criterion 8); adding a criterion to
  arch.md requires a matching matrix row in the same change. The totality — every
  C1–C28 and every system-level id present exactly once, each a valid class — is
  checked mechanically by `scripts/check-stability-and-criteria.sh`, authored
  before the matrix in the tests-first spirit.
- **The matrix is a checked-in, review-owned document, never a runtime
  registry.** It is edited only by pull request. It is never loaded at runtime,
  never a scheduler input, never a metadata store — that boundary is permanent
  (arch.md non-goals).
- **MSRV, semver, and schema rules are binding across the project.** A tokio
  major bump, an authoring-API break, or a non-additive schema change is a
  major-version event; an MSRV raise is a minor event announced in release notes;
  additive schema evolution stays within a version. Reviewers apply this closed
  set as written.
- **Platform-conditional criteria are visible to the platform matrix.** C12, C16,
  and C19 carry the flag, so T70 can gate them per platform instead of failing a
  macOS or (future) other-platform job on a Linux-only guarantee.
- **No code, no dependency, no crate change.** The MSRV number was read from the
  already-committed pins; `Cargo.lock` is unchanged, so `cargo audit` gains no
  surface.

## Rejected alternatives

- **A machine-readable matrix (YAML/JSON/TOML) loaded by tooling, instead of a
  reviewed Markdown table.** Rejected: arch.md criterion 8 calls the matrix a
  document that is *reviewed like code*, and the permanent non-goals forbid a
  runtime registry or metadata store. A data file invites a consumer to load it
  at runtime and grow it into a registry; a Markdown table reviewed in pull
  requests keeps the classification a human-owned decision. T7's enforcement
  script parses the table's rows — greppable, stable columns — which is all the
  machine-readability the coverage matrix needs, with none of the registry risk.
- **Classifying each acceptance-criterion *bullet* separately, rather than one
  row per criterion id.** Rejected: arch.md criterion 8 says "each criterion in
  this document appears there exactly once," and the ticket's totality test
  enumerates criterion *ids* (C1–C28, SL1–SL8 with 4's and 8's sub-parts), not
  every bullet. Per-bullet rows would multiply the matrix without changing any
  class and would fight the "exactly once per criterion id" totality check.
  Criterion-id granularity, with sub-parts only where arch.md itself splits a
  criterion into different classes (4, 8), is the granularity the spec and ticket
  name.
- **Classifying C1 and C21 as machine** (on the grounds that most of their
  sub-criteria are testable). Rejected: arch.md criterion 8 *explicitly* lists
  "documentation-at-point-of-use (C21)" and "C1's types-readable-from-the-
  declaration" among the human-classed criteria. Overriding the spec's own label
  to gain a machine mapping would misrepresent the classification the acceptance
  gate is built on; the honest record is human, with a note that the mechanical
  sub-criteria are still tested.
- **Omitting SL4c (the external-systems disclaimer) from the matrix** because it
  is neither machine nor human. Rejected: criterion 4(c) is explicit that the
  disclaimer is "carried unclassified in the criteria matrix," and criterion 8
  says a criterion *absent* from the matrix fails CI. A third class —
  `disclaimer` — carries it present-but-not-testable, which is exactly the
  spec's instruction.
- **Leaving the MSRV as a floating range or "latest stable."** Rejected: the
  Stability section requires a *pinned* MSRV, and the UI-snapshot harness (T8)
  needs a byte-reproducible toolchain. A single number (1.95.0), matched across
  the workspace pin, `rust-toolchain.toml`, and the README, is the only honest
  reading; raising it is a deliberate, release-noted minor event.
