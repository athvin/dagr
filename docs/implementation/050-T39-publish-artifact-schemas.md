# 050 · T39 — Publish artifact schemas

> **Milestone:** M3 · **Size:** M · **Type:** feature · **Components:** C19, C20, C22
> **Branch:** `feat/t39-publish-artifact-schemas` · **Depends on:** T4, T0.8, T0.10 · **Blocks:** T40, T42, T48

## Why / context
The three durable records dagr writes — the event stream (C19), the graph artifact (C20), and the run artifact (C22) — need a single, versioned, checked-in contract before anything emits or folds them. This ticket publishes those schemas so the emitters (T40 graph, T42 fold) and the compatibility CI (T48) all target one authoritative definition rather than inventing their own. It ratifies the encoding and versioning decisions from T4 (JSONL events, JSON artifacts, per-record schema-version field), carries the durable-reference field mandated by the T0.8 durable-output contract, and honors the schema-evolution and fixture-corpus rules fixed in T0.10 and the Stability section. No serialization or business logic ships here — only the schema documents and a validation helper the rest of M3 leans on.

## Objective
Publish versioned, checked-in schema definitions for all three artifact families and a shared validation helper usable from tests and CI. Concretely:

- Author the **event record** schema (C19): every record carries run identity, schema version, a gapless strictly-increasing sequence number, a wall-clock stamp (informational), and a monotonic offset from run start (authoritative). Cover every event kind — run started (carrying every run-artifact header field known at start), node became ready, node admitted, attempt started, attempt succeeded, attempt failed, attempt-outcome, node reached terminal state, **zombie-at-exit**, and run finished.
- Author the **graph artifact** schema (C20): header (schema version, tool version, generation time, pipeline identity, build provenance — tool version, git commit, lockfile hash — and both fingerprint hashes with the algorithm version), plus per-node and per-edge descriptions (name, group, stable task name, stable input/output type names, execution class, complete effective policy, declared resource requirements, dependency lists; edges tagged data-vs-ordering with the carried stable type name on data edges). Reserve an informational debug field for `type_name` and forbid it as identity.
- Author the **run artifact** schema (C22): header (run identity, pipeline identity, both fingerprint hashes, invocation parameters and data interval, allowlisted captured environment values, resume lineage — immediate parent and lineage root — and overall outcome); one record **per attempt** (node identity, attempt number, terminal status from the normative taxonomy, named phase durations, worker, message, structured error detail, reported metrics, declared-vs-measured cost, and the **durable-output reference** per T0.8); and a summary (total elapsed, critical path, peak slot residency, retained values, abandoned-work-pinned time and capacity).
- Model the required **variants and markings**: overall outcome must express distinct `assembly-failed` and `bootstrap-failed` values (with fingerprint present only when assembly succeeded, the complete error list, and zero attempts); the **single-node replay** artifact variant must permit nodes to be marked `not-requested`; attempt status must include `satisfied-from-prior` carrying an originating run identity; and durable references copied forward on resume must be representable.
- Define the **schema-version fields and evolution posture** (T0.10): each family carries its own version; within a version, evolution is additive-only and readers default missing / ignore unknown fields. Record which stream schema versions the fold reader (T42) declares it reads.
- Ship a **validation helper** (library function plus the wiring a CI step invokes) that validates a candidate artifact or stream against its published schema, for use by T40, T42, and T48.

## Test plan (write these first — TDD)
- **Event record validates.** Given a hand-authored minimal event record for each event kind (including run-started and zombie-at-exit), when it is run through the validation helper against the event schema, then every one validates and each is confirmed to carry run identity, schema version, sequence number, wall-clock stamp, and monotonic offset.
- **Run-started header completeness.** Given a run-started event fixture, when validated, then it is accepted only when it carries every run-artifact header field known at start (identity, pipeline identity, both fingerprints, parameters, data interval, allowlisted env, resume lineage) and omits the fields that exist only at end (overall outcome and summary); a fixture missing a required start-header field is rejected.
- **Sequence and offset typing.** Given an event fixture whose sequence number is negative or non-integer, or whose monotonic offset is absent, when validated, then the helper rejects it.
- **Graph artifact validates.** Given a hand-authored graph artifact with one node and one data edge and one ordering edge, when validated against the graph schema, then it is accepted, and a fixture whose data edge omits the carried stable type name, or whose header omits build provenance or a fingerprint hash, is rejected.
- **Stable-name-only identity.** Given a graph-artifact fixture that places a value in the reserved `type_name` debug field, when validated, then it is accepted as informational; given a fixture that uses `type_name` where a declared stable name is required, then it is rejected.
- **Run artifact full-run validates.** Given a run-artifact fixture with a success header, two attempt records for one node (a retry), and a summary, when validated against the run schema, then it is accepted; and a fixture whose phase-duration set or worker field is malformed is rejected.
- **Attempt taxonomy coverage.** Given attempt-record fixtures each carrying one terminal status drawn from the full normative taxonomy (including `abandoned`, `timed-out`, `upstream-skipped`, `upstream-failed`, and `satisfied-from-prior`), when validated, then all are accepted; a `satisfied-from-prior` record missing its originating run identity is rejected.
- **Durable-reference field present.** Given an attempt-record fixture that carries a durable-output reference in the T0.8 shape, when validated, then it is accepted; and a fixture that copies a durable reference forward under `satisfied-from-prior` (resume lineage) is also accepted.
- **assembly-failed variant.** Given a run-artifact fixture with overall outcome `assembly-failed`, no fingerprint, a non-empty error list, and zero attempts, when validated, then it is accepted; the same fixture bearing a fingerprint is rejected because a fingerprint is present only when assembly succeeded.
- **bootstrap-failed variant is distinct.** Given a `bootstrap-failed` fixture and an `assembly-failed` fixture, when validated, then both are accepted and the schema treats the two outcome values as distinct (a fixture that conflates them into one value fails a fixture-level assertion that the two variants are separately expressible).
- **not-requested single-node variant.** Given a single-node replay artifact fixture in which nodes outside the request carry the `not-requested` marking, when validated against the run schema's single-node variant, then it is accepted; a fixture that uses `not-requested` as a terminal state on an in-request node is rejected.
- **zombie-at-exit event.** Given a zombie-at-exit event fixture referencing an attempt, when validated against the event schema, then it is accepted and is confirmed to be an event record, not a terminal-state transition.
- **Allowlisted environment capture.** Given a run-artifact header fixture whose captured-environment map contains only allowlisted keys, when validated, then it is accepted; the schema and helper together are exercised by a negative fixture placing a non-allowlisted-shaped value, confirming the schema does not silently sanction unbounded environment capture (default allowlist empty).
- **Additive-only evolution.** Given an artifact fixture from schema version N carrying an extra field unknown to the version-N reader, when validated by the current helper, then the unknown field is tolerated (ignored) and validation passes; given a fixture missing a field added additively, the helper defaults it and still passes — proving the additive-only, ignore-unknown, default-missing posture.
- **Fixture-corpus round-trip.** Given the checked-in fixture corpus seeded by this ticket (at least one artifact per released schema version, for all three families), when the CI validation step runs, then every corpus fixture validates against its declared version's schema and the step fails loudly if any does not.
- **Invalid input rejected with a usable error.** Given a deliberately malformed artifact (missing schema-version field entirely), when passed to the validation helper, then validation fails and the returned error names the failing artifact and the reason, so tests and CI can assert on it.

## Definition of done
- [ ] The event-stream schema (C19) is published, checked in at the repo location fixed by T4, and defines run identity, a schema version, a gapless strictly-increasing sequence number, an informational wall-clock stamp, and an authoritative monotonic offset on every record.
- [ ] The event schema covers all state-transition and lifecycle event kinds, including run-started (carrying every run-artifact header field known at start), attempt-outcome, zombie-at-exit, and run-finished.
- [ ] The event schema shape supports concatenating records from concurrent runs and partitioning them safely by the carried run identity.
- [ ] The graph-artifact schema (C20) is published and describes every node (name, group, stable task name, stable input/output type names, execution class, complete effective policy, declared resource requirements, dependency lists) and every edge (data-vs-ordering, with carried stable type name on data edges).
- [ ] The graph-artifact schema carries the header (schema version, tool version, generation time, pipeline identity, build provenance — tool version, git commit, lockfile hash — and both fingerprint hashes with algorithm version), and reserves `type_name` as an informational debug field only, never as identity.
- [ ] The run-artifact schema (C22) is published with header, one record per attempt, and summary as specified, and a graph-artifact-emitted structural fingerprint is representable in the run-artifact header for cross-referencing.
- [ ] The run-artifact attempt record carries node identity, attempt number, a terminal status covering the full normative taxonomy, named phase durations, worker, message, structured error detail, reported metrics, declared-vs-measured cost, and the durable-output reference per T0.8.
- [ ] The `satisfied-from-prior` status is representable and carries an originating run identity; resumed artifacts can copy durable references forward.
- [ ] The overall-outcome value distinguishes `assembly-failed` from `bootstrap-failed`; both variants require zero attempts and a complete error list, and carry a fingerprint only when assembly succeeded.
- [ ] The single-node replay artifact variant permits nodes outside the request to be marked `not-requested` as an artifact marking, distinct from any terminal state.
- [ ] The run-artifact header captures only allowlisted environment values (default allowlist empty), and no unbounded environment capture is sanctioned by the schema.
- [ ] Each schema family carries its own schema-version field; the published rules state that within-version evolution is additive-only and that readers ignore unknown fields and default missing ones; the fold reader (T42) can declare which stream schema versions it reads.
- [ ] A validation helper (library function plus CI wiring) validates a candidate artifact or event stream against its published schema and returns an actionable error identifying the artifact and reason on failure.
- [ ] A checked-in fixture corpus with at least one artifact per released schema version for all three families is seeded, and a CI step validates every corpus fixture against its declared version's schema (per T0.10 and Stability).
- [ ] The Stability-section commitment that each of event stream, graph artifact, and run artifact carries a schema version, with a corpus parsed in CI forever after, is satisfied by the published schemas and the seeded corpus.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- Actually **emitting** the graph artifact (T40, C20) or **folding** a stream into a run artifact (T42, C22) — this ticket publishes the schemas and validation helper only; no emitter or fold logic ships.
- The **fingerprint algorithm** itself (T41, C21) and the stable-name trait (T0.7) — the schema reserves the fields and algorithm-version slots, but computing the hashes belongs elsewhere.
- The event-stream **writer** (T19), sink behavior, flush/fsync semantics, and mid-run sink-failure handling — those are runtime behaviors, not schema shape.
- **Node metrics** semantics, caps, reserved-prefix enforcement, and allocator attribution (T44, C23) — the schema treats metrics as an open numeric map; the framework's contribution rules live with C23.
- **Renderers** (T46/T47, C24), the CLI verbs (T55, C26), resume planning (T57/T58, C27), and the scale/benchmark artifact wiring (T69) beyond seeding the corpus fixtures this ticket needs.
- The full **artifact-validation-and-compatibility CI** with the ten-thousand-attempt scale artifact (T48) — this ticket provides the helper and seeds the initial corpus; the enduring compatibility gate is T48.
- Scope-boundary temptations: no metadata store or run index, no cross-run query surface, no web/UI schema, no runtime-mutable graph representation. Schemas describe a fixed graph shape and an append-only trail, nothing more.
