# 071 · T54b — C18: resume scratch carry-forward

> **Milestone:** M4 · **Size:** S · **Type:** feature · **Components:** C18, C27
> **Branch:** `feat/t54b-resume-scratch-carry-forward` · **Depends on:** T54a, T58 · **Blocks:** T59

## Why / context
A checkpoint is only half a feature if a resumed run cannot see it: a node that finished the first half of its work in the prior run must be handed that prior scratch when it re-executes in the resumed run, so it continues from its high-water mark rather than starting over. This ticket wires the scratch half of resume — the copy-forward step that C18 promises ("On resume (C27), the scratch of every node that will re-execute is copied forward from the linked prior run into the new run's namespace," arch.md `### C18 · Durable scratch store`, line 391) and that C27's fourth acceptance criterion assumes ("With a durable run store, a resumed run's re-executing nodes see the prior run's scratch values," arch.md line 399). It builds directly on T54a (prior-run non-succeeded scratch survives the process end and is readable from a fresh process) and on T58 (resume core computes which nodes re-execute — the seed/closure/demand set — and links the resumed run to its parent). This ticket adds only the transfer: for exactly the re-executing node set T58 hands it, copy each node's prior scratch into the resumed run's per-node namespace before those nodes run.

## Objective
Make a re-executing node in a resumed run observe the scratch its counterpart wrote in the linked prior run, by copying that scratch forward into the resumed run's namespace as part of resume startup — for exactly the nodes T58's algorithm marks for re-execution, and no others.

Concrete pieces of work:
- Consume the re-execution set that T58's resume algorithm produces (nodes not `satisfied-from-prior`: the must-run seed plus its downward closure plus demanded non-durable producers) and, for each such node, copy that node's scratch from the linked prior run's directory into the resumed run's per-node scratch namespace before the node is admitted.
- Read the prior scratch through the run-store base at the prior run's directory (`<base>/<pipeline>/<prior-run-id>/`, arch.md line 67) using the same per-run/per-node namespacing T53/T54a established, and write it under the resumed run's own run id so the resumed node reads its checkpoint through the ordinary C18 context API with no awareness that a copy happened.
- Copy forward only the re-executing nodes' scratch: a node marked `satisfied-from-prior` (not re-executed) has nothing copied, because it does not run and never reads scratch.
- Handle a prior node that has no retained scratch (it succeeded in the prior run, or never wrote any) as an empty carry-forward — the resumed node simply starts with an empty scratch namespace; absence of prior scratch is not an error.
- Treat a scratch read/write failure during carry-forward as a retry-eligible failure consistent with C18's I/O-failure classification (arch.md line 393), surfaced against the affected node rather than aborting the whole resume.
- Confirm cross-node isolation is preserved through the copy: each node's carried-forward scratch lands only in its own resumed namespace; no node can read another node's carried-forward values.
- Document the carry-forward behavior at the scratch/resume boundary: a resumed re-executing node continues from the prior run's checkpoint; a resumed run needs a durable (surviving) run store for the prior scratch to exist at all (arch.md lines 67, 391, 399).

## Test plan (write these first — TDD)
- **A re-executing node sees the prior run's scratch value.** Setup: run a pipeline once with a durable run store; a node writes a known key/value to scratch and then ends non-succeeded so the prior scratch is retained (T54a); the run leaves a resumable prior directory. Action: resume that run so the same node is in the re-execution set, and have the node read that key on its first resumed attempt. Expected: the node reads back the exact bytes written in the prior run — the checkpoint crossed from the prior run's namespace into the resumed run's namespace.
- **The value arrives in the resumed run's own namespace, not the prior one.** Setup: same as above. Action: after the resume, inspect the resumed run directory's scratch for that node. Expected: the carried-forward key/value is present under the resumed run id's per-node namespace; the resumed node read it through the ordinary context with no path to the prior run's directory.
- **A continued node resumes from its checkpoint, not from zero.** Setup: a node that records a high-water mark (e.g. "finished item K") to scratch, fails partway in the prior run, and on re-execution reads that mark to decide where to start. Action: resume so the node re-executes and reads its mark. Expected: the node observes the prior high-water mark and does the remaining work only — it did not restart from the beginning, which is the point of checkpoints (arch.md line 391).
- **A satisfied-from-prior node has nothing carried forward.** Setup: a prior run with a node that succeeded and is left outside the must-run set by T58 (marked `satisfied-from-prior`, arch.md line 35). Action: resume and inspect that node. Expected: the node is not re-executed and no scratch is copied for it; the resumed run's scratch namespace for that node is empty, because it never runs and never reads scratch.
- **A re-executing node with no prior scratch starts empty.** Setup: a prior run whose node ended non-succeeded but never wrote any scratch (or wrote none that was retained). Action: resume so the node re-executes and reads its scratch namespace. Expected: the node sees an empty namespace and proceeds normally — the absence of prior scratch is a clean empty start, not a resume failure.
- **Cross-node isolation survives the copy.** Setup: a prior run leaving two re-executing nodes A and B, each having written a distinct value under the same key name. Action: resume so both re-execute, and have each read that key. Expected: A reads A's carried-forward value and B reads B's; neither can read the other's; the per-run/per-node namespacing kept the two carried-forward sets disjoint (C18 isolation criterion).
- **Only re-executing nodes are copied.** Setup: a prior run with a mix — some nodes re-executing, some `satisfied-from-prior`, some that wrote scratch and some that did not. Action: resume and enumerate which nodes have carried-forward scratch in the resumed namespace. Expected: exactly the re-executing nodes that had retained prior scratch have it copied forward; satisfied nodes have none; the copy set equals the re-execution set intersected with nodes that had retained scratch.
- **Carry-forward I/O failure is retry-eligible against the node.** Setup: a resume in which reading or writing one re-executing node's scratch during carry-forward is made to fail (fault injection at the copy point). Action: run the resume. Expected: the failure is classified as a retry-eligible task failure attributed to that node (arch.md line 393), not a silent skip and not a whole-resume abort; the rest of the resumed run is unaffected by that node's isolated failure classification.
- **No durable store means no carry-forward to do (guard boundary).** Setup: a resume attempt whose prior run's store did not survive. Action: attempt the resume. Expected: resume itself refuses at T58's "store is gone" guard before any node runs (arch.md lines 67, 565) — carry-forward is never reached, confirming this ticket relies on T58's resumability gate and never fabricates missing prior scratch.

## Definition of done
- [ ] For every node in the resumed run's re-execution set (the nodes T58 does not mark `satisfied-from-prior`), that node's scratch is copied from the linked prior run's directory into the resumed run's per-node scratch namespace before the node is admitted (arch.md line 391).
- [ ] A re-executing node in a resumed run reads back the exact scratch values its counterpart wrote in the linked prior run, through the ordinary C18 context API, with a durable run store (arch.md line 399 — C18 acceptance: "a resumed run's re-executing nodes see the prior run's scratch values").
- [ ] Carried-forward scratch lands under the resumed run's own run id, preserving the T53/T54a per-run/per-node namespacing; the resumed node has no access to the prior run's directory.
- [ ] A node marked `satisfied-from-prior` (not re-executed) has no scratch copied forward — the copy set is the re-execution set only (arch.md lines 35, 391).
- [ ] A re-executing node whose prior scratch is absent (succeeded, or never written) resumes with an empty scratch namespace; missing prior scratch is not treated as an error.
- [ ] Cross-node isolation is preserved through the copy: each node's carried-forward values are readable only within its own resumed namespace, and no node can read another node's carried-forward scratch (C18 isolation criterion, enforced not conventional).
- [ ] A scratch read/write failure during carry-forward is classified as a retry-eligible task failure attributed to the affected node, not a whole-resume abort (arch.md line 393).
- [ ] Carry-forward runs only after T58's resumability gate confirms the prior run's store survived; a resume whose store is gone never reaches carry-forward (arch.md lines 67, 565).
- [ ] The carry-forward behavior is documented at the scratch/resume boundary (a re-executing node continues from the prior checkpoint; the durable store is what makes the prior scratch exist), and any machine-classed C18/C27 carry-forward criteria are registered in the criteria matrix.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- The resume seed/closure/demand algorithm, the structural-fingerprint and policy-hash gates, parameter/interval derivation, the reference existence check, rehydration of durable inputs, and artifact lineage linkage — all owned by T58 (C27); this ticket consumes T58's re-execution set and only copies scratch.
- The resume verb's CLI surface, refusal messages, and exit codes (C26/T55/T56) and the resume acceptance suite as a whole (T59, which this ticket blocks and which observes carry-forward end-to-end).
- Proving the prior run's scratch survived the process end and the run-end retention lifecycle — established by T54a and consumed here, not re-decided.
- The run-store base-location surface, directory layout, sink, and run-identity scheme (locked by T0.6); the durable-output reference contract and its rehydration (C27/T57) — scratch carry-forward is a byte copy, not a durable-output rehydration.
- Any hard size bound, quota, or eviction on scratch, and any use of scratch to pass data between nodes — data edges own that (C18, arch.md line 389); this ticket copies only what the prior run retained.
- Cross-tool-version or cross-schema-version resume of scratch — v1 makes no cross-version promise and refuses (C27, arch.md line 622); carry-forward operates only within a comparable same-version resume that T58 has already admitted.
- A metadata store, index, or cross-run registry of scratch, and any distributed or multi-process coordination of the copy — dagr is not a metadata store, a scheduler, or a distributed execution system; carry-forward is a local, single-process copy between two per-run directories under one run store.
