# 054 · T43 — C22: run summary and critical path

> **Milestone:** M3 · **Size:** S · **Type:** feature · **Components:** C22
> **Branch:** `feat/t43-run-summary-and-critical-path` · **Depends on:** T42 · **Blocks:** T49

## Why / context
The run artifact's summary block (C22 · Run artifact) exists to answer one operational question: was this run limited by its dependency *structure* or by its *resources* — the question that decides what machine to run it on. T42 already folds the event stream into the artifact and stands up the summary skeleton (critical-path slot, peak slot residency, retained values, zombie-pinned time and capacity); this ticket makes the two headline numbers — **total elapsed time** and **critical-path time** — correct and meaningful, and adds the test that distinguishes the two regimes. Because critical-path time is ambiguous under retries and permit waits (does queueing time count as "on the path"?), and that ambiguity changes what the number *means* to an operator, this ticket first locks a short ADR fixing the definition, then implements to it. Governed by C22, drawing phase timings from C22's monotonic-offset attempt records and slot-residency semantics from C10.

## Objective
Define, via ADR, and then compute the run-summary numbers that let an operator read structure-limited versus resource-limited off a single artifact. Concretely:
- Write a short ADR fixing the **critical-path definition** — specifically whether ready-wait and permit-wait time on the critical node chain is *on the path* or *excluded* — with an explicit convention for how retried attempts on a node collapse into that node's contribution, and a stated rule for whether abandoned-but-running (zombie) time can lie on the path. The ADR records the reasoning and the rejected alternative(s) so the number's meaning is documented, not folklore.
- Compute **total elapsed time** for the run as the authoritative monotonic wall of the run (last event offset minus run start), never from the informational wall-clock stamps.
- Compute **critical-path time** as the longest dependency-respecting chain of node contributions through the executed graph, where each node's contribution is derived from its attempt records' monotonic phase durations exactly as the ADR fixes it (which phases are counted, how retries collapse), summing to a single duration on the summary.
- Populate the C22 summary block with both numbers using only monotonic offsets already present in the folded artifact, so the computation is a pure function of an artifact (no live run, no store).
- Document, in rustdoc at the summary type, precisely what each number includes and excludes, cross-referencing the ADR, so a reader knows whether permit-wait is inside critical-path time.
- Provide the structure-limited-versus-resource-limited discrimination as observable output: on a run whose critical-path time is close to total elapsed, the run is structure-limited; on a run where total elapsed greatly exceeds critical-path time (work serialized behind permits/capacity, not dependencies), it is resource-limited.

## Test plan (write these first — TDD)
- **ADR exists and is decisive.** Setup: the ADR file for the critical-path definition. Action: read it. Expected: it states, unambiguously, whether ready-wait and permit-wait are on the critical path or excluded, how a node's retried attempts collapse into its contribution, and whether zombie time can lie on the path; it names at least one rejected alternative and the reason; implementation and rustdoc match its wording. This test is a human/reviewer check recorded as a done-condition, gating the code below.
- **Total elapsed is monotonic, not wall-clock.** Setup: a folded fixture artifact whose events carry monotonic offsets from run start and deliberately skewed informational wall-clock stamps (out of order or jumping). Action: compute the summary. Expected: total elapsed equals last-offset-minus-start from the monotonic offsets and is unaffected by the skewed wall clocks.
- **Structure-limited run.** Setup: a fixture stream for a long dependency chain (each node depends on the previous) where every node was admitted immediately with no permit-wait and no retries, on a machine with ample capacity. Action: fold and compute the summary. Expected: critical-path time is approximately equal to total elapsed (within the fixture's exact constructed offsets), and the artifact's two numbers together read "structure-limited."
- **Resource-limited run.** Setup: a fixture stream for many independent sibling nodes (no dependencies among them) that could all run at once but were serialized by a small permit pool, so each accrued substantial permit-wait before executing. Action: fold and compute the summary. Expected: total elapsed greatly exceeds critical-path time (the longest single node's contribution), and the two numbers together read "resource-limited"; the exact relationship between the two matches the ADR's inclusion/exclusion rule for permit-wait.
- **Critical path respects dependencies, not raw duration.** Setup: a diamond fixture (a → b, a → c, b → d, c → d) where the *shorter*-duration branch has larger per-node wait and the longer-duration branch has none, constructed so the dependency-respecting longest chain is not simply the node with the maximum standalone duration. Action: compute critical-path time. Expected: the result equals the sum along the true longest dependency chain per the ADR's counted phases, not the single largest node and not the sum of all nodes.
- **Retries collapse per the ADR.** Setup: a fixture where one node on the critical chain has three attempts (two failed with backoff, one succeeded), each with monotonic phase durations. Action: compute that node's contribution to the path. Expected: the node contributes exactly what the ADR specifies (e.g., the summed attempt durations including or excluding backoff as fixed by the ADR), and the critical-path total reflects that collapse rule consistently.
- **Permit-wait treatment matches the ADR.** Setup: two otherwise-identical fixtures for the same chain, one with zero permit-wait and one with large permit-wait on a critical node, differing only in that phase. Action: compute critical-path time for both. Expected: the difference between the two critical-path numbers equals the added permit-wait if and only if the ADR includes permit-wait on the path, and equals zero if the ADR excludes it — the code observably implements exactly the ADR's choice.
- **Zombie-pinned time is separated from the path per the ADR.** Setup: a fixture with an abandoned-but-running node whose thread returned after the run's natural end (zombie-at-exit), whose pinned time is recorded in the summary's zombie block (from T42). Action: compute the summary. Expected: critical-path time treats that node exactly as the ADR fixes (included or excluded), and zombie-pinned time remains reported in its own summary field, not silently merged into critical-path time.
- **Pure function of an artifact.** Setup: a folded fixture artifact and no run store, no live pipeline, no network. Action: compute total elapsed and critical-path time twice from the same artifact bytes. Expected: identical results both times, computed with no I/O, proving the summary is derivable from the artifact alone (C22's "folded by a function that needs no access to the original run").
- **Single-node and never-ran graphs.** Setup: (a) a one-node fixture and (b) a fixture where the critical chain runs through a node that never executed (a propagated-terminal node with no attempt records). Action: compute the summary for each. Expected: the single-node case gives critical-path time equal to that node's contribution; the never-ran case contributes zero executed time for the propagated node while still traversing the dependency chain, and neither case panics or produces a negative or NaN duration.

## Definition of done
- [ ] A short ADR fixing the critical-path definition is committed: it decides whether ready-wait and permit-wait are on the path or excluded, how retried attempts collapse into a node's contribution, and whether zombie time can lie on the path; it names the rejected alternative and reasoning (C22 · summary; open question in the task data resolved before implementation).
- [ ] The run-artifact summary carries **total elapsed time**, computed from monotonic offsets (last-offset-minus-start), never from informational wall-clock stamps (C22).
- [ ] The run-artifact summary carries **critical-path time**, computed as the longest dependency-respecting chain of node contributions, where each contribution is derived from monotonic attempt-phase durations exactly as the ADR fixes it (C22).
- [ ] Total elapsed and critical-path time together let a reader distinguish a structure-limited run from a resource-limited one, verified by the two dedicated fixtures (C22: "the first two numbers together answer whether the run was limited by its dependency structure or by its resources").
- [ ] Critical-path computation uses only offsets/durations already present in the folded artifact and is a pure function of an artifact — no store, no live run, no I/O — verified by test (C22: foldable by a function needing no access to the original run; C10 residency already recorded by T42).
- [ ] Retries on a path node collapse into that node's contribution per the ADR, verified by a multi-attempt fixture (C22: body carries one record per attempt; retries are the signal not to be thrown away).
- [ ] Permit-wait inclusion/exclusion in critical-path time observably matches the ADR, verified by the paired zero-wait / large-wait fixtures (C11: permit-wait recorded separately from executing time; C22).
- [ ] Zombie-pinned time is treated on the path exactly per the ADR and otherwise remains in its own summary field (from T42), not merged into critical-path time (C14/C12 zombie accounting honesty; C22).
- [ ] Edge cases — single-node graph, a chain through a never-ran propagated-terminal node, and an empty/zero-attempt run — produce well-defined, non-negative, non-NaN summary durations without panicking (C22).
- [ ] Rustdoc on the summary type and the critical-path function states precisely what each number includes and excludes and cross-references the ADR (C22 documentation of meaning).
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Open questions
- Critical-path definition under retries and permit waits — is wait time on the path or excluded? Resolve by writing the short ADR named in this ticket *before* implementing; the ADR is the authority the code and tests are written against.

- The crashed-run finalize path and interrupted-marking test — T68.

- Wiring the summary into any CLI verb output surface — the CLI contract is C26/T55.
- Any drift across the scope boundary: no scheduling heuristic, no cross-run aggregation or query surface over many summaries, no "recommended machine" advisor, and no runtime-mutable graph shape — the summary is a pure per-run readout of a single fixed-shape run, and comparing runs is the operator's job outside this tool.
