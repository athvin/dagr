# 011 · T0.5 — Bootstrap phase interface and cost model

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C5, C7, C12
> **Branch:** `adr/t0.5-bootstrap-phase-and-cost-model` · **Depends on:** — · **Blocks:** T14, T29, T31, T32

## Why / context
The boundary between assembly (pure) and bootstrap (deliberately impure) is the single load-bearing seam of a run's shape (arch.md "The shape of a run"): assembly may touch no network, filesystem, clock, credentials, or parameter values, which is exactly what lets the graph be emitted and validated in CI on every pull request; everything that needs the actual machine or invocation — run identity, store and stream, resource registry, container-limit probing and pool sizing, cost-fit rejection, parameter parsing, environment capture — happens in one named place that fails fast with artifacts and never hangs. This ticket exists to lock that seam as an *interface* — what assembly produces, what bootstrap consumes, where a bootstrap failure lands, and the exact per-pool cost-vector types — as an Architecture Decision Record, so that assembly validation and precomputation (T14), node policy (T29), admission pools (T31), and container-limit detection (T32) are all written against one settled contract rather than re-litigating it. It governs C7's assembly/bootstrap check partition, C5's declared-cost vector (working memory versus output residency, bytes versus thread counts), and C12's admission-pool units and cost-fit rejection. It jointly settles T24's run-identity seam with the T0.6 run-store ADR (which owns the store and sink contract); this ticket owns only the *type shape* of the seam and the cost model, not the store's on-disk format.

## Objective
Record, as a committed ADR, the assembly→bootstrap→execution interface and the per-pool cost model, leaving no seam or cost-type choice open to T14, T29, T31, or T32. Concretely, decide and write down:

- **What assembly produces.** Fix the shape of the immutable, machine-independent artifact that pure assembly (C7) hands forward: the graph structure, per-node effective policy including the declared cost vector, precomputed consumer counts, remaining-dependency counts, execution order, and the graph fingerprint slot — and the fact that it carries *no* parameter values, no clock, no filesystem or network state. Name the property that this value is fully constructible in an empty environment.
- **What bootstrap consumes and produces.** Fix the ordered list of bootstrap's inputs (the assembly artifact plus the actual invocation: parameters, environment, container limits, the resource registry) and its outputs (a validated run context ready for execution, or a bootstrap failure). Record the ordering constraint that, for the run verbs, run identity is minted and the store and stream are opened *before* assembly executes — so an assembly failure has a place to record itself — while the inspection verbs (validate, graph, render) run assembly with no store at all.
- **Where the two failure modes land.** Fix the distinction and the artifact path: an assembly failure yields the `assembly-failed` run-artifact variant (fingerprint absent, complete problem list, zero attempts); a bootstrap failure yields the distinct `bootstrap-failed` variant (fingerprint present, since assembly succeeded), each with its own exit code (C26). Record that a bootstrap failure produces artifacts (C22) at the predictable run-store location and never hangs, and that if the store itself cannot be opened there is nowhere to write — that case exits to stderr with the sink-failure code and promises nothing more.
- **The per-pool cost vector types.** Fix the cost vector as one entry per admission pool in that pool's native unit: bytes for the memory pool, a thread count for each thread pool. Fix the memory split into *working memory* (held for the attempt, released at its terminal state) and *output residency* (transferred to the output slot when the value is produced, released when the last consumer is terminal — C10). Record the conservative default of zero declared cost, and the honesty rule that the run artifact juxtaposes declared against measured cost (C12, C23) so dishonest declarations are visible.
- **The cost-fit rejection contract.** Fix which cost checks belong to bootstrap, not assembly: a declared cost no pool can ever satisfy (exceeds a pool's total capacity), a missing declared resource, and an invalid parameter are each rejected at bootstrap, before any node executes, with a distinct complete error report and the bootstrap-failure artifact. Record that assembly still owns the checks the compiler cannot make but the machine is not needed for — invalid execution-class overrides among them.
- **Record it.** Write the ADR at the ticket `path` with status, context, decision, consequences, and rejected alternatives, so T14/T29/T31/T32 inherit one seam and one cost model.

## Context

`docs/arch.md` makes the boundary between the first two phases of a run
**load-bearing**:

> "A run passes through four phases, and the boundary between the first two is
> load-bearing." ("The shape of a run")

Six arch.md sections meet at this decision and must mean *exactly* the same thing
by "assembly," "bootstrap," "declared cost," and "the two pre-execution
failures":

- **"The shape of a run".** Fixes the four phases; that assembly is pure (no
  network, filesystem, clock, credentials, parameter values) and bootstrap is
  deliberately impure and happens "in one named place"; that for the run verbs
  identity is minted and the store and stream are opened *before* assembly
  executes so an assembly failure has a place to record itself, while inspection
  verbs run assembly with no store; that bootstrap fails fast, produces artifacts
  and a distinct exit code, and never hangs; and that if the store itself cannot
  be opened the error goes to stderr with the sink-failure exit code and the spec
  promises nothing more.
- **C5 · Node policy.** Fixes declared cost as "a vector with one entry per
  admission pool (C12), in that pool's native unit: bytes for the memory pool, a
  thread count for the thread pools," the working-memory / output-residency
  split, the conservative zero default, and the declared-vs-measured honesty
  rule; and that an invalid execution-class override fails at **assembly**.
- **C7 · Flow assembly.** Fixes the assembly-output value (immutable pipeline
  carrying consumer counts, remaining-dependency counts, execution order, the
  graph fingerprint), that assembly is pure and provably runs in an empty
  environment, that no parameter value is reachable during assembly, and that
  "checks that need the actual machine — capacity, resources, credentials —
  belong to bootstrap," which "rejects, before any node executes: a declared cost
  no pool can ever satisfy, a missing declared resource, an invalid parameter —
  each with a distinct, complete error report and an assembly/bootstrap-failure
  artifact."
- **C10 · Output slot.** Fixes that output residency "transfers from the
  producing attempt to the slot when the value is produced, and is released only
  when the slot actually releases — which … waits for zombie consumers to
  return," the same honesty rule as C12's zombie accounting.
- **C12 · Admission controller.** Fixes that pools are "the genuinely constrained
  resources — memory and threads at minimum," that a node whose declared cost
  exceeds any pool's total capacity "is rejected at bootstrap — fail fast, never
  wedge at admission," and that the artifact juxtaposes declared against measured
  cost.
- **C22 · Run artifact** and **C26 · Command-line contract.** Fix that a run that
  failed before execution still produces an artifact — outcome `assembly-failed`
  or `bootstrap-failed`, "the two are distinct — one is the graph's fault, the
  other the machine's — fingerprint present only when assembly succeeded, the
  complete error list, zero attempts" — and that distinct exit codes exist for
  assembly failure, bootstrap failure, and sink failure.

Landing this wrong means T14, T29, T31, and T32 each build against a different
seam and a different cost model, so the interface below is fixed once, in one
place, in arch.md's exact vocabulary.

## Decision

### 1. What assembly produces — the assembly-output interface

Pure assembly (C7) produces a single **immutable, machine-independent value** —
call it the **assembly artifact** (the immutable pipeline plus the graph-artifact
data it computes). It is the *only* thing assembly hands forward, and it carries
**no parameter value, no clock reading, no filesystem or network state, no
credential**. It carries exactly:

| Field | Content |
|---|---|
| **Graph structure** | the node set (stable declared names), the edge set (data and ordering edges, with carried type names and edge kinds). |
| **Per-node effective policy** | every node's *full* effective policy including defaulted values — retry/backoff/timeout/trigger-rule/class/group/retention/durability **and the declared cost vector** (§4). T29 owns the full struct; this ADR fixes only that the cost vector is part of it and appears with defaults written out. |
| **Consumer counts** | the precomputed exact consumer count per node. |
| **Remaining-dependency counts** | the precomputed remaining-dependency count per node (the readiness countdown seed, C11). |
| **Execution order** | the precomputed execution order. |
| **Fingerprint slot** | the graph-fingerprint slot (structural fingerprint + policy hash, T0.7); the *slot* is part of the interface here, the algorithm is T0.7/T41. |

**Illustrative type shape** (not shipping code — the concrete definition is T14):

```rust
/// Immutable, machine-independent output of pure assembly (C7).
/// Constructible with every external resource absent; carries no parameter
/// value, no clock, no filesystem/network state, no credential.
pub struct AssemblyArtifact {
    graph: GraphStructure,                 // node set + edge set (names, types, edge kinds)
    policies: NodeMap<EffectivePolicy>,    // full effective policy incl. defaulted cost vector
    consumer_counts: NodeMap<u32>,         // precomputed, exact before any execution
    remaining_deps: NodeMap<u32>,          // readiness countdown seed (C11)
    order: Vec<NodeId>,                     // precomputed execution order
    fingerprint: FingerprintSlot,           // structural fingerprint + policy hash slot (T0.7)
    // No Parameters field, by construction — see §2.
}
```

**Constructible in an empty environment (C7).** The value is `AssemblyArtifact`
above with *no* type reachable from it that requires a network, a filesystem, a
clock, a credential, or a parameter value to construct. Assembling the same
pipeline twice in one process yields byte-identical graph-artifact data (the
generation-time field aside, C20). This is the property T15 later proves by
running assembly in an empty environment.

### 2. Assembly is pure, and no parameter value is reachable

Assembly touches **no network, no filesystem, no clock, no credentials, and no
parameter values**. Parameters are parsed at **bootstrap**, *after* assembly, so
the graph provably cannot depend on them (C7). This is an **interface-shape**
guarantee, not merely a convention: **no field or method of the assembly-output
type exposes a path to a parsed parameter value.** The illustrative
`AssemblyArtifact` above has no `Parameters` field and no accessor that returns
one; the typed parameter struct (C26) is threaded *only* into the bootstrap entry
point (§3), never into assembly. This is what T14's "enforce no-parameters-during-
assembly" and T15's "no parameter value is reachable" tests assert against.

### 3. What bootstrap consumes and produces — inputs, outputs, and ordering

**Bootstrap** is the single, named, deliberately impure phase. Its **inputs**, in
the fixed order they are consumed, are the **assembly artifact** plus the **actual
invocation**:

1. the assembly artifact (§1);
2. the raw parameters (the typed parameter struct's unparsed source, C26);
3. the environment (for allowlisted capture, C22);
4. the container-limit source (cgroup v2 → v1 → host — probed here, detected by
   T32);
5. the resource registry (built in `main`, C9).

Its **outputs** are exactly one of:

- a **validated run context** ready for execution (C8), or
- a **bootstrap failure** (§4).

**Illustrative entry-point shape** (not shipping code — the concrete signature is
T24/T31/T32):

```rust
/// The single, named, deliberately-impure phase. Never hangs; on failure it
/// returns a BootstrapFailure that lands as the `bootstrap-failed` artifact.
pub fn bootstrap(
    artifact: AssemblyArtifact,        // from pure assembly (§1)
    params: RawParameters,             // parsed HERE, never in assembly (§2)
    env: Environment,                  // allowlisted capture (C22)
    limits: &dyn ContainerLimitSource, // cgroup v2 -> v1 -> host (T32)
    registry: ResourceRegistry,        // built in main (C9)
) -> Result<RunContext, BootstrapFailure>;
```

**Ordered bootstrap steps.** Bootstrap runs its steps in this recorded order and
produces a run context or a bootstrap failure — never a partial hang:

1. **(run verbs only) mint run identity, open the run store, open the event
   stream** — *before* assembly executes, so an assembly failure has a place to
   record itself. (Assembly itself stays pure; opening the store is the runtime's
   act, not the graph's.)
2. **construct the resource registry and check it** against the declared
   requirements (a missing resource is a bootstrap failure naming the resource and
   every node requiring it — C9);
3. **probe container limits and size the admission pools** (T32 detection; 20%
   headroom default; ≥1 unit per pool; pinning flag overrides);
4. **cost-fit** — reject any node whose declared cost exceeds a pool's total
   capacity (§5);
5. **parse and validate parameters** (C26; an invalid parameter is a bootstrap
   failure);
6. **capture allowlisted environment values** (C22; allowlist empty by default).

For the **run verbs** (`run`, single-node, resume), step 1 happens *before*
assembly. **Inspection verbs** (`validate`, `graph`, `render`) run assembly with
**no store at all** — no run-store directory is created, output goes to standard
output and an exit code. Bootstrap **fails fast**: every rejection returns a
distinct, complete error report promptly, before any node executes, with no wait
and no wedge.

### 4. The per-pool cost model — vector types, native units, memory split

**Declared cost is a vector with exactly one entry per admission pool, each in
that pool's native unit** (C5, C12):

| Pool | Native unit | Entry |
|---|---|---|
| memory | **bytes** | a byte count, split into two components (below) |
| each thread pool (blocking, compute — T2) | **thread count** | an integer thread count |

**Illustrative type shape** (not shipping code — T29 owns the concrete struct):

```rust
/// One entry per admission pool, in that pool's native unit (C5, C12).
pub struct DeclaredCost {
    memory: MemoryCost,   // bytes, split into working + output residency
    blocking: u32,        // thread count for the blocking pool (T2)
    compute: u32,         // thread count for the compute pool (T2)
}

/// The memory entry splits into two independently-charged lifetimes (C5, C10).
pub struct MemoryCost {
    /// Held for the attempt; released at the node's terminal state
    /// (when the closure has returned — a zombie holds it past the decision, T0.3).
    working: u64,   // bytes
    /// Transferred to the output slot when the value is produced; released when
    /// the last consumer is terminal AND every consumer's closure has returned (C10).
    output_residency: u64, // bytes
}
```

**The memory split is normative** (C5, C10):

- **Working memory** — held for the attempt, **released at its terminal state**.
  This is the permit-held memory of T0.3's ledger; a zombie (abandoned-but-
  running) attempt holds it past the *terminal decision* until its closure
  actually returns (T0.3), which is the honest ledger, not a contradiction.
- **Output residency** — **transferred to the output slot when the value is
  produced**, and **released when the last consumer is terminal** (C10) — more
  precisely, when every consumer is terminal *and* every consumer's closure has
  returned, so a zombie consumer pins both the value and its residency lease
  (amended C10). Counted **once** against the memory pool regardless of consumer
  count (a slot's value is counted once, not once per consumer — C10).

Thread-pool entries are plain thread counts (how many threads of that pool the
node's internal parallelism draws — the blessed one-node-iterates-internally
pattern declares this honestly, C5/C12).

### 5. Defaults, honesty, and cost-fit rejection

**Zero declared cost is the conservative default, applied uniformly** (C5). A
node with **no stated cost** behaves **identically** to one with an all-zero cost
written out across every pool (`working = 0`, `output_residency = 0`,
`blocking = 0`, `compute = 0`) — including under the fingerprint's policy hash
(C21). **Every node's full effective cost vector, including defaulted zeros,
appears in the assembly/graph artifact** (§1; C5).

**Declared-vs-measured honesty rule.** The run artifact **juxtaposes declared
against measured cost** per node so that **dishonest declarations are visible**
(C5, C12, C23). Measured-cost capture and the juxtaposition's emission are later
work (C23/T42/T44); this ADR fixes only that the cost model **supports** it — the
declared vector is recorded per node so a later measured vector can sit beside it.

**Cost-fit rejection is a bootstrap check, not an assembly check** (C12). The
following are rejected at **bootstrap**, before any node executes, each with a
**distinct, complete error report** and the **`bootstrap-failed` artifact**:

- a **declared cost no pool can ever satisfy** — a node whose declared cost for
  some pool exceeds that pool's **total capacity** (fail fast, never wedge at
  admission — C12);
- a **missing declared resource** — a node declaring a resource type absent from
  the registry, naming the resource and every node requiring it (C9);
- an **invalid parameter** — a parameter that fails parsing/validation (C26).

Assembly **cannot** make the cost-fit check — it does not know pool sizes, because
the machine is absent — so a node whose declared cost exceeds a pool's total
capacity **passes assembly** and is caught at bootstrap. Conversely, **assembly
still owns the checks the compiler cannot make but the machine is not needed
for**, including the **invalid execution-class override** (an await-bound task
overridden to a synchronous class needs no machine to be judged wrong — it fails
at **assembly**, C5/C7).

### 6. Where the two pre-execution failures land — the two variants

The two pre-execution failure modes are **distinct** and land as **distinct
run-artifact variants** (C22), each with its **own exit code** (C26):

| Variant | Cause | Fingerprint | Contents | Exit code | Artifact |
|---|---|---|---|---|---|
| **`assembly-failed`** | the graph's fault (a check assembly owns) | **absent** (assembly did not succeed) | the **complete problem list**, **zero attempts** | the assembly-failure code | written at the predictable run-store location |
| **`bootstrap-failed`** | the machine's fault (a check bootstrap owns) | **present** (assembly *did* succeed) | the **complete error report**, **zero attempts** | the distinct bootstrap-failure code | written at the predictable run-store location |

In prose: the **`assembly-failed`** variant has its **fingerprint absent** (a
fingerprint is computed only when assembly succeeds), a **complete problem list**,
and **zero attempts**; the **`bootstrap-failed`** variant has its **fingerprint
present** (assembly succeeded), a **complete error report**, and **zero
attempts**.

Both variants **produce an artifact** at the predictable `<base>/<pipeline>/<run-
id>/` run-store location (because, for the run verbs, the store and stream were
opened *before* assembly — §3) and **neither hangs**. The fingerprint's
presence/absence is the observable that distinguishes them: **assembly succeeded
iff the fingerprint is present.**

**Store-open failure — the one case with no artifact.** If the store itself
cannot be opened, there is **nowhere to write an artifact by definition**: the
error goes to **stderr** with the **sink-failure exit code**, **no artifact is
written**, and the spec **promises nothing more** ("The shape of a run"; the
event-stream store-open clause).

### 7. The compile / assembly / bootstrap partition table

So that T14, T29, T31, and T32 inherit **one** boundary, this ADR fixes which
failures are caught **where**. A failure belongs at the **earliest** phase whose
information suffices to detect it: the compiler when the types settle it,
assembly when a whole-graph fact settles it with the machine absent, bootstrap
when the actual machine or invocation is required.

| Phase | Detects | Examples (normative) |
|---|---|---|
| **Compile-time** | anything the type system settles | wrong-type binding (C3), wrong arity (C3/T5), a cycle (unrepresentable — C2), a non-default trigger rule on a data-dependent node (C4 typestate, T0.4). |
| **Assembly-time** (pure, machine absent) | whole-graph facts needing no machine | **duplicate node names** (naming both declarations), **empty pipeline**, **invalid execution-class override**, **duplicate stable type/task names** (C20), **durable-marked node whose output type lacks the durability contract** (C27/T0.8), **ownership-mode conflicts** (owned demand on a multi-consumer value; owned edge into a retrying node without clone-on-read; nonzero teardown cost). The zero-consumer non-unit-output case is a **warning**, not a failure (C7). |
| **Bootstrap-time** (machine + invocation present) | facts needing the actual machine or invocation | **a declared cost no pool can ever satisfy** (exceeds a pool's total capacity — C12), **a missing declared resource** (C9), **an invalid parameter** (C26). |

The load-bearing consistency point: **cost-fit is bootstrap, the invalid
execution-class override is assembly.** The former needs pool sizes (the machine);
the latter needs only the task's declared work shape (no machine). This split is
what C7's "capacity, resources, credentials belong to bootstrap" and C5's "an
invalid override fails at assembly" jointly require, and it is fixed here so no
consumer re-litigates it.

### 8. Downstream hand-off — what each blocked ticket consumes

This ADR is the single source the blocked tickets reference; none re-derives the
seam or the cost model:

- **T14** (assembly validation and precomputation, C7) consumes the
  **assembly-output interface** (§1), the **assembly-side rows of the partition
  table** (§7) — the checks it must make and report *all* of — the
  **no-parameters-during-assembly** guarantee (§2), and the fact that **cost-fit
  moved to bootstrap** (§5), so assembly does *not* make the capacity check.
- **T29** (node policy, C5) consumes the **declared-cost vector types and
  defaults** (§4, §5) — one entry per pool in native units, the working/output-
  residency split, the zero default applied uniformly, and the requirement that
  the effective cost (incl. defaults) appears in the graph artifact. T29 owns the
  full policy struct; this ADR fixes only the cost field's shape.
- **T31** (admission pools and permit lifecycle, C12) consumes the **cost-vector
  units** (§4) its ledger charges (composing with T0.3's permit-held-until-return
  and slot-lease accounting) and the **cost-fit-at-bootstrap rejection contract**
  (§5) — a too-big node is rejected at bootstrap, never wedged at admission.
- **T32** (container-limit detection, C12) consumes the **pool-sizing step's
  place in the bootstrap order** (§3 step 3) and the **too-big-node rejection**
  it feeds (§5) — it probes limits and sizes pools, then cost-fit rejects nodes
  that cannot fit, producing the `bootstrap-failed` artifact.

The seam also **jointly settles T24's run-identity seam with the T0.6 run-store
ADR**: this ticket owns the **type shape** of the seam (identity minted and
store/stream opened before assembly for run verbs — §3, §6); T0.6 owns the
store's on-disk format, the sink contract, and durability. There is no overlap and
no conflict: T0.6 is referenced here only as the place the `bootstrap-failed`
artifact lands.

## Consequences

- **One seam, one cost model, inherited not re-litigated.** T14, T29, T31, and
  T32 each build against the same assembly-output interface (§1), the same
  bootstrap inputs/outputs/ordering (§3), the same two failure variants (§6), the
  same per-pool cost vector (§4), and the same compile/assembly/bootstrap
  partition (§7). No downstream ticket re-decides where a check lives or what the
  cost vector's units are.
- **Purity is an interface property, not a convention.** Because no parameter
  value is reachable from the assembly-output type (§2), the graph *provably*
  cannot depend on parameters — the empty-environment assembly test (T15) is
  testing a structural fact, not a discipline.
- **The two pre-execution failures stay distinguishable forever.** Fingerprint
  presence is the observable that tells `assembly-failed` from `bootstrap-failed`
  (§6), which is exactly what artifacts, diagrams, and exit codes need to attribute
  a pre-execution failure to the graph or to the machine.
- **The cost model composes with the merged M0 ADRs.** The working/output-
  residency split (§4) is what lets T0.3's ledger charge the two memory lifetimes
  independently; the thread-count-per-pool structure matches T2's blocking/compute
  pools; "terminal" is read as T0.4 defines it. No merged decision moves.
- **Cost-fit at bootstrap keeps fail-fast honest.** A node too big for the machine
  is rejected before any node executes (§5), producing an artifact and a distinct
  exit code — never a mid-run wedge at admission. This is the behavior T31/T32
  depend on.
- **No runtime, no scheduler, no cross-process coordination.** This ADR fixes the
  interface and the model only; it implements no bootstrap phase, no scheduler, no
  cost evaluation, and no admission machinery (that is T14/T29/T31/T32). It adds no
  cross-run capacity coordination and no runtime graph mutation — dagr's graph
  shape never changes at runtime, it is one run per container, and it is never a
  scheduler or a metadata store.
- **No coverage-matrix change.** This ADR ships **no covering test**, so it makes
  **no change to `docs/coverage-matrix.md`**. C5, C7, and C12 remain `unmapped`,
  owed by their implementing tickets (C5 → T29, C7 → T13, C12 → T31), which flip
  them as they land their tests. The matrix contract (a decision ticket that owes
  no test changes nothing) is honored exactly.

## Rejected alternatives

- **A single scalar cost** (one number per node instead of a per-pool vector).
  Rejected: the admission pools are **independently constrained** and acquisition
  is **all-or-nothing per pool** (C12) — a node needing 4 GiB and 2 compute
  threads fits only when *both* pools have room, and a scalar cannot express "fits
  memory but not threads." A scalar also cannot carry the memory-vs-thread unit
  distinction (bytes vs counts) the pools require. The vector with one entry per
  pool in native units is what makes atomic multi-pool admission expressible
  (C5, C12).
- **Undifferentiated memory without the working/output-residency split** (a single
  byte figure per node). Rejected: it **cannot express C10's accounting** —
  working memory is released at the attempt's terminal state, while output
  residency is *transferred to the slot* on production and released only when the
  last consumer is terminal (and its closure has returned). One figure cannot
  release at two different times against two different lifetimes; the split is
  what lets the ledger (T0.3/T31) return working memory promptly while the slot
  still pins residency for downstream consumers.
- **Making cost-fit an assembly check** (rejecting a too-big node at assembly).
  Rejected: assembly is pure and the machine is absent, so **assembly does not
  know pool sizes** — a cost that "no pool can satisfy" is only knowable once
  bootstrap has sized the pools from the container's limits (C7, C12). Putting it
  at assembly would either require the machine during assembly (breaking purity
  and empty-environment emission) or bake a fixed capacity into the graph (making
  the graph machine-dependent). It is a **bootstrap** check.
- **Making the invalid execution-class override a bootstrap check.** Rejected
  (the mirror of the above): the override's validity depends only on the task's
  **declared work shape** (await-bound cannot become synchronous), which is a
  compile-time-known, machine-independent fact — so it is caught at **assembly**,
  with no machine needed (C5/C7). Deferring it to bootstrap would delay a failure
  that assembly can already prove.
- **Letting assembly see parameter values** (threading the parsed parameter struct
  into assembly for convenience). Rejected as a permanent boundary: if the graph
  could depend on parameter values it could no longer be emitted and validated in
  CI on every pull request, and the empty-environment guarantee (C7) would be
  false. Parameters enter **only** at bootstrap (§2, §3).
- **A third pre-execution failure variant, or collapsing the two into one.**
  Rejected: `assembly-failed` and `bootstrap-failed` are distinct because one is
  the graph's fault and the other the machine's, and the fingerprint's
  presence/absence must stay the observable that tells them apart (C22). Collapsing
  them would lose the graph-vs-machine attribution the exit codes and artifacts
  need; a third variant has no cause arch.md defines.
- **Coordinating pool capacity across simultaneous runs** (so two runs on one host
  split the machine). Rejected as a permanent non-goal: the tool does not
  coordinate between processes — "that road ends in building a scheduler." Each
  process sizes its pools from the same machine limits; splitting capacity is the
  operator's call via the pinning flag (C12), not a cost-model feature.

**Reopen condition.** This ADR is DOC-ONLY because every choice is a direct
reading of arch.md and needs no spike. It **reopens** only if T14, T29, T31, or
T32 discovers that the chosen seam or cost model **cannot be expressed** — e.g.
if the assembly-output type cannot be made both parameter-free *and* carry the
full effective policy the artifact needs, or if the working/output-residency
split cannot be charged as two independent lifetimes against one memory pool in
the real ledger. In that case the implementing ticket STOPs and this decision is
revisited, rather than papering over the defeat (the ticket-conventions
decision(spike) reopen rule, applied to the interface). No such obstacle is known
or expected: the split is exactly T0.3's already-proven permit/slot-lease ledger,
and the parameter-free assembly type is exactly what C7 already asserts is
constructible in an empty environment.
