# 011 · T0.5 — Bootstrap phase interface and cost model

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C5, C7, C12
> **Branch:** `adr/t0.5-bootstrap-phase-and-cost-model` · **Depends on:** — · **Blocks:** T14, T29, T31, T32

## Why / context
The boundary between assembly (pure) and bootstrap (deliberately impure) is the single load-bearing seam of a run's shape (arch.md "The shape of a run"): assembly may touch no network, filesystem, clock, credentials, or parameter values, which is exactly what lets the graph be emitted and validated in CI on every pull request; everything that needs the actual machine or invocation — run identity, store and stream, resource registry, container-limit probing and pool sizing, cost-fit rejection, parameter parsing, environment capture — happens in one named place that fails fast with artifacts and never hangs. This ticket exists to lock that seam as an *interface* — what assembly produces, what bootstrap consumes, where a bootstrap failure lands, and the exact per-pool cost-vector types — as an Architecture Decision Record, so that assembly validation and precomputation (T14), node policy (T29), admission pools (T31), and container-limit detection (T32) are all written against one settled contract rather than re-litigating it. It governs C7's assembly/bootstrap check partition, C5's declared-cost vector (working memory versus output residency, bytes versus thread counts), and C12's admission-pool units and cost-fit rejection. It jointly settles T24's run-identity seam with the T0.6 run-store ADR (which owns the store and sink contract); this ticket owns only the *type shape* of the seam and the cost model, not the store's on-disk format.

## Objective
Record, as a committed ADR, the assembly→bootstrap→execution interface and the per-pool cost model, leaving no seam or cost-type choice open to T14, T29, T31, or T32. Concretely, decide and write down:

- **What assembly produces.** Fix the shape of the immutable, machine-independent artifact that pure assembly (C7) hands forward: the graph structure, per-node effective policy including the declared cost vector, precomputed consumer counts, remaining-dependency counts, execution order, and the graph fingerprint slot — and the fact that it carries *no* parameter values, no clock, no filesystem or network state. Name the property that this value is fully constructible in an empty environment.
- **What bootstrap consumes and produces.** Fix the ordered list of bootstrap's inputs (the assembly artifact plus the actual invocation: parameters, environment, container limits, the resource registry) and its outputs (a validated run context ready for execution, or a bootstrap failure). Record the ordering constraint that, for the run verbs, run identity is minted and the store and stream are opened *before* assembly executes — so an assembly failure has a place to record itself — while the inspection verbs (validate, graph, render) run assembly with no store at all.
- **Where the two failure modes land.** Fix the distinction and the artifact path: an assembly failure yields the `assembly-failed` run-artifact variant (fingerprint absent, complete problem list, zero attempts); a bootstrap failure yields the distinct `bootstrap-failed` variant (fingerprint present, since assembly succeeded), each with its own exit code (C26). Record that a bootstrap failure produces artifacts (C22) at the predictable run-store location and never hangs, and that if the store itself cannot be opened there is nowhere to write — that case exits to stderr with the sink-failure code and promises nothing more.
- **The per-pool cost vector types.** Fix the cost vector as one entry per admission pool in that pool's native unit: bytes for the memory pool, a thread count for each thread pool. Fix the memory split into *working memory* (held for the attempt, released at its terminal state) and *output residency* (transferred to the output slot when the value is produced, released when the last consumer is terminal — C10). Record the conservative default of zero declared cost, and the honesty rule that the run artifact juxtaposes declared against measured cost (C12, C23) so dishonest declarations are visible.
- **The cost-fit rejection contract.** Fix which cost checks belong to bootstrap, not assembly: a declared cost no pool can ever satisfy (exceeds a pool's total capacity), a missing declared resource, and an invalid parameter are each rejected at bootstrap, before any node executes, with a distinct complete error report and the bootstrap-failure artifact. Record that assembly still owns the checks the compiler cannot make but the machine is not needed for — invalid execution-class overrides among them.
- **Record it.** Write the ADR at the ticket `path` with status, context, decision, consequences, and rejected alternatives, so T14/T29/T31/T32 inherit one seam and one cost model.

## Test plan (write these first — TDD)
These are the evidence and decision-record checks that validate the choices. Each pairs a throwaway prototype (kept out of the shipping crates, or gated so it never becomes production surface, and discarded or quarantined once the ADR is blessed) or an assertable check against the committed ADR with a concrete, independently checkable outcome.

- **Assembly artifact is constructible in an empty environment.** Setup: a prototype assembly-output value built with no network, no filesystem, no clock, no credentials, and no parameter values available. Action: construct the assembly artifact and read back its fields. Expected outcome: the value carries graph structure, per-node effective policy including the declared cost vector, consumer counts, remaining-dependency counts, execution order, and a fingerprint slot, and construction succeeds with every external resource absent — validating C7's "assembly succeeds with every external resource absent."
- **No parameter value is reachable during assembly.** Setup: the assembly-artifact type and a parameter struct. Action: inspect the assembly type's surface for any path to a parameter value. Expected outcome: no field or method exposes a parsed parameter value; parameters enter only at bootstrap — an assertable shape check validating C7's "no parameter value is reachable during registration or assembly."
- **Bootstrap consumes the assembly artifact plus invocation, in the fixed order.** Setup: a prototype bootstrap entry point given an assembly artifact and a synthetic invocation (parameters, environment, container-limit source, resource registry). Action: run bootstrap through its ordered steps. Expected outcome: the steps execute in the recorded order — identity/store/stream first (for run verbs), then resource-registry construction and check, then pool sizing, then cost-fit, then parameter parse and validation, then environment capture — and produce either a validated run context or a bootstrap failure, never a partial hang.
- **Run identity and store open before assembly for run verbs.** Setup: a prototype run-verb path with an assembly step scripted to fail. Action: invoke the run verb. Expected outcome: run identity was minted and the store/stream were opened before assembly ran, so the assembly failure lands as an artifact rather than vanishing — validating the seam ordering from "The shape of a run" and jointly with T0.6.
- **Inspection verbs run assembly with no store.** Setup: a prototype validate/graph/render path. Action: run it. Expected outcome: assembly runs without opening a store, and output goes to standard output and an exit code — no run-store directory is created for an inspection verb.
- **Assembly failure yields the assembly-failed variant.** Setup: an assembly that fails (for example a duplicate node name). Action: run a run verb over it. Expected outcome: the run artifact carries outcome `assembly-failed`, the fingerprint is absent, the complete problem list is present, and there are zero attempts — validating C22's assembly-failure variant and the distinctness from bootstrap failure.
- **Bootstrap failure yields the distinct bootstrap-failed variant with a fingerprint.** Setup: an assembly that succeeds followed by a bootstrap that fails (for example a node whose declared cost exceeds a pool's total capacity). Action: run a run verb over it. Expected outcome: the run artifact carries outcome `bootstrap-failed`, the fingerprint is *present* (assembly succeeded), the complete error report is present, and the artifact is written at the predictable run-store location — distinct from `assembly-failed`.
- **Bootstrap failure fails fast and never hangs.** Setup: each of the three bootstrap rejection causes — a cost no pool can satisfy, a missing declared resource, an invalid parameter. Action: bootstrap each. Expected outcome: each returns a distinct, complete error report and the bootstrap-failure artifact promptly, before any node executes, with no wait or wedge — validating C7's bootstrap-rejects acceptance criterion.
- **Store-open failure exits to stderr with the sink-failure code.** Setup: a run verb whose store base cannot be opened. Action: invoke it. Expected outcome: the error goes to stderr with the sink-failure exit code and no artifact is written, because there is nowhere to write one — validating "The shape of a run" and the event-stream store-open clause.
- **Cost vector has one entry per pool in native units.** Setup: a prototype declared-cost value for a node. Action: read its per-pool entries. Expected outcome: the memory entry is in bytes and each thread-pool entry is a thread count; there is exactly one entry per admission pool — validating C5's "a vector with one entry per admission pool, in that pool's native unit."
- **Memory cost splits into working memory and output residency.** Setup: a prototype declared memory cost. Action: read its two components. Expected outcome: it carries a working-memory component (held for the attempt, released at terminal state) distinct from an output-residency component (transferred to the output slot on production, released when the last consumer is terminal per C10) — validating C5's memory split.
- **Zero declared cost is the default and behaves uniformly.** Setup: a node with no stated cost and a node with cost written out as zero. Action: read the effective cost vector of each. Expected outcome: both are identical (all-zero across every pool) — validating C5's "zero declared cost" default and "a node with no stated policy behaves identically to one with every default written out."
- **Effective declared cost appears in the assembly artifact.** Setup: nodes with mixed stated and defaulted costs. Action: read the per-node effective policy in the assembly artifact. Expected outcome: every node's full declared cost vector, including defaulted zeros, is present — validating C5's "every node's full effective policy appears in the graph artifact, including defaulted values."
- **Cost-fit rejection is a bootstrap check, not an assembly check.** Setup: a node whose declared cost exceeds a pool's total capacity. Action: run assembly, then bootstrap. Expected outcome: assembly *passes* (it does not know pool sizes — the machine is absent), and bootstrap rejects the node before admission, producing the bootstrap-failure artifact — validating C12's "a node whose declared cost exceeds total capacity fails at bootstrap, not at admission time."
- **Invalid execution-class override stays an assembly failure.** Setup: an await-bound task overridden to a synchronous class. Action: run assembly. Expected outcome: assembly fails (the override needs no machine to be judged wrong) — confirming the seam correctly leaves this on the assembly side, validating C5/C7's invalid-override criterion.
- **Compile-time / assembly-time / bootstrap-time partition record check.** Setup: the committed ADR. Action: read the Decision section's partition table. Expected outcome: it lists which failures are compile errors, which are assembly failures (duplicate names, empty pipeline, invalid class override, durable-without-contract, mode conflicts), and which are bootstrap failures (cost no pool can satisfy, missing resource, invalid parameter) — an assertable presence check against the file.
- **Cost-model record check.** Setup: the committed ADR. Action: read the cost-model section. Expected outcome: it names bytes for memory and thread counts for thread pools, the working-memory-versus-output-residency split with each component's release timing, the zero default, and the declared-versus-measured honesty rule — an assertable presence check.
- **Rejected-alternatives record check.** Setup: the committed ADR. Action: read the rejected-alternatives section. Expected outcome: at least the alternative of a single scalar cost (rejected because pools are independently constrained and acquisition is all-or-nothing per pool, C12) and the alternative of undifferentiated memory without the working/residency split (rejected because it cannot express C10's transfer-to-slot-and-release-with-last-consumer accounting) are named with the reason each was rejected.

## Definition of done
- [ ] The ADR fixes the assembly-output interface: an immutable, machine-independent value carrying graph structure, per-node effective policy (including the declared cost vector), consumer counts, remaining-dependency counts, execution order, and a fingerprint slot, provably constructible with every external resource absent (C7).
- [ ] The ADR records that assembly is pure — no network, filesystem, clock, credentials, or parameter values — and that no parameter value is reachable from the assembly-output type (C7).
- [ ] The ADR fixes bootstrap's ordered inputs (assembly artifact plus invocation: parameters, environment, container limits, resource registry) and outputs (validated run context, or a bootstrap failure), and the ordering that — for run verbs — identity is minted and store/stream opened before assembly runs, while inspection verbs run assembly with no store (arch.md "The shape of a run", C22).
- [ ] The ADR fixes the two pre-execution failure variants as distinct: `assembly-failed` (fingerprint absent, complete problem list, zero attempts) and `bootstrap-failed` (fingerprint present), each with its own exit code, each producing an artifact at the predictable run-store location, and neither hanging (C7, C22, C26).
- [ ] The ADR records the store-open-failure case: no artifact is possible, the error goes to stderr with the sink-failure exit code, and the spec promises nothing more (arch.md "The shape of a run").
- [ ] The ADR fixes the per-pool cost-vector types: one entry per admission pool in native units — bytes for the memory pool, a thread count for each thread pool (C5, C12).
- [ ] The ADR fixes the memory-cost split into working memory (held for the attempt, released at terminal state) and output residency (transferred to the output slot on production, released when the last consumer is terminal — C10) (C5).
- [ ] The ADR records the conservative default of zero declared cost applied uniformly, so a node with no stated cost behaves identically to one with an all-zero cost written out, and every node's effective cost appears in the assembly/graph artifact including defaulted values (C5).

- [ ] The ADR fixes the bootstrap cost-fit and resource/parameter rejection contract: a declared cost no pool can satisfy, a missing declared resource, and an invalid parameter are each rejected at bootstrap before any node executes, each with a distinct complete error report and the bootstrap-failure artifact — and confirms assembly still owns machine-independent checks including the invalid execution-class override (C5, C7, C12).

- [ ] The ADR names the rejected alternatives (at minimum a single scalar cost, and undifferentiated memory without the working/residency split) with the reason each was rejected, and states the reopen condition if any prototype below cannot be made to express the chosen seam or cost model.

## Out of scope
- The run-store on-disk format, the sink contract, and store durability (C18/C19) — owned by the T0.6 run-store ADR; this ticket references the store only as the place the bootstrap-failure artifact lands and only fixes the *type shape* of the seam, jointly with T0.6, not the store itself.
- The artifact serialization format and schema versioning (C19/C20/C22) — decided in T4; this ticket names the `assembly-failed`/`bootstrap-failed` *variants* and their contents, not their encoding.
- The full assembly validation and precomputation implementation (C7) — that is T14; here we fix only the assembly-output interface and the assembly/bootstrap check partition it must honor.
- The full node-policy struct and registry (C5) — that is T29; here we fix only the declared-cost vector's types and defaults, not the retry/backoff/timeout/trigger/durability fields' implementation.
- The admission controller's pool weighting, permit lifecycle, oldest-ready-first admission, and zombie accounting (C12) — that is T31; and container-limit probing (cgroup v2→v1→host, headroom, pinning) is T32. This ticket fixes only the cost-vector units and the cost-fit-at-bootstrap rejection contract those tickets consume.
- Measured-cost capture and the declared-versus-measured juxtaposition in the run artifact (C23) — implemented later; referenced here only as the honesty rule the cost model must support.
- Any temptation to let assembly see parameter values, to let the graph shape or declared cost change at runtime, to add cross-run coordination of pool capacity between processes, or to turn the run store into a metadata store or scheduler — dagr's graph shape never changes at runtime, it is one run per container with no cross-process coordination, and it is never a scheduler or a metadata store; those boundaries are named so the seam cannot smuggle them in.
