# 012 · T0.6 — ADR: run store contract

> **Milestone:** M0 · **Size:** M · **Type:** decision · **Components:** C18, C19
> **Branch:** `adr/t0.6-run-store-contract-adr` · **Depends on:** — · **Blocks:** T19, T24, T27, T36, T53, T54a, T58

## Why / context
The run store is the single piece of operator-supplied infrastructure dagr asks for, and everything a run leaves behind — the event stream, both artifacts, and per-node scratch — lives under one base location the operator points at storage that survives the container (arch.md "The shape of a run"). This ADR locks that contract *before* it leaks backward into M1: the event-stream writer (T19), the run-loop driver that mints identity and opens the store before assembly (T24), the crash-safety fault-injection suite (T27), the signal/final-flush path (T36), the scratch store (T53/T54a), and resume (T58) all bind against the sink shape, the directory layout, the run-identity scheme, and the write-failure/flush/prune semantics decided here. It governs C18 (durable scratch store) and C19 (event stream). No writer, no scratch KV, and no CLI verb is written until this decision names the seam each of them targets, so the ADR's deliverable is a committed decision record that leaves those seams non-negotiable.

## Objective
Record, as a committed Architecture Decision Record at the ticket `path`, the complete run store contract so every downstream consumer inherits one boundary instead of re-deciding it. Concretely, the ADR must fix:

- **The sink trait.** The two-operation injected sink through which the event stream is written — append a single line, and flush — with a named default implementation (a local file under the run's directory) and the seam by which an operator or a test supplies an alternative. Define what "append a line" guarantees and what "flush" guarantees at the trait level versus what is delegated to the concrete sink.
- **The base-location surface.** The one operator-supplied base: a local path by default, overridable by a library-owned flag *and* an environment variable, living in the reserved library-flag namespace (C26) so it can never collide with a pipeline parameter. Fix the precedence between flag, environment variable, and the built-in default.
- **The directory layout.** The `<base>/<pipeline>/<run-id>/` layout under which the event stream, both artifacts, and scratch live; which file names are reserved; and the guarantee that two runs (even of the same binary, same pipeline, concurrently) write disjoint directories and therefore disjoint files.
- **Run identity.** UUIDv7 run ids, operator-overridable, minted at bootstrap *before* assembly executes for the run verbs; the property that time-ordered ids keep a listing of a pipeline's runs naturally sorted; and the rule that an operator-supplied id is honored verbatim.
- **The event-write-failure path.** Two distinct cases: failure to *open* the store (nowhere to write — error to stderr with the sink-failure exit code, nothing more promised), and a *mid-run* sink failure (run moves to cancelling with reason "event stream unwritable," best-effort final report to stderr, exit with the distinct sink-failure code). Name the exit code by cause (C26), not by number.
- **Flush semantics.** "Flushed" means no user-space buffering — each record is written to the sink before its transition counts as recorded — with an fsync at run end and at cancellation; per-event fsync is the sink's business and the default local-file sink does not do it; and a reader tolerates and discards at most one trailing partial record.
- **Prune / retention semantics.** Retention is operator-owned: nothing is deleted implicitly at run end; the prune verb (C26) is the only implicit-deletion path, removing old runs by count or by age, including the scratch of nodes that did not succeed. State the unit prune operates over (the per-run directory) and that it is the sole mechanism that reclaims non-succeeded scratch.
- **Scratch placement and lifecycle within the store.** Where per-run, per-node scratch lives under the run directory; that it is namespaced by run and node so two nodes cannot collide and one node cannot reach another's; that a succeeded node's scratch is deleted while a non-succeeded node's scratch is retained for a later resume to copy forward; and that scratch I/O failure is a retry-eligible task failure. (C18's *implementation* is T53; this ADR fixes only the placement/lifecycle/isolation contract it must honor.)
- **The record identity carried by every event.** That every event record carries run identity and schema version so records from concurrent runs concatenate and partition safely, sequence numbers are gapless and strictly increasing within a run, and each record carries both a wall-clock stamp (informational) and a monotonic offset from run start (authoritative for durations) — so T19 inherits the record header shape.
- **Record it.** Write the ADR with status, context, decision, consequences, and rejected alternatives; state the reopen condition; and name each blocked ticket's seam.

## Decision

### 1. The injected sink — exactly two operations

The event stream is written through an **injected sink** with **exactly two
operations** and no third:

| Operation | Trait-level guarantee |
|---|---|
| **append a line** | Hands the sink one complete record as a single line (a byte slice ending in the record terminator). The trait guarantees **line atomicity at the framework boundary**: the writer never hands the sink a partial record and never interleaves two records, so any bytes the sink accepts are a whole line or nothing. What the sink does with that line — whether the underlying write is itself atomic against a concurrent reader or a crash — is the concrete sink's business. |
| **flush** | Guarantees that every line **already appended** has left the framework's user-space buffers and been handed to the sink (there is no user-space buffering to begin with — see §6 — so `flush` is the point at which the writer additionally asks the sink to make its accepted bytes durable, e.g. `fsync`). The trait guarantees the framework has nothing buffered; **stronger per-line durability is delegated to the concrete sink.** |

These two guarantees are stated **separately from what a concrete sink may
additionally provide.** A concrete sink is free to fsync on every append, to
mirror to a second location, or to compress — none of that is promised by the
trait, and no consumer may rely on it.

Illustrative shape (not shipping code; T19 owns the real definition):

```rust
/// The event stream is written through this. Two operations, no more.
pub trait EventSink: Send {
    /// Append one complete record as a single line. Line-atomic at the
    /// framework boundary: the caller never passes a partial or interleaved
    /// line.
    fn append_line(&mut self, line: &[u8]) -> io::Result<()>;

    /// Ensure everything already appended has been handed to the sink and
    /// (for durable sinks) made durable. No user-space buffering remains.
    fn flush(&mut self) -> io::Result<()>;
}
```

**The named default sink is a local file under the run directory** —
`<base>/<pipeline>/<run-id>/events.jsonl` (§3) — opened append-only. It does
**not** fsync per append (§6).

**The injection seam** is a bootstrap-time parameter: the run-loop driver (T24)
constructs the default local-file sink from the resolved base location, and an
operator or a test supplies an alternative by providing a different `EventSink`
at the same seam (a library-owned constructor hook, not a pipeline-parameter
surface). The seam exists so the crash-safety suite (T27) can inject a failing
sink and so an operator can point the stream at a different local target without
touching the pipeline; it is **not** a network-store abstraction (see
**Rejected alternatives**).

### 2. The base-location surface and its precedence

There is exactly **one** operator-supplied base: a **local path**. It is
resolved from three sources, in this precedence order (highest wins):

1. **The library-owned flag** — e.g. `--dagr-run-store <path>` — highest
   precedence, because an explicit invocation-time flag is the most specific
   intent.
2. **The environment variable** — e.g. `DAGR_RUN_STORE` — next, because it is
   the ambient container/orchestrator configuration.
3. **The built-in default** — a local path under the working directory (e.g.
   `./.dagr/runs`) — lowest, used when neither of the above is set.

The flag **lives in the reserved library-flag namespace (C26)** — the `--dagr-*`
prefix set aside for library-owned flags — so **no pipeline parameter can ever
shadow or collide with it**: pipeline parameters are typed arguments in a
disjoint namespace, and the collision is rejected at bootstrap (C26). The exact
flag and env-var spellings are fixed by the CLI ticket (T55) within this reserved
namespace; what this ADR fixes is that both exist, that the flag is reserved, and
that the precedence is flag > env > default.

### 3. The directory layout and reserved file names

Every run's outputs live under one directory:

```
<base>/<pipeline>/<run-id>/
    events.jsonl        # the C19 event stream (reserved)
    graph.json          # the C20 graph artifact (reserved)
    run.json            # the C22 run artifact (reserved)
    scratch/            # per-node scratch subtree (reserved; §9)
        <node>/…
```

`<pipeline>` is the pipeline identity (its stable name); `<run-id>` is the
UUIDv7 run id (§4). The **reserved file names** are `events.jsonl`, `graph.json`,
`run.json`, and the `scratch/` subtree — reserved so no task and no future
component collides with them; the concrete extensions/encodings are confirmed by
T4/T19 (this ADR reserves the *names and roles*, not the byte encodings, per Out
of scope). A node's own working files go under a per-run temp directory
convention (arch.md "Teardown"), **not** under these reserved names.

**The disjoint-directory guarantee.** Because the path embeds both the pipeline
identity and a run-unique UUIDv7, **two runs — even of the same binary, the same
pipeline, started concurrently — write disjoint directories and therefore
disjoint files.** No two runs ever contend for the same path; this is the
mechanism behind C19's acceptance "two simultaneous runs of the same binary write
disjoint files and both produce valid streams." No lock, no coordinator, no
shared index is involved — the uniqueness of the run id is the whole mechanism.

### 4. Run identity

Run identity is a **UUIDv7**, **operator-overridable**, **minted at bootstrap —
before assembly executes — for the run verbs** (per T0.5's ordering; the store
and stream open on it so even an assembly failure has a place to record itself).

- **Time-ordered ⇒ naturally sorted.** UUIDv7 embeds a millisecond timestamp in
  its high bits, so a lexical listing of a pipeline's run directories is also a
  chronological listing — `ls <base>/<pipeline>/` sorts runs oldest-to-newest
  with no index and no metadata store. This is why UUIDv7 is chosen over UUIDv4:
  the sort is a free property of the id, not a feature the tool has to build.
- **Operator override is honored verbatim.** An operator who supplies a run id
  (e.g. to match an external orchestrator's job id) has that id used **exactly as
  given** — the tool does not validate it into UUIDv7 shape, re-hash it, or
  prefix it; it becomes `<run-id>` in the path verbatim. The natural-sort
  property is then the operator's to preserve or forgo.

### 5. The two write-failure paths

Two **distinct** cases, each with a stated, bounded promise:

- **Store-open failure (nowhere to write).** If opening the store itself fails —
  the base path is unwritable, the directory cannot be created — there is
  **nowhere to write an artifact by definition**. The error goes **to stderr**
  with the **sink-failure exit code**, and **the spec promises nothing more** (no
  artifact, because there is no place to put one). This matches arch.md verbatim.
- **Mid-run sink failure.** If the sink fails *after* the store opened — a
  disk-full mid-run, a failing injected sink — it is a **run-level fault**: the
  run **moves to cancelling** with reason **"event stream unwritable,"** makes a
  **best-effort final report to stderr**, and **exits with the distinct
  sink-failure code**. A pipeline that cannot record what it did should stop
  doing things.

**Exit codes are named by cause, not by number (C26).** This ADR names the
**sink-failure** cause; the concrete numeric value and its place in the exit-code
precedence table are fixed by the CLI ticket (T55). Both failure paths share the
same *cause* (sink-failure) because both are "the event record could not be
written"; they differ in *what else* is promised (an open failure promises no
artifact; a mid-run failure promises a best-effort stream and stderr report).

### 6. Flush semantics

- **No user-space buffering.** "Flushed" means the framework keeps **no
  user-space buffer** of records: **each record is written to the sink before its
  transition is considered recorded.** A transition that has not reached the sink
  has not happened as far as the record is concerned. Writing continuously rather
  than at exit is the whole point — the dominant container failure mode is an
  abrupt kill, and a record that only materializes on clean shutdown is absent
  exactly when it is most needed.
- **fsync at run end and at cancellation.** The framework issues an `fsync`
  (via the sink's `flush`) at **run end** and at **cancellation** — the two
  points where the stream is known-complete and must survive.
- **Per-event fsync is the sink's business; the default does not do it.**
  Stronger per-event durability is **delegated to the concrete sink**; the
  **default local-file sink does not fsync per event** (that cost is one the
  operator opts into by supplying a stricter sink — see **Rejected
  alternatives**). The spec does not pretend the last block always survives a
  power cut.
- **A reader tolerates and discards at most one trailing partial record.**
  Because the default sink does not fsync per event, an abrupt kill can leave a
  final record half-written. A conforming reader **tolerates and discards at most
  one trailing partial record**; every record before it is valid and parseable.
  This is the contract T27's kill-point test asserts (C19 acceptance: "every
  record but at most one trailing partial is valid and parseable").

### 7. The per-record header

Every event record carries a fixed header so records from concurrent runs
**concatenate and partition safely**:

| Field | Role |
|---|---|
| **run identity** | The UUIDv7 (§4), on **every** record — so a concatenation of two runs' streams partitions cleanly by run id. |
| **schema version** | On every record — so a reader knows how to parse it and evolution stays additive (T4 owns the encoding). |
| **sequence number** | A **gapless, strictly increasing** integer **within a run** — a reader detects a lost record as a gap. |
| **wall-clock stamp** | **Informational only.** Never used for durations. |
| **monotonic offset from run start** | **Authoritative.** Durations are computed from offsets, never from wall clocks — offsets are immune to clock skew and NTP steps. |

T19 inherits this header shape verbatim. The gapless-sequence *machinery* and the
run-started event that carries the full artifact header are T19's to build; this
ADR fixes the **fields and their guarantees** only.

### 8. Prune / retention semantics

- **Nothing is deleted implicitly at run end.** A run leaves its whole directory
  behind — event stream, artifacts, and the scratch of every non-succeeded node —
  and the tool deletes **nothing** on its own schedule.
- **Prune (C26) is the SOLE implicit-deletion path.** The **prune verb** is the
  only mechanism that reclaims space. It operates over **whole per-run
  directories**, removing old runs **by count or by age**. Nothing else in the
  tool ever deletes a run directory.
- **Prune is the only mechanism that removes non-succeeded scratch.** Because a
  succeeded node's scratch is deleted at success (§9) but a non-succeeded node's
  scratch is retained for a possible resume, the retained scratch is reclaimed
  **only** when prune removes the whole run directory. Retention is therefore
  entirely operator-owned: the operator chooses the prune policy; the tool never
  second-guesses it. (An implicit "delete on the tool's schedule" policy is a
  rejected alternative — see below.)

### 9. Scratch placement, isolation, and lifecycle

- **Placement.** Per-run, per-node scratch lives under the run directory at
  `<base>/<pipeline>/<run-id>/scratch/<node>/…`. When the operator has pointed
  the base at storage that survives the container, scratch survives with it (the
  basis for T54a); on ephemeral local disk it does not, and such a run is simply
  not resumable.
- **Namespacing and enforced isolation.** Scratch is **namespaced by run and by
  node** so that **two nodes cannot collide** and **one node cannot reach
  another's** scratch. The isolation is **enforced, not conventional**: the
  context hands each node a scratch handle scoped to its own `<node>` subtree,
  with no API to name another node's namespace. (The enforcement *mechanism* —
  the KV, the handle — is T53; this ADR fixes that the isolation is a guarantee,
  not a convention.)
- **Lifecycle.** A **succeeded** node's scratch is **deleted** when the node
  succeeds — its checkpoints have served their purpose. A **non-succeeded**
  node's scratch (any terminal state that is not `succeeded`, per §T0.4-class
  reading) is **retained** in the run's directory — that retained scratch is
  exactly what a later **resume copies forward** into the new run's namespace so
  a re-executing node continues from its checkpoint. Retained scratch is removed
  only by prune (§8).
- **Failure classification.** A scratch **read or write failure** is classified a
  **retry-eligible task failure** — disk trouble is transient more often than
  not, so the node's retry budget absorbs it.

### 10. Concurrency posture

- **Two simultaneous runs are safe with respect to the run store.** Per-run
  directories **never collide** (§3), so two runs of the same binary on one
  machine write **disjoint files** and each produces a valid, gapless stream.
- **Cross-run analysis is safe concatenation.** Because every record carries the
  run identity (§7), analysis across runs is **concatenation partitioned by run
  identity** — no cross-run index and no shared metadata are needed or built.
- **The tool coordinates nothing between processes.** The supported model is
  **one run per container**. Concurrent runs on a shared host are safe *for the
  store*, but the tool does **not** coordinate between processes — no shared lock,
  no cross-process handshake — because "that road ends in building a scheduler"
  (arch.md "Operational model"). Splitting *capacity* across co-located runs is
  the operator's call via the pool-pinning flag (C12), outside this contract.

**Scope boundary (permanent non-goals).** Everything above describes a **local,
file-based, embedded** persistence contract: one local base path, a file-backed
default sink, per-run directories on a filesystem the operator supplies. It is
**not** a networked metadata service, **not** a distributed store, **not** a
cross-run index or database, and **not** a coordinated multi-process store. dagr
is permanently never a metadata store, a scheduler, or a distributed execution
system; this contract stays strictly inside that boundary and the **Rejected
alternatives** below name each temptation that would breach it.

## Consequences

**Each blocked ticket inherits a named seam and reopens no question this ADR
closed:**

- **T19** (029 — event stream writer, C19) binds against **the per-record header
  (§7) and the two-operation sink (§1)**: it implements the gapless-sequence
  machinery, the wall-clock + monotonic-offset fields, run identity + schema
  version on every record, and the run-started event carrying the full artifact
  header — writing every line through this sink's `append_line`/`flush`.
- **T24** (034 — run-loop driver) binds against **identity-minted-and-store-
  opened-before-assembly (§4, §5) and the sink injection seam (§1)**: it mints
  the UUIDv7 at bootstrap, resolves the base by the §2 precedence, opens the
  store and stream before assembly executes, and constructs the default
  local-file sink at the injection seam.
- **T27** (037 — crash-safety and fault-injection) binds against **the kill-point
  and induced-sink-failure semantics (§5, §6)**: it kills the process at random
  points and asserts ≤1 trailing partial record with gapless sequences, and it
  injects a failing sink to prove the mid-run "event stream unwritable"
  cancellation path and the distinct sink-failure exit code.
- **T36** (046 — signals, final flush, temp cleanup, C16) binds against **the
  final fsync (§6) and the bounded-wait-on-unwritable-sink-at-shutdown (§5)**: on
  SIGTERM/SIGINT it flushes a complete stream (fsync) before exit within the
  budget, and if the sink is unwritable at shutdown it waits a bounded time then
  exits with the sink-failure code rather than hanging.
- **T53** (065 — durable scratch store, C18) binds against **the scratch
  placement, isolation, and lifecycle contract (§9)**: it implements the
  per-run/per-node namespaced KV under the run directory with enforced cross-node
  isolation, deletion on success, retention otherwise, and retry-eligible I/O
  failure — exactly the contract this ADR fixes.
- **T54a** (066 — scratch survives restart) binds against **scratch-survives-
  restart-and-is-pruned-not-implicitly-deleted (§8, §9)**: scratch under the
  run-store base survives a full process restart, retained for non-succeeded
  nodes, nothing deleted implicitly at run end, prune removes it.
- **T58** (070 — resume core, C27) binds against **scratch copy-forward and
  store-required-for-resume (§9, and arch.md "resume requires the original run to
  have used a run store")**: resume copies forward the retained scratch of every
  re-executing node into the new run's namespace, and refuses when the prior
  run's store is gone.

**Coverage matrix: no change.** C18 and C19 remain `unmapped` in
`docs/coverage-matrix.md`, deferred to their covering-test tickets (C18 → T53,
C19 → T19). A decision ticket owes no covering test; the covering tests land with
the component tickets above, which will each replace `unmapped` with a test id
per the matrix's per-ticket duty. This ADR makes **no edit** to the coverage
matrix or to the criteria-matrix partition (T0.10's).

**Reopen condition.** If a downstream ticket cannot honor a seam as written — for
example, if the two-operation sink proves insufficient for the record writer, if
the UUIDv7-in-path layout cannot deliver the disjointness guarantee on a real
filesystem, or if the succeeded-node-scratch-deletion races a zombie thread in a
way T0.3's discard rule does not already cover — **the contract reopens here**, in
this ADR, rather than being worked around locally. A local workaround that
silently diverges from this contract is a defect, not a fix.

## Rejected alternatives

- **A buffer-until-exit event writer** (accumulate records in memory, write the
  stream on clean shutdown). **Rejected:** the dominant container failure mode is
  an abrupt kill with no chance to run an exit handler, so a record that only
  materializes on clean shutdown is **absent exactly when it is most needed**.
  The whole point of the stream is to be crash-proof (C19); write-through (§6) is
  non-negotiable.
- **A per-event-fsync default** (fsync the stream on every append by default).
  **Rejected as a default:** per-event fsync is a real durability cost the
  operator should **opt into by supplying a stricter sink** (§1, §6), not one the
  tool imposes on every run. The default local-file sink fsyncs at run end and
  cancellation only; the ≤1-trailing-partial tolerance (§6) is the honest
  statement of what that buys.
- **A shared cross-run index or metadata store** (a database or index over all
  runs under the base, to list/query runs centrally). **Rejected on the scope
  boundary:** a cross-run index is **a metadata store dagr will never be**
  (arch.md permanent non-goals). Cross-run analysis is concatenation partitioned
  by the run identity every record carries (§7, §10), and time-ordered UUIDv7 ids
  give a naturally sorted listing (§4) with no index to build, corrupt, or keep
  consistent.
- **A coordinated multi-process store lock** (a lock or handshake over the shared
  base so co-located runs coordinate). **Rejected on the scope boundary:**
  cross-process coordination is **the road to a scheduler** (arch.md "Operational
  model"). Disjoint per-run directories (§3, §10) make coordination unnecessary —
  two runs never contend for a path — so the tool coordinates nothing between
  processes.
- **An implicit retention policy** (delete old runs on the tool's own schedule).
  **Rejected:** retention stays **operator-owned via prune** (§8). The tool
  deleting a run the operator wanted is a worse failure than a full disk the
  operator can prune; nothing is deleted implicitly at run end.
- **A network-backed store abstraction beyond the injected sink** (an S3/HTTP
  store trait the framework calls directly). **Rejected on the scope boundary:**
  the only store abstraction is the local base path plus the two-operation sink
  (§1, §2). Durability across the container is achieved by the operator pointing
  the **local base** at storage that survives (a mounted volume, a synced
  directory) — the tool's one infrastructure ask — not by the framework speaking
  a network protocol.

*(Reopen condition stated in §Consequences: if a downstream ticket cannot honor a
seam as written, the contract reopens here rather than being worked around
locally.)*
