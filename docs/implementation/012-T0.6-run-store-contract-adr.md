# 012 · T0.6 — ADR: run store contract

> **Milestone:** M0 · **Size:** M · **Type:** decision · **Components:** C18, C19
> **Branch:** `adr/t0.6-run-store-contract-adr` · **Depends on:** — · **Blocks:** T19, T24, T27, T36, T53, T54a, T58

## Why / context
The run store is the single piece of operator-supplied infrastructure dagr asks for, and everything a run leaves behind — the event stream, both artifacts, and per-node scratch — lives under one base location the operator points at storage that survives the container (arch.md "The shape of a run"). This ADR locks that contract *before* it leaks backward into M1: the event-stream writer (T19), the run-loop driver that mints identity and opens the store before assembly (T24), the crash-safety fault-injection suite (T27), the signal/final-flush path (T36), the scratch store (T53/T54a), and resume (T58) all bind against the sink shape, the directory layout, the run-identity scheme, and the write-failure/flush/prune semantics decided here. It governs C18 (durable scratch store) and C19 (event stream). No writer, no scratch KV, and no CLI verb is written until this decision names the seam each of them targets, so the ADR's deliverable is a committed decision record that leaves those seams non-negotiable.

## Objective
Record, as a committed Architecture Decision Record at the ticket `path`, the complete run store contract so every downstream consumer inherits one boundary instead of re-deciding it. Concretely, the ADR must fix:

- **The sink trait.** The two-operation injected sink through which the event stream is written — append a single line, and flush — with a named default implementation (a local file under the run's directory) and the seam by which an operator or a test supplies an alternative. Define what "append a line" guarantees and what "flush" guarantees at the trait level versus what is delegated to the concrete sink.
- **The base-location surface.** The one operator-supplied base: a local path by default, overridable by a library-owned flag *and* an environment variable, living in the reserved library-flag namespace (C26) so it can never collide with a pipeline parameter. Fix the precedence between flag, environment variable, and the built-in default.
- **The directory layout.** The `<base>/<pipeline>/<run-id>/` layout under which the event stream, both artifacts, and scratch live; which file names are reserved; and the guarantee that two runs (even of the same binary, same pipeline, concurrently) write disjoint directories and therefore disjoint files.
- **Run identity.** UUIDv7 run ids, operator-overridable, minted at bootstrap *before* assembly executes for the run verbs; the property that time-ordered ids keep a listing of a pipeline's runs naturally sorted; and the rule that an operator-supplied id is honored verbatim.
- **The event-write-failure path.** Two distinct cases: failure to *open* the store (nowhere to write — error to stderr with the sink-failure exit code, nothing more promised), and a *mid-run* sink failure (run moves to cancelling with reason "event stream unwritable," best-effort final report to stderr, exit with the distinct sink-failure code). Name the exit code by cause (C26), not by number.
- **Flush semantics.** "Flushed" means no user-space buffering — each record is written to the sink before its transition counts as recorded — with an fsync at run end and at cancellation; per-event fsync is the sink's business and the default local-file sink does not do it; and a reader tolerates and discards at most one trailing partial record.
- **Prune / retention semantics.** Retention is operator-owned: nothing is deleted implicitly at run end; the prune verb (C26) is the only implicit-deletion path, removing old runs by count or by age, including the scratch of nodes that did not succeed. State the unit prune operates over (the per-run directory) and that it is the sole mechanism that reclaims non-succeeded scratch.
- **Scratch placement and lifecycle within the store.** Where per-run, per-node scratch lives under the run directory; that it is namespaced by run and node so two nodes cannot collide and one node cannot reach another's; that a succeeded node's scratch is deleted while a non-succeeded node's scratch is retained for a later resume to copy forward; and that scratch I/O failure is a retry-eligible task failure. (C18's *implementation* is T53; this ADR fixes only the placement/lifecycle/isolation contract it must honor.)
- **The record identity carried by every event.** That every event record carries run identity and schema version so records from concurrent runs concatenate and partition safely, sequence numbers are gapless and strictly increasing within a run, and each record carries both a wall-clock stamp (informational) and a monotonic offset from run start (authoritative for durations) — so T19 inherits the record header shape.
- **Record it.** Write the ADR with status, context, decision, consequences, and rejected alternatives; state the reopen condition; and name each blocked ticket's seam.

## Test plan (write these first — TDD)
These are the decision-record and evidence checks that validate the contract. Each is an independently checkable assertion against the committed ADR or a small evidence check the downstream tickets will inherit; no shipping run-store code is built here.

- **Sink-trait shape record check.** Setup: the committed ADR. Action: read the decision section defining the sink. Expected outcome: the sink is exactly two operations — append one line, flush — with a named default (local file under the run directory) and an explicit injection seam; the append operation's line-atomicity guarantee and the flush operation's guarantee are stated separately from what a concrete sink may additionally provide.
- **Base-location precedence record check.** Setup: the ADR. Action: read the base-location section. Expected outcome: a local path default, a library-owned flag, and an environment variable are all named; their precedence order is stated unambiguously; and the flag is declared to live in the reserved library-flag namespace so a pipeline parameter can never shadow it (C26).
- **Directory-layout record check.** Setup: the ADR. Action: read the layout section. Expected outcome: the `<base>/<pipeline>/<run-id>/` path is specified; the reserved file names for the event stream and each artifact are listed; and the disjoint-directory guarantee for two concurrent same-binary runs is stated as the mechanism behind C19's "two simultaneous runs write disjoint files."
- **Run-identity record check.** Setup: the ADR. Action: read the run-identity section. Expected outcome: run identity is UUIDv7, operator-overridable, minted at bootstrap before assembly for the run verbs; the natural-sort property of time-ordered ids is called out; and an operator-supplied override is stated to be honored verbatim.
- **Open-failure path record check.** Setup: the ADR. Action: read the failure-path section for the store-open case. Expected outcome: failure to open the store writes to stderr with the sink-failure exit code and the spec promises nothing more — matching arch.md's "if opening the store itself fails, there is nowhere to write an artifact by definition."
- **Mid-run sink-failure path record check.** Setup: the ADR. Action: read the failure-path section for the mid-run case. Expected outcome: a mid-run sink failure moves the run to cancelling with reason "event stream unwritable," makes a best-effort final report to stderr, and exits with the distinct sink-failure code — the contract T27's induced-failure test will assert against (C19 acceptance: "an induced mid-run sink failure cancels the run and exits with the sink-failure code").
- **Flush-semantics record check.** Setup: the ADR. Action: read the flush section. Expected outcome: "flushed" is defined as no user-space buffering with each record written to the sink before its transition is recorded; fsync happens at run end and at cancellation; per-event fsync is delegated to the sink and the default local-file sink does not do it; and a reader tolerates and discards at most one trailing partial record — the contract T27's kill-point test asserts (C19 acceptance: "every record but at most one trailing partial is valid and parseable").
- **Gapless-sequence and record-identity record check.** Setup: the ADR. Action: read the record-header section. Expected outcome: every record carries run identity and schema version; sequence numbers are gapless and strictly increasing within a run; and each record carries both a wall-clock stamp (informational) and a monotonic offset (authoritative) — so a later concatenation of concurrent runs partitions safely (C19 acceptance).
- **Prune / no-implicit-deletion record check.** Setup: the ADR. Action: read the retention section. Expected outcome: nothing is deleted implicitly at run end; prune (C26) is the sole implicit-deletion path, operating over whole per-run directories by count or age; and prune is named as the only mechanism that removes the scratch of non-succeeded nodes — the contract T54a inherits ("nothing deleted implicitly at run end; prune removes it").
- **Scratch placement / isolation / lifecycle record check.** Setup: the ADR. Action: read the scratch section. Expected outcome: scratch lives under the run directory, namespaced by run and node such that two nodes cannot collide and one node cannot read another's; a succeeded node's scratch is deleted while a non-succeeded node's is retained for resume copy-forward; and scratch I/O failure is classified retry-eligible — the contract T53 implements and T58's resume copies forward (C18 acceptance).
- **Concurrent-runs safety record check.** Setup: the ADR. Action: read the concurrency section against arch.md's operational model. Expected outcome: two simultaneous runs on one machine are stated to be safe with respect to the run store because per-run directories never collide, and cross-run analysis is concatenation partitioned by the run identity every record carries — matching the "one run per container" model without implying any cross-process coordination.
- **Blocked-seam coverage record check.** Setup: the ADR. Action: read the consequences section. Expected outcome: each of T19, T24, T27, T36, T53, T54a, and T58 is named with the specific seam of this contract it binds against (record header and sink for T19; identity-minted-and-store-opened-before-assembly for T24; kill-point and induced-sink-failure semantics for T27; final fsync and bounded-wait-on-unwritable-sink-at-shutdown for T36; scratch placement/isolation/lifecycle for T53; scratch survives restart and is pruned not implicitly deleted for T54a; scratch copy-forward and store-required-for-resume for T58) — so no downstream ticket reopens a question this ADR closed.
- **Rejected-alternatives record check.** Setup: the ADR. Action: read the rejected-alternatives section. Expected outcome: at least a buffer-until-exit event writer, a per-event-fsync default, a shared cross-run index/metadata store, and a coordinated multi-process store lock are named with the reason each was rejected (a record that only materializes on clean shutdown is absent exactly when needed; per-event fsync is a cost the operator opts into via the sink, not a default; a cross-run index is a metadata store dagr will never be; cross-process coordination is the road to a scheduler) — proving the decision was made against considered options and against the scope boundary.

## Definition of done
- [ ] The ADR defines the injected sink as exactly two operations — append a line, flush — names the default local-file sink under the run directory, and specifies the injection seam by which an operator or test supplies an alternative (arch.md "The shape of a run"; C19).
- [ ] The ADR fixes the base-location surface: a local path default, a library-owned flag in the reserved namespace, and an environment variable, with an unambiguous precedence order and a stated guarantee that no pipeline parameter can shadow the flag (C26).
- [ ] The ADR specifies the `<base>/<pipeline>/<run-id>/` directory layout, the reserved file names for the event stream and both artifacts, and the disjoint-directory guarantee that makes two concurrent same-binary runs write disjoint files (C19 acceptance: "two simultaneous runs write disjoint files and both produce valid streams").
- [ ] The ADR fixes run identity as UUIDv7, operator-overridable, minted at bootstrap before assembly executes for the run verbs, and records the natural-sort property and the honor-override-verbatim rule (C19; arch.md "The shape of a run").
- [ ] The ADR records both write-failure paths: store-open failure goes to stderr with the sink-failure exit code and promises nothing more; a mid-run sink failure moves the run to cancelling with reason "event stream unwritable," makes a best-effort stderr report, and exits with the distinct sink-failure code (C19 acceptance: "an induced mid-run sink failure cancels the run and exits with the sink-failure code").
- [ ] The ADR defines flush semantics: no user-space buffering, each record written to the sink before its transition is recorded, fsync at run end and at cancellation, per-event fsync delegated to the sink (default local-file sink does not do it), and a reader that tolerates and discards at most one trailing partial record (C19 acceptance: "killing the process abruptly leaves a stream whose every record but at most one trailing partial is valid and parseable").
- [ ] The ADR fixes the per-record header: run identity and schema version on every record, gapless strictly-increasing sequence numbers within a run, and both a wall-clock stamp (informational) and a monotonic offset (authoritative) — so records from concurrent runs concatenate and partition safely (C19 acceptance: "every record carries the run identity and schema version"; "sequence numbers are gapless and strictly increasing").
- [ ] The ADR fixes retention semantics: nothing is deleted implicitly at run end, and the prune verb (C26) is the sole implicit-deletion path, operating over whole per-run directories by count or age and removing the scratch of non-succeeded nodes (C18; C19: "retention is handled by the prune verb; nothing is deleted implicitly").
- [ ] The ADR fixes scratch placement and lifecycle within the store: scratch lives under the run directory, namespaced by run and node so two nodes cannot collide and one cannot read another's; a succeeded node's scratch is deleted while a non-succeeded node's is retained for resume copy-forward; scratch I/O failure is classified retry-eligible (C18 acceptance: keys namespaced by run and node; cross-node isolation enforced; succeeded-node scratch gone; I/O failure retry-eligible).
- [ ] The ADR records the concurrency posture: per-run directories never collide, cross-run analysis is safe concatenation partitioned by run identity, and the tool coordinates nothing between processes (arch.md "Operational model," "one run per container").
- [ ] The ADR's consequences section names each blocked ticket — T19, T24, T27, T36, T53, T54a, T58 — with the specific seam of this contract it binds against, leaving no run-store question open to any of them.
- [ ] The ADR names the rejected alternatives (at minimum buffer-until-exit writing, per-event-fsync default, a cross-run index/metadata store, and a coordinated multi-process store lock) with the reason each was rejected, and states the reopen condition: if a downstream ticket cannot honor a seam as written, the contract reopens here rather than being worked around locally.

## Out of scope
- The event-stream writer implementation — the record encoding, the actual gapless-sequence machinery, and the run-started event carrying the full artifact header — is T19; this ADR fixes only the sink shape, record header, and flush contract it binds against.
- The concrete artifact serialization format and schema-version field semantics (JSONL events, JSON artifacts) — that is T4, which lands *after* this ADR; here we fix only that every record carries a schema version and that layout reserves the artifact file names, not the encodings.
- The run-loop driver, run-started/run-finished emission, and the bootstrap sequencing of mint-identity-then-open-store-then-assemble — that is T24; this ADR fixes only the ordering contract (identity and store open before assembly) it must obey.
- The crash-safety and fault-injection suite (kill-points, disk-full, failing sinks) — that is T27; this ADR fixes only the crash-tolerance and sink-failure semantics the suite asserts, not the tests.
- The signal-handling, final-flush, and per-run temp-dir cleanup path — that is T36; this ADR fixes only the fsync-at-cancel and bounded-wait-on-unwritable-sink-at-shutdown contract, not the signal wiring.
- The scratch store implementation (the KV, the enforced isolation, the attempt-1-write/attempt-2-read behavior), its survival across process restart, and resume copy-forward — those are T53, T54a, and T54b/T58; this ADR fixes only the placement, isolation, and lifecycle contract they honor.
- The resume algorithm, durable-output references, and lineage — that is T58/C27; this ADR fixes only that resume requires a surviving store and that non-succeeded scratch is retained for copy-forward.
- Any temptation to add a cross-run metadata store or index, a coordinated multi-process lock over the shared base, an implicit retention policy that deletes runs on the tool's own schedule, or a network-backed store abstraction beyond the injected sink — dagr is never a metadata store, never a scheduler, and never coordinates distributed execution; retention stays operator-owned via prune, and the graph shape never changes at runtime. That boundary is named here so the run-store contract cannot smuggle it in.
