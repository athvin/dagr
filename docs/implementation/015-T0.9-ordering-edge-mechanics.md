# 015 · T0.9 — C4 ordering-edge mechanics

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C2, C4
> **Branch:** `adr/t0.9-ordering-edge-mechanics` · **Depends on:** — · **Blocks:** T12, T50

## Why / context
The spec (C2 · Handle, C4 · Ordering dependency) already chose Option A: ordering edges are declared at the downstream node's registration against handles of *already-registered* upstreams, so the backward-reference discipline that makes a data-edge cycle inexpressible extends unchanged to ordering edges. This is a decision ticket that locks the **API shape** enforcing that discipline and the **exact assertions** the wiring compile-failure suite (T12) will make for ordering-edge cycles, before either the ordering-edge implementation (T50) or the compile-fail suite (T12) is written. It records those choices as a checked-in ADR plus at least one skeletal (may be `#[ignore]`d or `trybuild`-stubbed) compile-fail fixture that proves the guarantee is structural, not a later validation pass. Nothing here implements the runtime propagation or graph-artifact rendering of ordering edges — only the constructive shape and the failure contract.

## Objective
Decide and record, as an ADR, the ordering-edge API surface that makes an ordering-edge cycle inexpressible by construction, and pin the compile-fail contract T12 must assert. Concretely:

- Fix the **registration-time API shape**: an ordering edge is attached only at the downstream node's registration, taking one or more existing handles (of any node, any value type) as ordering upstreams, with no argument or method anywhere that accepts a not-yet-registered node or adds an edge between two already-registered nodes after the fact.
- Confirm the **type-erasure rule** for ordering upstreams: an ordering edge ignores the handle's value type (it constrains sequence, not data), so a handle of any `T` is an acceptable ordering upstream, and mixing ordering upstreams of different value types is legal.
- Confirm that a node may carry **both** data dependencies and additional ordering edges, and that a node attached *only* by ordering edges receives **no value** (its task body sees no bound input), distinguishing this from the non-default-trigger-rule restriction which is *out of scope here* (owned by T50/C5 vocabulary).
- Specify the **cycle-inexpressibility argument** in the ADR: because a handle is obtainable only by registering a node, and an ordering edge accepts only existing handles, no expression can name a node that is not yet registered — so a back-edge to self or to a descendant cannot be written. State that this is structural (C2), not a runtime cycle-detection pass.
- Enumerate the **compile-fail assertions for T12** so the suite author can copy them: for each targeted misuse, the case name, the misuse it demonstrates, and the observable expectation (fails to compile; where a message is asserted, that both relevant type names or a curated diagnostic appear — pinned to the workspace toolchain per C28).
- Deliver **at least one skeletal compile-fail fixture** for the ordering-edge cycle case, wired to the same harness T12 will use (e.g. `trybuild`), demonstrating the guarantee compiles-fail today; it may be marked pending/ignored but must be executable by the suite.

## Test plan (write these first — TDD)
Because this is a decision ticket, the "tests" are the evidence that validates the chosen API shape and the failure contract. Each is an independently checkable check on the ADR or a throwaway/skeletal fixture.

- **Cycle to self is inexpressible.** Setup: a throwaway prototype pipeline where a single node attempts to declare an ordering edge against its own not-yet-returned handle. Action: attempt to compile it under the pinned workspace toolchain. Expected: it fails to compile because no handle for the node exists at the point of registration — recorded as a `trybuild` compile-fail fixture, and the ADR names this as case `ordering_edge_self_cycle`.
- **Cycle to a descendant is inexpressible.** Setup: a prototype where node A registers, then node B registers an ordering edge on A, then an attempt is made to have A depend (via ordering edge) on B. Action: compile. Expected: it fails to compile because A's registration is already closed and B's handle did not exist when A was registered; the ADR notes there is no post-hoc "add edge" API to reach back to A. Recorded as case `ordering_edge_back_edge`.
- **No forward reference exists.** Setup: inspect the decided API surface as described in the ADR. Action: enumerate every entry point that accepts an ordering upstream. Expected: every such entry point takes an existing handle value; the ADR asserts (and a reviewer can confirm against the surface) that none accepts a node name, index, string key, or a builder position that could denote an unregistered node — mirroring C2's "no lookup by name/index/string key."
- **No after-the-fact edge API exists.** Setup: read the API surface in the ADR. Action: search for any method that connects two already-registered nodes. Expected: none exists; the ADR states this explicitly as the enforcement of C4's "no API to add an edge between two existing nodes afterward."

Expected: it compiles — proving ordering edges are type-erased and the value type does not participate — and the ADR records this as a *positive* (compiles) fixture so a future regression that accidentally couples value type into ordering fails review.
- **Data-plus-ordering coexists; ordering-only yields no value.** Setup: a prototype where one node has both a data dependency and an extra ordering edge, and a second node has ordering edges only. Action: compile and inspect the second node's task-body input. Expected: both compile; the ordering-only node's body is given no bound value (its declaration reflects "receives no value" per C4), and the ADR documents where this is enforced structurally so T50 can rely on it.
- **Assertion table is complete and matches the acceptance criteria.** Setup: the ADR's compile-fail assertion table. Action: cross-check it against C2 and C4 acceptance criteria. Expected: every cycle-related criterion (C2 "cycle through data or ordering edges fails to compile, demonstrated by a checked-in compile-failure test"; C4 "no API to add an edge afterward") maps to at least one named case in the table, and each table row states case name, the misuse, and the observable outcome.

## Decision

### 1. The registration-time API shape — a backward reference, at the downstream node only

An ordering edge is declared **only at the downstream node's own registration**,
taking **one or more already-existing handles** as ordering upstreams. The
discipline, identical to a data edge's:

- **A handle is obtainable only by registering a node.** Registration returns the
  node's handle; the handle is the **only** way to refer to that node afterward
  (C2). There is **no** other constructor for a handle — no lookup by node name,
  index, or string key, and no way to fabricate a handle for a node that has not
  been registered.
- **Ordering upstreams are passed by handle, at registration.** The single
  entry point that attaches an ordering edge (illustratively `register`/the
  builder's ordering-edge method) takes ordering upstreams **as existing handle
  values**. Every ordering-upstream argument is therefore a handle that already
  exists — i.e. a node registered **strictly before** this one.
- **No API accepts an ordering upstream by name, index, or string key.** Mirroring
  C2's "no lookup by name/index/string key," **no** argument or method anywhere
  accepts a node name, a positional index, a string key, or a builder position
  that could denote a **not-yet-registered** node. The only currency is an
  existing handle value.

Illustrative shape (not shipping code; T13/T50 own the real definitions):

```rust
#[derive(Clone, Copy)]
struct Handle<T> { /* node identity + PhantomData<T> */ }

#[derive(Clone, Copy)]
struct OrderingHandle(/* NodeId — value type erased */);

impl<T> Handle<T> {
    // Any typed handle yields a type-erased ordering upstream; T is dropped.
    fn ordering(self) -> OrderingHandle { /* ... */ }
}

impl Flow {
    // An ordering edge is declared HERE, at the downstream node's own
    // registration, against already-existing OrderingHandles. The returned
    // Handle is the only way to refer to this node afterward.
    fn register<Out>(&mut self, ordering_upstreams: &[OrderingHandle], /* body */)
        -> Handle<Out> { /* ... */ }
    // There is deliberately NO `add_ordering_edge(existing, existing)`.
}
```

### 2. Type-erasure — any value type is an acceptable ordering upstream

An ordering edge constrains **sequence, not data**, so it **ignores the handle's
value type**:

- A handle of **any** `T` is an acceptable ordering upstream. The
  ordering-upstream argument type is a **type-erased** view (`OrderingHandle`
  above) obtained from a typed `Handle<T>` by dropping `T` while **keeping the
  node's identity** (C2). The value type does **not** participate.
- **Mixing ordering upstreams of differing value types is legal.** Because every
  ordering upstream erases to the same `OrderingHandle`, a downstream node may take
  ordering edges from a `Handle<Rows>` and a `Handle<Bytes>` in the same call, and
  it **compiles**. This is recorded as the **positive** fixture
  `ordering_edge_any_value_type_ok`: a future regression that accidentally coupled
  the value type into the ordering edge would make that fixture **fail to compile**
  and thus fail review.

### 3. Data + ordering coexist; an ordering-only node receives no value

- **A node may carry both.** A node may have **data dependencies and additional
  ordering edges** at the same registration (C4). The data dependencies bind
  typed inputs; the extra ordering edges are the type-erased ordering upstreams of
  §2. Nothing forces a node to be either purely data or purely ordering.
- **An ordering-only node receives no value.** A node attached **only** by ordering
  edges takes **no typed input**: its task body sees **no bound input value** (its
  input is the empty tuple `()` — it "receives no value," C4). This is
  **structural** — the ordering-only registration entry point takes no data
  handle, so there is no place for a value to enter — not a runtime check.
- **Where T50 enforces it.** This ADR fixes the *shape* (an ordering-only
  registration has no data-input parameter, so the body's input is `()`); **T50
  implements and enforces** it structurally in the real builder — the ordering-only
  registration path yields a task whose body input is `()`, and the graph artifact
  records the node's data-dependency list as empty. **T50 may rely on this shape.**
  This is recorded as the positive fixture
  `data_plus_ordering_and_ordering_only`.

### 4. Cycle inexpressibility — structural (C2), never a runtime pass

An ordering-edge **cycle is inexpressible by construction**, by exactly the C2
argument extended (as C2 already states) to ordering edges:

- **A handle is obtainable only by registering a node** (§1), and an **ordering
  edge accepts only existing handles** (§1). Therefore **no expression can name a
  node that is not yet registered.**
- **A back-edge to self cannot be written.** At a node's own registration, its
  handle does not yet exist (registration is what *produces* it), so the node
  cannot pass its own handle among its own ordering upstreams — the binding is
  undeclared at that point. Case `ordering_edge_self_cycle`.
- **A back-edge to a descendant cannot be written.** Once a node A is registered,
  its registration is **closed**; a later node B's handle did not exist when A was
  registered, and there is **no post-hoc "add edge" API** to reopen A and make it
  order-after B (§5). Case `ordering_edge_back_edge`.

This is **structural per C2** — a property of what expressions *can be written* —
**not a later validation pass** and **not a runtime cycle-detection pass**. It is
**demonstrated by a checked-in compile-failure test**: the two skeletal fixtures
`ordering_edge_self_cycle` and `ordering_edge_back_edge` fail to compile under the
pinned toolchain, wired to the T8 UI harness that T12 reuses.

### 5. No after-the-fact edge API

There is **no API to add an ordering edge between two already-registered nodes
after the fact** (C4). The **only** edge-declaring entry point is a node's own
registration (§1), which is closed once it returns the handle. Concretely, there
is **no** `Flow::add_ordering_edge(existing, existing)` method (or any equivalent
that connects two existing nodes) — its **absence is the guarantee**, and the
`ordering_edge_back_edge` fixture demonstrates that a call to such a method **fails
to compile** (there is no such method to call). This is the ordering-edge
enforcement of C4's "no API to add an edge between two existing nodes afterward,"
and it is what closes the back-edge path in §4.

### 6. Triggering across ordering edges (default rule) — deferred semantics, fixed interaction

Under the default `all-succeeded` rule, an **ordering upstream counts exactly like
a data upstream** (C4, Vocabulary): it must be **success-like**, and a **skip or
failure propagates across the ordering edge** to the downstream node exactly as it
would across a data edge (`upstream-skipped` / `upstream-failed` per T0.4's
tables). A node that must run **regardless** of how its ordering upstreams ended
says so with a non-default rule (`all-terminal`). This ADR fixes only the
**interaction** — that an ordering edge is a genuine upstream for triggering, not a
no-op — while the **propagation *implementation*** and the terminal-state tables
are **T0.4's decision and T50's implementation**, not re-decided here.

### 7. The non-default-trigger-rule restriction is EXCLUDED from this ticket

The rule that **non-default trigger rules are only expressible on nodes that
consume nothing** (Vocabulary; C3 "a node with data dependencies cannot be given
any trigger rule other than `all-succeeded`") is a **C5 / builder-typestate**
restriction, **decided in T0.4 and implemented in T50** — it is **out of scope
here**. This ticket fixes only **ordering-edge attachment** (§§1–5) and the
default-rule **interaction** (§6); it deliberately does **not** conflate ordering
attachment with the trigger-rule typestate. An ordering-only node *happens* to be
a "consume-nothing" node and is therefore *eligible* for a non-default rule, but
**assigning** that rule and enforcing the consume-nothing precondition is T50/C5,
not this ADR.

### 8. Compile-fail assertion table for T12

real authoring API (T13/T50) — but are **not dead**: the harness compiles each
under the pinned toolchain **on every `cargo test`** and fails if either ever
stops compile-failing, proving the guarantee is structural **today**. The two
**positive** fixtures live in a normal integration test
(`crates/core/tests/ordering_edge_positive.rs`, compiled by `cargo test
--workspace`) — **not** in `tests/ui/`, because the T8 harness asserts every
`tests/ui/*.rs` sample **fails** to compile, so a positive sample there would
break it; their **compilation is the assertion**.

### 10. Scope boundary (permanent non-goals)

The cycle guarantee is **structural and compile-time**: there is **no runtime
cycle detection and no graph-validation pass**, and adding one would cross the
design intent (C2). The **graph's shape never changes at runtime** — there is no
dynamic fan-out, no post-hoc edge mutation, no after-the-fact `add_ordering_edge`
(§5). dagr is permanently **not a scheduler, distributed execution system,
metadata store, or DSL**. This ADR fixes a **fixed-by-construction** ordering-edge
shape and never introduces a runtime knob that would breach that boundary (see
**Rejected alternatives**).

## Consequences

**Each blocked ticket inherits a named seam and reopens no question this ADR
closed:**

- **T12** (024 — compile-failure suite for wiring, C2/C3) consumes the
  **compile-fail assertion table (§8)** and the **two skeletal cycle fixtures
  (§9)**: it authors the ordering-edge cycle cases `ordering_edge_self_cycle` and
  `ordering_edge_back_edge` against the same T8 UI harness these fixtures already
  use, alongside its other wiring cases (wrong-type binding, wrong arity,
  unforgeable handles, ownership-demand on a shared value, non-default rule on a
  data-dependent node — all **T12's**, not here). This ADR is the source of T12's
  **ordering-edge cycle contract**.
- **T50** (062 — ordering dependencies, C4) consumes the **registration-time API
  shape (§1)**, the **type-erasure rule (§2)**, the **data+ordering coexistence
  and ordering-only-no-value shape (§3)**, the **no-after-the-fact-edge rule
  (§5)**, and the **default-rule propagation interaction (§6)**: it **implements**
  ordering-edge attachment, recording (distinct edge kind in the graph artifact),
  rendering (distinct styling in diagrams), and default-rule propagation across
  ordering edges, and it **enforces** the ordering-only-no-value shape and the
  non-default-rule → consume-nothing typestate (§7). It re-decides none of the

**Spike disposition.** A throwaway prototype was built **outside the workspace**

**Coverage matrix: no change.** C2 and C4 remain as they stand in

covering test**; the covering tests land with T12 (C2's ordering-edge cycle case)

makes **no edit** to the coverage matrix and **no edit** to the criteria-matrix
partition (T0.10's), and it **agrees** with both. (The T8 harness already ships as
C2/C3's UI infrastructure; adding fixtures to it maps no new criterion here.)

**Reopen condition.** If a downstream ticket cannot honor a seam as written — for
example, if the real handle type cannot yield a type-erased ordering upstream
without coupling the value type (an inference or coherence obstruction the derive
cannot route around), if the ordering-only registration path cannot structurally
force a `()` body input, or if the T8 harness cannot host an ordering-edge cycle
fixture without a harness change — **the contract reopens here**, in this ADR,
rather than being worked around locally. A local workaround that silently diverges
(a runtime cycle check, an after-the-fact add-edge method, a value-type-coupled
ordering edge) is a defect, not a fix.

## Rejected alternatives

- **A runtime cycle-detection / graph-validation pass** (accept edges freely, then
  reject cycles at assembly or run time). **Rejected on the design intent (C2):**
  the guarantee is **structural and compile-time** — "a cycle **cannot be
  expressed** … structural, not a validation pass that runs later." A runtime (or
  even assembly-time) cycle check would concede that a cycle *can* be written and
  is caught late, which is precisely the property arch.md rejects; it is also
  explicitly out of scope for this ticket. The backward-reference discipline
  (§§1, 4) makes the cycle **unwritable**, so no such pass is needed or wanted.
- **An after-the-fact "add edge" API** (a `Flow::add_ordering_edge(a, b)` that
  connects two already-registered nodes). **Rejected on C4 and the scope
  boundary:** C4 states "**no API exists to add an edge between two existing nodes
  afterward**," and a mutable-graph "add edge" method would (a) reopen the
  closed-registration invariant that makes back-edges unwritable (§§4–5) and
  (b) edge toward dynamic graph mutation, which the permanent scope boundary
  forbids (the graph's shape never changes at runtime). The **only** edge-declaring
  entry point is a node's own registration; the absent method is the guarantee.
- **Coupling the value type into the ordering edge** (make the ordering-upstream
  argument typed, e.g. require `Handle<T>` and thread `T` into the edge or the
  fingerprint). **Rejected on C4 and T0.7:** an ordering edge constrains
  **sequence, not data**, so it must be **type-erased** (§2) — a handle of any `T`
  must be an acceptable ordering upstream and mixing differing types must be
  legal. A typed ordering edge would reject legal pipelines (a cleanup ordered
  after a `Handle<Rows>` **and** a `Handle<Bytes>`) and would wrongly leak the
  value type into the structural fingerprint via the ordering edge, contradicting
  T0.7 (an ordering edge carries **no** carried type). The positive fixture
  `ordering_edge_any_value_type_ok` guards against this regression.
- **Conflating ordering attachment with the non-default-trigger-rule typestate**
  (decide here that ordering-only nodes *must* / *may* take non-default rules).
  **Rejected as out of scope:** the consume-nothing → non-default-rule restriction
  is a **C5 / T0.4 / T50** decision (Vocabulary; C3), and folding it into
  ordering-edge attachment would steal T50's scope and risk contradicting the
  already-merged T0.4 tables. This ticket fixes attachment and default-rule
  interaction only (§§6–7).

*(Reopen condition stated in §Consequences: if a downstream ticket cannot honor a
seam as written, the contract reopens here rather than being worked around
locally.)*
