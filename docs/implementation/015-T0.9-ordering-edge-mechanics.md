# 015 · T0.9 — C4 ordering-edge mechanics

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C2, C4
> **Branch:** `adr/t0.9-ordering-edge-mechanics` · **Depends on:** — · **Blocks:** T12, T50

## Why / context
The spec (C2 · Handle, C4 · Ordering dependency) already chose Option A: ordering edges are declared at the downstream node's registration against handles of *already-registered* upstreams, so the backward-reference discipline that makes a data-edge cycle inexpressible extends unchanged to ordering edges. This is a decision ticket that locks the **API shape** enforcing that discipline and the **exact assertions** the wiring compile-failure suite (T12) will make for ordering-edge cycles, before either the ordering-edge implementation (T50) or the compile-fail suite (T12) is written. It records those choices as a checked-in ADR plus at least one skeletal (may be `#[ignore]`d or `trybuild`-stubbed) compile-fail fixture that proves the guarantee is structural, not a later validation pass. Nothing here implements the runtime propagation or graph-artifact rendering of ordering edges — only the constructive shape and the failure contract.

## Objective
Decide and record, as an ADR, the ordering-edge API surface that makes an ordering-edge cycle inexpressible by construction, and pin the compile-fail contract T12 must assert. Concretely:

- Fix the **registration-time API shape**: an ordering edge is attached only at the downstream node's registration, taking one or more existing handles (of any node, any value type) as ordering upstreams, with no argument or method anywhere that accepts a not-yet-registered node or adds an edge between two already-registered nodes after the fact.
- Confirm the **type-erasure rule** for ordering upstreams: an ordering edge ignores the handle's value type (it constrains sequence, not data), so a handle of any `T` is an acceptable ordering upstream, and mixing ordering upstreams of different value types is legal.
- Confirm that a node may carry **both** data dependencies and additional ordering edges, and that a node attached *only* by ordering edges receives **no value** (its task body sees no bound input), distinguishing this from the non-default-trigger-rule restriction which is *out of scope here* (owned by T50/C5 vocabulary).
- Specify the **cycle-inexpressibility argument** in the ADR: because a handle is obtainable only by registering a node, and an ordering edge accepts only existing handles, no expression can name a node that is not yet registered — so a back-edge to self or to a descendant cannot be written. State that this is structural (C2), not a runtime cycle-detection pass.
- Enumerate the **compile-fail assertions for T12** so the suite author can copy them: for each targeted misuse, the case name, the misuse it demonstrates, and the observable expectation (fails to compile; where a message is asserted, that both relevant type names or a curated diagnostic appear — pinned to the workspace toolchain per C28).
- Deliver **at least one skeletal compile-fail fixture** for the ordering-edge cycle case, wired to the same harness T12 will use (e.g. `trybuild`), demonstrating the guarantee compiles-fail today; it may be marked pending/ignored but must be executable by the suite.

## Test plan (write these first — TDD)
Because this is a decision ticket, the "tests" are the evidence that validates the chosen API shape and the failure contract. Each is an independently checkable check on the ADR or a throwaway/skeletal fixture.

- **Cycle to self is inexpressible.** Setup: a throwaway prototype pipeline where a single node attempts to declare an ordering edge against its own not-yet-returned handle. Action: attempt to compile it under the pinned workspace toolchain. Expected: it fails to compile because no handle for the node exists at the point of registration — recorded as a `trybuild` compile-fail fixture, and the ADR names this as case `ordering_edge_self_cycle`.
- **Cycle to a descendant is inexpressible.** Setup: a prototype where node A registers, then node B registers an ordering edge on A, then an attempt is made to have A depend (via ordering edge) on B. Action: compile. Expected: it fails to compile because A's registration is already closed and B's handle did not exist when A was registered; the ADR notes there is no post-hoc "add edge" API to reach back to A. Recorded as case `ordering_edge_back_edge`.
- **No forward reference exists.** Setup: inspect the decided API surface as described in the ADR. Action: enumerate every entry point that accepts an ordering upstream. Expected: every such entry point takes an existing handle value; the ADR asserts (and a reviewer can confirm against the surface) that none accepts a node name, index, string key, or a builder position that could denote an unregistered node — mirroring C2's "no lookup by name/index/string key."
- **No after-the-fact edge API exists.** Setup: read the API surface in the ADR. Action: search for any method that connects two already-registered nodes. Expected: none exists; the ADR states this explicitly as the enforcement of C4's "no API to add an edge between two existing nodes afterward."
- **Any value type is an acceptable ordering upstream.** Setup: a prototype registers nodes producing distinct value types and attaches ordering edges from each to a downstream node. Action: compile. Expected: it compiles — proving ordering edges are type-erased and the value type does not participate — and the ADR records this as a *positive* (compiles) fixture so a future regression that accidentally couples value type into ordering fails review.
- **Data-plus-ordering coexists; ordering-only yields no value.** Setup: a prototype where one node has both a data dependency and an extra ordering edge, and a second node has ordering edges only. Action: compile and inspect the second node's task-body input. Expected: both compile; the ordering-only node's body is given no bound value (its declaration reflects "receives no value" per C4), and the ADR documents where this is enforced structurally so T50 can rely on it.
- **Assertion table is complete and matches the acceptance criteria.** Setup: the ADR's compile-fail assertion table. Action: cross-check it against C2 and C4 acceptance criteria. Expected: every cycle-related criterion (C2 "cycle through data or ordering edges fails to compile, demonstrated by a checked-in compile-failure test"; C4 "no API to add an edge afterward") maps to at least one named case in the table, and each table row states case name, the misuse, and the observable outcome.

## Definition of done
- [ ] The ADR fixes an ordering-edge API shape where an ordering edge is declared **only at the downstream node's registration** against **already-registered** upstream handles (C4).
- [ ] The ADR confirms **no API** accepts an ordering upstream by node name, index, or string key, and **no API** adds an ordering edge between two already-registered nodes after the fact (C4, mirroring C2).
- [ ] The ADR states, with the constructive argument, that an ordering-edge cycle (self or back to a descendant) is **inexpressible by construction** — structural per C2, not a later validation pass — and that this is demonstrated by a checked-in compile-failure test.
- [ ] The ADR confirms an ordering edge may attach to **any** node regardless of its value type (type-erased), and that mixing ordering upstreams of differing value types is legal.
- [ ] The ADR confirms a node may carry **both** data dependencies and additional ordering edges (C4).
- [ ] The ADR confirms a node attached **only** by ordering edges **receives no value** and that its declaration reflects that (C4), and points to where T50 enforces it.
- [ ] The ADR explicitly excludes the non-default-trigger-rule restriction (that belongs to C5/vocabulary and T50) so this ticket does not conflate ordering-edge attachment with trigger-rule typestate.
- [ ] The ADR contains a **compile-fail assertion table** for T12 listing, per case: case name, the misuse demonstrated, and the observable expectation (fails to compile; and where a message is asserted, that both type names or the curated diagnostic appear, pinned to the workspace toolchain per C28).
- [ ] The table covers at minimum: `ordering_edge_self_cycle`, `ordering_edge_back_edge`, and a positive `ordering_edge_any_value_type_ok` fixture; each is cross-referenced to the C2/C4 acceptance criterion it satisfies.
- [ ] At least one **skeletal compile-fail fixture** for the ordering-edge cycle case is checked in, wired to the same harness T12 will use (e.g. `trybuild`), and is executable (may be pending/ignored but not dead) — proving the guarantee compiles-fail under the pinned toolchain.
- [ ] The ADR is filed at the project's decision-record location and linked from T12 and T50 as the source of their ordering-edge cycle contract.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- The actual **implementation** of ordering-edge attachment, recording, and rendering (C4) — that is T50; this ticket only locks the API shape and the failure contract.
- Writing the **full wiring compile-failure suite** (wrong-type binding, wrong arity, unforgeable handles, ownership-demand on a shared value, non-default trigger rule on a data-dependent node) — that is T12; this ticket delivers only the ordering-edge cycle assertions and one skeletal fixture.
- The **non-default-trigger-rule → consume-nothing typestate** restriction and its `all-terminal`/`all-succeeded` propagation semantics (C5 vocabulary, C4 propagation) — decided/implemented in T0.4 and T50, not here.
- Any **runtime cycle detection or graph-validation pass** — explicitly rejected; the guarantee is structural and compile-time (C2). Adding a runtime check would cross the design intent.
- **Graph-artifact serialization and diagram rendering** of ordering edges (C4 "recorded distinctly / drawn distinctly") — owned by T50 and the graph-artifact tickets.
- Anything touching runtime scheduling, distributed execution, or dynamic graph mutation — permanent scope boundary; the graph shape never changes at runtime.
