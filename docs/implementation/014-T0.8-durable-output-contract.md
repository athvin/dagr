# 014 · T0.8 — Durable-output contract

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C27
> **Branch:** `adr/t0.8-durable-output-contract` · **Depends on:** — · **Blocks:** T14, T39, T57, T58

## Why / context
Resume (C27) can only skip completed work if a durable node's output value can be reconstructed later from a serialized reference rather than re-computed. This ticket is the M0 decision that locks that contract — the durability policy flag on C5, the serialize-reference / rehydrate trait pair, the assembly-time rule that a durable-marked node's output type must implement the contract, and the existence-check semantics used at resume — *before* its consumers land: T14 needs the assembly check, T39 needs the reference field in the schemas, and T57/T58 implement declaration/recording and resume core against this shape. It governs the "durable-output contract" and "declared per node in policy" text of C27 (arch.md §C27, lines 567–580) and the durability flag on C5 (§C5, lines 151, 157). As a backward-leaking M0 decision, the contract is *decided* here (ADR + trait signatures + enforcement points) while full implementation stays in its spec milestone (M4).

## Objective
Produce a decision record (ADR) plus the minimal, compiling type surface that pins the durable-output contract so its four downstream consumers can build against a stable seam.

Concrete pieces of work:
- **Durability policy flag (C5).** Decide where "durable" lives in node policy, its default (`not durable`), and that toggling it requires no task-code change and hashes into the policy hash — not the structural fingerprint — like every other C5 field (C21).
- **Reference trait pair.** Decide the two-operation contract on a durable node's *output type*: **serialize-reference** (produce a serializable, self-describing reference to where the value durably lives) and **rehydrate** (reconstruct the typed value from that reference later). Fix the associated reference type, its serde-serializable shape, error typing for both operations, and whether rehydrate is fallible/async. Record why the contract sits on the output type (not the task) so any durable node's output is reconstructable regardless of which node produced it (needed for single-node replay, C26/T57).
- **Existence-check operation.** Decide the *cheap* existence-check semantics: a reference must answer "does the referent still exist?" without fetching the value, its error/ambiguity classification (present / absent / cannot-determine), and that a dangling reference fails the resume *plan* up front — not mid-run.
- **Assembly-enforcement seam (C7).** Decide how assembly proves, from the durability flag plus the output type, that a durable-marked node implements the contract, and that violation is one of the *all-problems-reported* assembly errors naming the offending node (feeds T14).
- **Schema field (C22/C20).** Decide the durable-reference field recorded per attempt in the artifact and copied forward on resume so artifacts stay self-contained (feeds T39).
- **ADR document** capturing the decision, the alternatives rejected, and the exact enforcement points, cross-referenced by tid.

## Test plan (write these first — TDD)
Because this is a decision ticket, the "tests" are throwaway-prototype compile/behaviour probes and decision-record checks that prove the chosen shape is enforceable and usable. Each must be independently checkable.

- **Contract is implementable for a reference-carrying output.** Setup: a throwaway prototype output type representing an external reference (for example, a storage key) that implements the chosen trait pair. Action: mark a node durable and assemble the pipeline. Expected: assembly succeeds and the prototype compiles — proving the trait is satisfiable by a realistic durable output.
- **Durable-marked node lacking the contract is rejected at assembly.** Setup: a node whose output type does *not* implement the contract, marked durable. Action: run assembly. Expected observable: assembly fails with an error that names the offending node and states the output type lacks the durability contract; assembly still reports any *other* problems it found (all-problems rule, C7).
- **Non-durable node needs nothing.** Setup: the same non-implementing output type on a node left at the default (not durable). Action: assemble. Expected: assembly succeeds — the contract is required only when the durability flag is set.
- **Default and policy-hash behaviour.** Setup: two otherwise-identical nodes, one with durability left defaulted and one with `not durable` written out explicitly. Action: compute effective policy and the policy hash for both. Expected: identical effective policy and identical policy hash (defaulted == written-out default, C5/C21); the durability flag appears in the emitted effective policy; toggling it changes the policy hash but not the structural fingerprint.
- **Round-trip: serialize-reference then rehydrate.** Setup: a value written to a fake durable location behind the prototype reference type. Action: serialize the reference, then rehydrate the typed value from the deserialized reference. Expected: the rehydrated value equals the original — the two operations compose into a lossless round-trip through a serializable reference.
- **Existence check is present / absent / cannot-determine.** Setup: three references — one to a value that exists in a fake store, one to a deleted object, one to a store that errors on probe. Action: run the existence check on each. Expected: the first reports present, the second reports absent, the third reports cannot-determine (or the chosen error classification) — and the probe does *not* fetch the value.
- **Dangling reference fails the plan up front (decision-level).** Setup: a resume-plan sketch that existence-checks candidate references before marking any node satisfied. Action: feed it a reference whose referent is absent. Expected: the plan is rejected before any node is marked satisfied / before execution begins — encoding C27's "not the eleventh node" guarantee at the contract boundary.
- **Attempt-record field is present and serializable.** Setup: the schema field decided for the durable reference. Action: serialize an attempt record carrying a reference and deserialize it. Expected: the reference round-trips through the artifact schema — proving T39 has a concrete field shape to adopt.
- **ADR decision-record check.** Setup: the ADR file. Action: read it. Expected: it states the durability flag and its default, the two trait operations with their signatures and error typing, the existence-check classification, the assembly-enforcement point, and the schema field — each cross-referenced to the consuming tid (T14, T39, T57, T58) — and lists the alternatives considered and rejected.

## Definition of done
- [ ] The durability flag is a per-node policy field (C5), attached at registration, with a documented default of `not durable`; changing it requires no change to task code.
- [ ] The durability flag appears in every node's full effective policy in the graph artifact, including when defaulted; a defaulted value behaves identically to the written-out default and hashes identically under the policy hash (C5/C21).
- [ ] The durability flag participates in the **policy hash** and is **excluded** from the structural fingerprint (a policy-only durability change proceeds at resume, C21/C27).
- [ ] The reference contract is defined as a trait pair on the node's **output type**: serialize a self-describing, serde-serializable reference to where the value durably lives, and rehydrate the typed value from that reference later; error typing and fallibility/async-ness for both operations are fixed and recorded.
- [ ] A prototype durable output type implements the contract, is marked durable, and assembles successfully — proving the contract is satisfiable.
- [ ] Assembly rejects a durable-marked node whose output type lacks the contract, naming the offending node, and does so as part of the all-problems-reported assembly pass (C7) — the enforcement point is decided and specified for T14.
- [ ] A non-durable node with the same non-implementing output type assembles without error.
- [ ] The existence-check semantics are decided: a cheap probe that does not fetch the value, with a present / absent / cannot-determine (or equivalent) classification, and a dangling reference fails the resume **plan before execution begins** (C27) rather than mid-run.
- [ ] The durable reference is recorded per attempt in the artifact and is copied forward on resume so artifacts remain self-contained (C22/C27) — the schema field shape is decided and specified for T39.
- [ ] It is recorded that in-memory (non-durable) outputs cannot be rehydrated and therefore re-execute on demand at resume (C27), so the contract deliberately applies only to durable-marked nodes.
- [ ] An ADR at the ticket path captures the decision, the trait signatures and error types, the enforcement points, the schema field, the alternatives rejected, and the cross-references to T14, T39, T57, T58.
- [ ] The minimal type surface (flag, trait pair, reference type, existence-check operation) compiles and is documented with rustdoc on all public items.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- **Resume algorithm itself** (seed / downward-closure / demand-resolution, fingerprint gates, parameter derivation, lineage linkage) — that is C27's resume core, T58; this ticket only fixes the contract it stands on.
- **Durable-output declaration/recording implementation** — the full wiring of the flag into policy and the reference into the attempt record is T57; here only the seam is decided.
- **Assembly validation implementation** — T14 implements the durable-without-contract check (and every other assembly check) against this decision; this ticket does not build the validation pass.
- **Artifact/event schema publication** — T39 owns the versioned checked-in schemas; this ticket only decides the durable-reference field they must include.
- **Single-node replay** and rehydrating inputs from references at the CLI (C26/T55) — consumes this contract later; not built here.
- **Durable scratch store and its resume carry-forward** (C18/T53/T54b) — a separate durability surface; not this contract.
- **Concrete storage backends or an object-store abstraction** — the contract is over *references* the task's own output type provides; dagr does not become a metadata store or storage layer (permanent scope boundary), and no built-in remote/object backend is introduced.
- Any temptation toward a metadata store, a distributed reference registry, or runtime-mutable graph shape — named here and explicitly excluded.
