# 014 · T0.8 — Durable-output contract

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C27
> **Branch:** `adr/t0.8-durable-output-contract` · **Depends on:** — · **Blocks:** T14, T39, T57, T58

## Why / context
Resume (C27) can only skip completed work if a durable node's output value can be reconstructed later from a serialized reference rather than re-computed. This ticket is the M0 decision that locks that contract — the durability policy flag on C5, the serialize-reference / rehydrate trait pair, the assembly-time rule that a durable-marked node's output type must implement the contract, and the existence-check semantics used at resume — *before* its consumers land: T14 needs the assembly check, T39 needs the reference field in the schemas, and T57/T58 implement declaration/recording and resume core against this shape. It governs the "durable-output contract" and "declared per node in policy" text of C27 (arch.md §C27, lines 567–580) and the durability flag on C5 (§C5, lines 151, 157). As a backward-leaking M0 decision, the contract is *decided* here (ADR + trait signatures + enforcement points) while full implementation stays in its spec milestone (M4).

## Objective
Produce a decision record (ADR) plus the minimal, compiling type surface that pins the durable-output contract so its four downstream consumers can build against a stable seam.

Concrete pieces of work:
- **Durability policy flag (C5).** Decide where "durable" lives in node policy, its default (`not durable`), and that toggling it requires no task-code change and hashes into the policy hash — not the structural fingerprint — like every other C5 field (C21).
- **Reference trait pair.** Decide the two-operation contract on a durable node's *output type*: **serialize-reference** (produce a serializable, self-describing reference to where the value durably lives) and **rehydrate** (reconstruct the typed value from that reference later). Fix the associated reference type, its serde-serializable shape, error typing for both operations, and whether rehydrate is fallible/async. Record why the contract sits on the output type (not the task) so any durable node's output is reconstructable regardless of which node produced it (needed for single-node replay, C26/T57).
- **Existence-check operation.** Decide the *cheap* existence-check semantics: a reference must answer "does the referent still exist?" without fetching the value, its error/ambiguity classification (present / absent / cannot-determine), and that a dangling reference fails the resume *plan* up front — not mid-run.
- **Assembly-enforcement seam (C7).** Decide how assembly proves, from the durability flag plus the output type, that a durable-marked node implements the contract, and that violation is one of the *all-problems-reported* assembly errors naming the offending node (feeds T14).
- **Schema field (C22/C20).** Decide the durable-reference field recorded per attempt in the artifact and copied forward on resume so artifacts stay self-contained (feeds T39).
- **ADR document** capturing the decision, the alternatives rejected, and the exact enforcement points, cross-referenced by tid.

## Test plan (write these first — TDD)
Because this is a decision ticket, the "tests" are throwaway-prototype compile/behaviour probes and decision-record checks that prove the chosen shape is enforceable and usable. Each must be independently checkable.

- **Contract is implementable for a reference-carrying output.** Setup: a throwaway prototype output type representing an external reference (for example, a storage key) that implements the chosen trait pair. Action: mark a node durable and assemble the pipeline. Expected: assembly succeeds and the prototype compiles — proving the trait is satisfiable by a realistic durable output.
- **Durable-marked node lacking the contract is rejected at assembly.** Setup: a node whose output type does *not* implement the contract, marked durable. Action: run assembly. Expected observable: assembly fails with an error that names the offending node and states the output type lacks the durability contract; assembly still reports any *other* problems it found (all-problems rule, C7).
- **Non-durable node needs nothing.** Setup: the same non-implementing output type on a node left at the default (not durable). Action: assemble. Expected: assembly succeeds — the contract is required only when the durability flag is set.
- **Default and policy-hash behaviour.** Setup: two otherwise-identical nodes, one with durability left defaulted and one with `not durable` written out explicitly. Action: compute effective policy and the policy hash for both. Expected: identical effective policy and identical policy hash (defaulted == written-out default, C5/C21); the durability flag appears in the emitted effective policy; toggling it changes the policy hash but not the structural fingerprint.
- **Round-trip: serialize-reference then rehydrate.** Setup: a value written to a fake durable location behind the prototype reference type. Action: serialize the reference, then rehydrate the typed value from the deserialized reference. Expected: the rehydrated value equals the original — the two operations compose into a lossless round-trip through a serializable reference.
- **Existence check is present / absent / cannot-determine.** Setup: three references — one to a value that exists in a fake store, one to a deleted object, one to a store that errors on probe. Action: run the existence check on each. Expected: the first reports present, the second reports absent, the third reports cannot-determine (or the chosen error classification) — and the probe does *not* fetch the value.
- **Dangling reference fails the plan up front (decision-level).** Setup: a resume-plan sketch that existence-checks candidate references before marking any node satisfied. Action: feed it a reference whose referent is absent. Expected: the plan is rejected before any node is marked satisfied / before execution begins — encoding C27's "not the eleventh node" guarantee at the contract boundary.
- **Attempt-record field is present and serializable.** Setup: the schema field decided for the durable reference. Action: serialize an attempt record carrying a reference and deserialize it. Expected: the reference round-trips through the artifact schema — proving T39 has a concrete field shape to adopt.
- **ADR decision-record check.** Setup: the ADR file. Action: read it. Expected: it states the durability flag and its default, the two trait operations with their signatures and error typing, the existence-check classification, the assembly-enforcement point, and the schema field — each cross-referenced to the consuming tid (T14, T39, T57, T58) — and lists the alternatives considered and rejected.

## Definition of done
- [ ] The durability flag is a per-node policy field (C5), attached at registration, with a documented default of `not durable`; changing it requires no change to task code.
- [ ] The durability flag appears in every node's full effective policy in the graph artifact, including when defaulted; a defaulted value behaves identically to the written-out default and hashes identically under the policy hash (C5/C21).
- [ ] The durability flag participates in the **policy hash** and is **excluded** from the structural fingerprint (a policy-only durability change proceeds at resume, C21/C27).
- [ ] The reference contract is defined as a trait pair on the node's **output type**: serialize a self-describing, serde-serializable reference to where the value durably lives, and rehydrate the typed value from that reference later; error typing and fallibility/async-ness for both operations are fixed and recorded.
- [ ] A prototype durable output type implements the contract, is marked durable, and assembles successfully — proving the contract is satisfiable.
- [ ] Assembly rejects a durable-marked node whose output type lacks the contract, naming the offending node, and does so as part of the all-problems-reported assembly pass (C7) — the enforcement point is decided and specified for T14.
- [ ] A non-durable node with the same non-implementing output type assembles without error.
- [ ] The existence-check semantics are decided: a cheap probe that does not fetch the value, with a present / absent / cannot-determine (or equivalent) classification, and a dangling reference fails the resume **plan before execution begins** (C27) rather than mid-run.
- [ ] The durable reference is recorded per attempt in the artifact and is copied forward on resume so artifacts remain self-contained (C22/C27) — the schema field shape is decided and specified for T39.
- [ ] It is recorded that in-memory (non-durable) outputs cannot be rehydrated and therefore re-execute on demand at resume (C27), so the contract deliberately applies only to durable-marked nodes.
- [ ] An ADR at the ticket path captures the decision, the trait signatures and error types, the enforcement points, the schema field, the alternatives rejected, and the cross-references to T14, T39, T57, T58.
- [ ] The minimal type surface (flag, trait pair, reference type, existence-check operation) compiles and is documented with rustdoc on all public items.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Decision

### 1. The durability flag — a per-node policy field (C5), default *not durable*

**Durability is a per-node policy field**, one of the C5 knobs "attached at
registration, never inside the task." Its documented **default is `not
durable`** (arch.md C5: "release the output once consumed, not durable"), applied
uniformly like every other C5 default.

- **Setting it requires no change to task code.** Durability is toggled at the
  registration/policy site (the same place retries, timeout, and retention are
  set), not in the task's `run`. A task authored today becomes durable tomorrow by
  a one-line policy change at its registration — the task body is untouched. This
  is C5's "changing any policy value requires no change to task code," applied to
  this field.
- **It appears in the node's full effective policy in the graph artifact,
  including when defaulted.** Per C5, every node's *effective* policy is emitted
  with defaults written out, so a node that never mentions durability still
  records `durable: false` in the artifact. A **defaulted** value behaves
  **identically** to the written-out default and, per §2, **hashes identically**.
- **It is a boolean knob, nothing more.** "Durable" / "not durable" is a single
  policy bit. It carries no store selection, no retention interplay beyond what
  C10 already fixes, and no runtime configurability — the graph's shape and policy
  never change at runtime (scope boundary, §8).

*(The flag is **declared/recorded** by T57 and its default/emission behaviour is
tested under C5's covering ticket T29; this ADR fixes the field, its default, and
its policy-vs-fingerprint placement.)*

### 2. Effective-policy emission and the defaulted-hashes-identically property

Two otherwise-identical nodes — one leaving durability **defaulted**, one writing
**`not durable`** out explicitly — produce:

- **identical effective policy** (both resolve to `durable: false`);
- an **identical policy hash** (the hash is computed over the *effective* policy,
  defaults resolved, per T0.7 §4 and C5); and
- the flag **visibly present** in the emitted effective policy of both.

Toggling the flag (`false` → `true`) **changes the policy hash** but **not the
structural fingerprint** (§3). This is the C5/C21 "defaulted == written-out
default" property, read for the durability field specifically, and it is what
lets a durable-marked and a defaulted node be compared honestly in the artifact.

### 3. The flag is in the policy hash, out of the structural fingerprint (C21/C27)

Per T0.7 §4 (which this ADR consumes, not re-decides), the **durability flag
participates in the POLICY HASH** and is **excluded from the structural
fingerprint**. The consequence at resume (C21/C27):

- A run that only **toggled durability** (or any other policy value) has an
  **unchanged structural fingerprint** — the topology is identical — so resume
  **proceeds**, printing the per-node policy diff. A policy divergence is a
  **proceed-with-diff, never a refusal**: "I marked this expensive stage durable
  and want to resume" is a *motivating* case for resume, not a reason to refuse
  it.
- Only a **structural** change (nodes, edges, carried types, trigger rules) moves
  the structural fingerprint and gates resume. Durability is tuning, not shape.

This placement is what makes a durability change *safe* to resume across — the
prior run's outputs still mean the same thing to this binary.

### 4. The reference contract — a trait pair on the node's OUTPUT TYPE

The contract is **two operations on a durable node's output type** — **not** on
the task. It sits on the output type because, per T0.2/C10, a durable stage
boundary's output value *is* a reference to where the value durably lives, and
because single-node replay (C26/T57) must rehydrate an input from a reference
**regardless of which node produced it** — the consumer that replays never ran
the producer, so the reconstruction logic must travel with the *value type*, not
the producing task.

The two operations:

| Operation | Signature shape | Guarantee |
|---|---|---|
| **serialize-reference** | `fn durable_reference(&self) -> Self::Reference` (**infallible**) | Produces a **self-describing, serde-serializable** reference to where the value durably lives. Called **on success**, after the value is already durably written by the task, so there is nothing left that can fail — the value is in external storage and the reference merely *names* it. |
| **rehydrate** | `async fn rehydrate(reference: Self::Reference) -> Result<Self, RehydrateError>` (**fallible, async**) | Reconstructs the typed value from the deserialized reference **later** (possibly in a different process, at resume or single-node replay). **Fallible** because the referent may be gone, unreachable, or corrupt; **async** because reconstruction is I/O (a fetch/read) and must not block a runtime worker. |

**The associated reference type.** `Self::Reference` is an associated type bound
`Serialize + DeserializeOwned + Send + Sync + 'static` (serde-serializable so it
round-trips through the C22 artifact schema, §6; `Send + Sync + 'static` so it
lives in the artifact and crosses threads like any T0.2 output). It is
**self-describing**: it carries enough to both existence-check (§7) and rehydrate
without any ambient dagr configuration — dagr never interprets its bytes, it only
serializes, stores, and hands them back.

**Error typing.**
- `serialize-reference` is **infallible** — it names an already-written value.
  (Writing the value durably is the *task's* job during `run`; if that write
  fails, the **attempt** fails through the normal C1/C3 error taxonomy, and no
  reference is produced. The contract's serialize step never sees a
  not-yet-written value.)
- `rehydrate` returns a **typed, structured error** (`RehydrateError`) in the T3
  error-taxonomy family, distinguishing at least *referent absent* (the value is
  gone) from *transient/unreachable* (a retryable fetch failure) from
  *deserialization/corruption* (the bytes fetched are not a valid value). At
  resume this feeds the plan (§7); at single-node replay it feeds the
  "which input and why" refusal message (C26).

**Fallibility/async, fixed:** serialize-reference **infallible + synchronous**;
rehydrate **fallible + async**. This is recorded so T57 and T58 build the same
signatures.

Illustrative shape (not shipping code; T57/T58 own the real definitions):

```rust
/// Implemented by the OUTPUT TYPE of a node marked durable. The reference is
/// self-describing: dagr serializes, stores, existence-checks, and rehydrates it
/// without ever interpreting where the value physically lives.
pub trait DurableOutput: Send + Sync + 'static {
    /// A self-describing, serde-serializable reference to where the value
    /// durably lives (a storage key, a URL, a content hash — the task's choice).
    type Reference: Serialize + DeserializeOwned + Send + Sync + 'static;

    /// Produce the reference for an already-durably-written value. Infallible:
    /// the value is in external storage by the time this is called; this only
    /// names it.
    fn durable_reference(&self) -> Self::Reference;

    /// A cheap probe: does the referent still exist, without fetching the value?
    /// (present / absent / cannot-determine — see §7.)
    fn probe(reference: &Self::Reference)
        -> impl Future<Output = Existence> + Send;

    /// Reconstruct the typed value from the reference later — fallible (the
    /// referent may be gone/corrupt) and async (reconstruction is I/O).
    fn rehydrate(reference: Self::Reference)
        -> impl Future<Output = Result<Self, RehydrateError>> + Send
    where
        Self: Sized;
}
```

**Round-trip is lossless.** A value written to a durable location, its reference
serialized, the reference deserialized, and `rehydrate` awaited yields a value
**equal** to the original — the two operations compose into a lossless round-trip
through a serializable reference. This is the property T57/T58's tests assert and
the reason the contract is *usable*: what you serialize a reference to is exactly
what you get back.

### 5. The assembly-enforcement seam (C7)

Assembly proves, from the **durability flag** plus the node's **output type**,
that a durable-marked node **implements `DurableOutput`**:

- **A durable-marked node whose output type lacks the contract is REJECTED at
  assembly**, with an error that **names the offending node** and states that its
  output type lacks the durability contract (e.g. "node `snapshot` is marked
  durable, but its output type `HugeInMemoryBlob` does not implement the durable-
  output contract; either implement the reference contract on `HugeInMemoryBlob`
  or drop durability on `snapshot`").
- The rejection is **one of the all-problems-reported assembly errors (C7)** —
  assembly still reports **every other** problem it found (duplicate names,
  invalid class overrides, ownership conflicts, …); the durable-without-contract
  error does not short-circuit the pass.
- **A non-durable node with the same non-implementing output type assembles
  without error** — the contract is required **only when the durability flag is
  set**. The default (not durable, §1) demands nothing of the output type, so an
  ordinary in-memory output that happens not to implement `DurableOutput` is
  perfectly legal on a non-durable node.

**How the proof is expressed.** Because "is this output type durable-capable" is
a per-node fact the compiler cannot see across the whole graph (the flag is set
at registration, a whole-graph fact), the enforcement is an **assembly check**,
not a compile error — the same partition T0.2 fixed for mode conflicts. The
registration API records, per node, both the durability flag **and** whether the
node's statically-known output type satisfies `DurableOutput` (a bound the
builder can capture when the node is added); assembly then reports every node
where `durable == true` but the captured "implements the contract" witness is
absent. The *mechanism* (how the witness is captured — a trait-object registration
hook, a witness flag on the typed builder) is **T14/T57's** to implement; this
ADR fixes that the check exists, is at assembly, names the offending node, and is
part of the all-problems pass.

### 6. The schema field — a per-attempt durable reference, copied forward (C22)

The durable reference is recorded **per attempt** in the run artifact:

- **Field.** Each **attempt record** (C22 — the artifact body is one record per
  attempt) carries an **optional `durable_reference`** field: `Some(reference)`
  when the node is durable and the attempt **succeeded** (produced a value and
  therefore a reference), `None` otherwise (non-durable node, or an attempt that
  did not succeed). It is the **serde-serialized `Self::Reference`** (§4), so it
  **round-trips through the artifact schema** — serialize an attempt record
  carrying a reference, deserialize it, and the reference is intact.
- **Copied forward on resume.** A resumed run's artifact **copies durable
  references forward** from the prior artifact for nodes it did not re-execute
  (marked `satisfied-from-prior`), so **every artifact is self-contained** (C22)
  — a later resume or a single-node replay can read the reference from *this*
  run's artifact without walking back through the lineage.
- **Feeds T39.** T39 owns the versioned, checked-in schema; this ADR fixes the
  **field shape** (an optional, serde-serializable reference on the attempt
  record) that T39's schema must include and that T39's validation helper checks
  round-trips.

The reference type is **opaque to the schema**: the artifact stores whatever the
task's `Self::Reference` serializes to (a JSON object, a string key). The schema
reserves the *slot and its optionality*, not the reference's internal shape,
which is the task's.

### 7. Existence check and the plan-time dangling refusal (C27)

- **A cheap probe that does NOT fetch the value.** The `probe` operation (§4)
  answers "**does the referent still exist?**" **without fetching the value** — a
  `HEAD`/`stat`/`exists` against the referent, not a `GET`. This is what makes an
  existence check over many candidate references affordable at resume plan time.
- **Three-way classification: present / absent / cannot-determine.** The probe
  returns one of exactly three outcomes:
  - **present** — the referent exists; the reference is intact and the node may
    be satisfied and rehydrated on demand.
  - **absent** — the referent is gone (a deleted object, a `404`); the reference
    is **dangling**.
  - **cannot-determine** — the probe itself failed to get a definitive answer (a
    store that errors on probe, a timeout, a permission error). This is
    **distinct** from *absent*: dagr does not know the referent is gone, only
    that it could not confirm.
- **A dangling reference fails the resume PLAN before execution begins.** During
  resume planning, the references of **candidate** nodes (prior successes the
  demand algorithm would satisfy-and-rehydrate) are existence-checked **up
  front**, before any node is marked satisfied and before execution begins. An
  **absent** referent **fails the resume plan** — "a reference to a deleted
  object fails the resume plan, **not the eleventh node** of the resumed run"
  (C27). A **cannot-determine** result is treated conservatively as a plan-level
  failure of the same kind (dagr will not silently skip a node whose input it
  cannot prove is recoverable) — the refusal names the reference and its
  classification so the operator can act. Either way the failure is **at plan
  time, not mid-run**, encoding C27's "not the eleventh node" guarantee at the
  contract boundary.

*(The plan-level machinery — collecting candidate references, running the probe
before marking anything satisfied — is **T58's** to build; this ADR fixes the
probe's classification and that the refusal is up-front.)*

### 8. In-memory (non-durable) outputs re-execute on demand (C27)

- **In-memory outputs cannot be rehydrated.** A node left non-durable produces an
  in-memory value that lives only in this run's output slot (C10). There is **no
  reference** to it and **no rehydration** — the moment a re-running consumer
  demands its value at resume, the non-durable producer **re-executes**, and its
  own demands cascade upward (C27's demand-driven algorithm, T58). This is a
  **genuine property of the design, stated plainly** — it creates "useful
  pressure to make expensive stage boundaries produce durable, addressable
  outputs" (C10).
- **The contract deliberately applies only to durable-marked nodes.** Because a
  non-durable node has nothing to serialize a reference to, the trait contract is
  **required only when the durability flag is set** (§5) and is meaningless
  otherwise. An undemanded prior success — durable or not — is
  `satisfied-from-prior` and never needs rehydrating (C27); a *demanded*
  non-durable success re-executes. This is why the contract is on **durable**
  outputs specifically, not on every output.

### 9. Scope boundary (permanent non-goals)

Everything above describes a **local, embedded** durability contract: a boolean
policy flag, a trait pair over a **serde-serializable reference the task's own
output type provides**, an existence probe, and an artifact field. It is **not**
an object-store abstraction, **not** a networked or distributed store, **not** a
metadata store or a reference registry, and **not** a backfill orchestrator or
scheduler. dagr **never** learns where the referent physically lives beyond the
opaque bytes the task hands it; there is **no built-in remote/object backend**.
Durability is **local** and composes with **T0.6's local, file-based run store**
— resume requires a surviving local store, and the durable reference is over the
task's output type, not a dagr storage layer. The graph's shape and policy
**never change at runtime**. The **Rejected alternatives** below name each
temptation that would breach this boundary.

## Consequences

**Each blocked ticket inherits a named seam and reopens no question this ADR
closed:**

- **T14** (025 — assembly validation and precomputation, C7) binds against **the
  assembly-enforcement seam (§5)**: it implements the durable-without-contract
  check — rejecting a durable-marked node whose output type lacks
  `DurableOutput`, naming the offending node — as one of its all-problems-reported
  assembly errors, alongside duplicate names, invalid class overrides, and the
  ownership-mode conflicts T0.2 fixed.
- **T39** (050 — publish artifact schemas, C22) binds against **the schema field
  (§6)**: it publishes the versioned attempt-record schema including the optional,
  serde-serializable `durable_reference` field, and its validation helper asserts
  the field round-trips and is copied forward on resume so artifacts stay
  self-contained.
- **T57** (067 — durable-output declaration and recording, C27) binds against
  **the durability flag (§§1–2), the reference trait pair (§4), and the recording
  (§6)**: it wires the flag into node policy with the `not durable` default,
  defines the `DurableOutput` trait pair on output types, captures the
  implements-the-contract witness at registration for T14's check, and records
  the reference per attempt in the artifact.
- **T58** (070 — resume core, C27) binds against **the existence check and the
  plan-time dangling refusal (§7), rehydration-on-demand (§4), and the
  in-memory-re-execute rule (§8)**: it existence-checks candidate references up
  front (failing the plan on absent / cannot-determine before execution begins),
  fills demanded durable producers' slots by rehydration, re-executes demanded
  non-durable producers, marks undemanded prior successes `satisfied-from-prior`,
  and copies durable references forward into the resumed artifact.

**Coverage matrix: no change.** C27 remains `machine` / `unmapped` in
`docs/coverage-matrix.md`, deferred to its covering-test tickets (Covered-by
T57; the resume acceptance suite is T58/T59). A decision ticket owes no covering
test; the covering tests land with the component tickets above, which will each
replace `unmapped` with a test id per the matrix's per-ticket duty. This ADR
makes **no edit** to the coverage matrix and **no edit** to the criteria-matrix
partition (T0.10's), and it **agrees** with both — in particular it does not fill
T57/T58's C27 seam and does not treat C27 as covered here.

**Reopen condition.** If a downstream ticket cannot honor a seam as written — for
example, if the `DurableOutput` trait pair cannot be implemented for an output
type the design requires (a coherence/orphan obstruction the registration hook
cannot route around), if the assembly witness for "output type implements the
contract" cannot be captured at registration under T0.2's typed-builder shape, or
if the three-way existence classification proves insufficient for a real
referent's failure modes — **the contract reopens here**, in this ADR, rather
than being worked around locally. A local workaround that silently diverges from
this contract (a per-task contract, a built-in object-store backend, a mid-run
dangling detection) is a defect, not a fix.

## Rejected alternatives

- **Put the contract on the TASK, not the output type** (the task implements
  "serialize my output's reference / rehydrate my output"). **Rejected:**
  single-node replay (C26/T57) and resume (C27) must rehydrate an input from a
  reference **without running the producing task** — the consumer that replays
  never instantiated the producer. If the reconstruction logic lived on the task,
  a replayed consumer could not rehydrate an input whose producer it never ran.
  Putting the contract on the **output type** means any durable value is
  reconstructable regardless of which node produced it, which is exactly what
  makes single-node replay and demand-driven resume work. It also composes with
  T0.2, where the reference already *is* the node's output value.
- **A built-in object-store / storage-backend abstraction** (dagr grows an
  `ObjectStore` trait it calls to write and read durable values, or a networked
  reference registry). **Rejected on the scope boundary:** the contract is over
  **references the task's own output type provides** — dagr **never** becomes a
  metadata store or a storage layer (arch.md permanent non-goals), and **no
  built-in remote/object backend** is introduced. Where the value physically
  lives (S3, GCS, a local file, a database) is the task's business; dagr only
  serializes, existence-checks, and rehydrates the opaque, self-describing
  reference. Durability stays **local** and composes with T0.6's local run store.
- **An existence check that FETCHES the value** (probe by downloading the object
  and checking it parses). **Rejected:** the existence check must be **cheap** —
  resume plans over many candidate references, and fetching each defeats the whole
  point ("a cheap existence check before anything is skipped," C27). The probe is
  a `HEAD`/`stat`/`exists`, never a `GET`; the fetch happens only later, on
  demand, in `rehydrate`.
- **Detect a dangling reference mid-run** (skip the node, then fail when
  rehydration is actually attempted during execution). **Rejected:** C27 is
  explicit — "a reference to a deleted object fails the resume plan, **not the
  eleventh node** of the resumed run." A mid-run discovery wastes the ten nodes'
  worth of work already done and surfaces the failure at the worst time. The
  existence check is at **plan time, before execution begins** (§7), so a dangling
  reference refuses the resume up front.
- **A single combined durability signal folded into the structural fingerprint**
  (treat marking a node durable as a topology change that gates resume).
  **Rejected:** durability is **tuning, not shape** — marking an expensive stage
  durable so it can be resumed is a *motivating* case for resume, and gating on it
  would refuse exactly the resume the operator wants. The flag lives in the
  **policy hash** (§3, per T0.7), a proceed-with-diff at resume, never a refusal.

**Reopen condition** stated in §Consequences: if a downstream ticket cannot honor
a seam as written, the contract reopens here rather than being worked around
locally.
