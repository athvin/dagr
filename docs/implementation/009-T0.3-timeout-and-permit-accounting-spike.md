# 009 · T0.3 — ADR + spike: timeout abandonment and permit accounting

> **Milestone:** M0 · **Size:** M · **Type:** decision (spike) · **Components:** C12, C14
> **Branch:** `adr/t0.3-timeout-and-permit-accounting-spike` · **Depends on:** — · **Blocks:** T21, T31, T37

## Why / context
Rust cannot kill a running thread, so a blocking or compute closure that overruns its timeout keeps executing after the framework has decided the node's fate. This ticket exists to prove — with a throwaway prototype — that the permit ledger of the admission controller (C12) can stay honest in that situation: cost stays counted against the pools until the closure actually returns, never before. It locks the per-class timeout and permit-lifecycle mechanics that arch.md §C12 (Admission controller) and §C14 (Attempt runner) mandate, so that the three consumers can implement against a settled contract rather than re-litigating it. The decision this ADR records is the concrete shape of: future-drop cancellation for await-bound work, zombie bookkeeping and deferred retry for blocking/compute work, and the reporting shape that surfaces zombie cost. It gates T21 (per-attempt timeout), T31 (admission pools and permit lifecycle), and T37 (permit-release outcome matrix tests); no production code from those tickets should assume anything this ADR does not conclude.

## Objective
Produce an ADR that fixes the timeout-abandonment and permit-accounting model, backed by a throwaway prototype that demonstrates each mechanism is expressible in Rust and behaves as claimed. The ADR is the durable deliverable; the prototype is evidence and is deleted (or quarantined outside the shipping crates) before this branch merges.

Concrete pieces of work:
- **Decide and record per-class timeout semantics.** Await-bound: the attempt future is dropped on timeout and its permit releases immediately. Blocking/compute: the attempt is *marked* timed out immediately (fate decided, event emittable) while the thread runs on as abandoned-but-running work whose permit is held until the closure returns.
- **Decide the zombie ledger model.** Specify how the admission controller keeps abandoned-but-running cost counted against every pool it drew from until the closure returns, and how the ledger observes that return (the join/handoff mechanism) without blocking the run's forward progress.
- **Decide permit-held-until-return and deferred-retry rules.** Specify that a retry of a timed-out blocking/compute node is deferred until the previous attempt's closure has returned, preventing a task instance from running concurrently with its own zombie (C1 exclusivity).
- **Decide the late-result barrier.** Specify how a result produced by an abandoned closure after its timeout is discarded — never fills an output slot and never writes scratch.
- **Decide the zombie-cost reporting shape.** Specify how many zombies exist, what cost they pin per pool, and how that surfaces (the fields/records that later feed C19's zombie-at-exit event and C23's declared-vs-measured juxtaposition) — the reporting *shape*, not its final wiring.
- **Build the throwaway prototype** that exercises each of the above against real threads and a real async runtime (tokio) and records the observed behaviour in the ADR.
- **Name the seams** the three consumers will implement against: the permit-ledger operations T31 relies on, the timeout-classification points T21 relies on, and the outcome/capacity assertions T37 relies on.

## Test plan (write these first — TDD)
These are prototype-evidence and decision-record checks. Each is an observable outcome the throwaway prototype must demonstrate (or the ADR must record), stated as setup, action, expected outcome. "Ledger" means the prototype's stand-in permit accounting; final tests live in T31/T37.

- **Await-bound timeout releases the permit immediately.** *Setup:* a prototype await-bound attempt that never completes, declaring a nonzero cost on one pool, running under a short per-attempt timeout. *Action:* let the timeout fire. *Expected:* the attempt's future is dropped, the ledger returns to its pre-admission level immediately (within the timeout plus a small bound), and no zombie is recorded.

- **Blocking timeout keeps the permit held until the closure returns.** *Setup:* a prototype blocking attempt that sleeps well past its timeout before returning, declaring a nonzero cost. *Action:* let the timeout fire, observe the ledger, then let the closure return and observe again. *Expected:* immediately after timeout the attempt is marked timed out and the ledger still counts the full declared cost (one zombie present); only after the closure actually returns does the ledger release that cost and drop the zombie count to zero.

- **Compute timeout behaves identically to blocking.** *Setup:* same as the blocking case but on the compute class. *Action:* fire the timeout, then let the closure finish. *Expected:* cost held while abandoned, released only on return — proving the abandonment path is class-shape-driven (synchronous) rather than blocking-specific.

- **The capacity invariant never lies while a zombie is live.** *Setup:* pools pinned so total capacity admits exactly one node of the declared cost. *Action:* admit a blocking node, time it out (creating a live zombie), then present a second ready node of the same cost while the zombie is still running. *Expected:* the second node is not admitted until the zombie's closure returns; at no instant does the sum of counted cost (including the zombie) exceed pinned capacity.

- **Retry is deferred until the previous closure returns.** *Setup:* a blocking node with a retry budget, whose first attempt overruns the timeout and is retry-eligible. *Action:* fire the timeout, then let the first closure return. *Expected:* no second attempt for that node is dispatched while the first closure is still running; the retry is dispatched only after the return, so the same task instance is never concurrent with its own zombie.

- **A late result from an abandoned closure is discarded.** *Setup:* a blocking attempt that, after its timeout, eventually produces a value and attempts to write scratch. *Action:* fire the timeout, then let the closure complete and hand back its would-be output. *Expected:* the output slot is never filled by that value and no scratch is written by that attempt; whatever it computed after the timeout has no observable effect.

- **Terminal state is decided exactly once.** *Setup:* a blocking node timed out per above. *Action:* observe its state at timeout and again after the closure returns. *Expected:* the state is `timed-out` at timeout and stays `timed-out` after return; no `abandoned` transition appears on the timeout path (per the ADR, `abandoned` arises only on the cancellation path, C16).

- **Zombie cost is reportable.** *Setup:* two blocking nodes timed out and still running, each with a distinct per-pool declared cost. *Action:* query the prototype's zombie-cost report. *Expected:* the report names two zombies and the exact per-pool cost each pins, in a shape the ADR fixes and that the consumers can fold into the event stream (C19) and the declared-vs-measured artifact (C23).

- **ADR decision-record completeness.** *Setup:* the finished ADR. *Action:* read it against arch.md §C12 and §C14 acceptance criteria. *Expected:* it resolves every timeout/permit-lifecycle question those sections raise, states each decision and its rationale, names the seams for T21/T31/T37, and marks the prototype as throwaway with its disposition recorded.

The shipping crates (`core`, `artifact`, `render`, `cli`) are unchanged,

The consumers it unblocks build against the contract below and re-decide none of

The model is **consistent with the two already-merged ADRs it depends on**: T2
(004 — tokio + `tokio-util` async runtime, `spawn_blocking` for blocking work, a

contradicts neither.

Every decision was validated by a **throwaway spike built outside the dagr
workspace** — a standalone Cargo project at `/tmp/dagr-t0.3-spike` depending on
`tokio` (feature `full`), `tokio-util` (feature `rt`), and `rayon`. The spike ran
one prototype per Test-plan scenario against **real threads** (`spawn_blocking`
for the blocking class, a dedicated `rayon` pool for the compute class) and a
**real multi-threaded tokio runtime**, printing the `EVIDENCE …` line each
scenario is quoted against below. The spike was **deleted before this PR**
(`rm -rf /tmp/dagr-t0.3-spike`); nothing was promoted into a shipping crate, no
dependency was added to any shipping `Cargo.toml`, and the model is reproducible
from the quoted evidence and the sketches here without the spike code.

The spike's stand-in ledger is deliberately *not* the production T31 ledger: it
is a `Ledger` holding pinned per-pool capacity and a counted-cost map, with a
`Permit` guard that returns cost to the ledger **only on `Drop`**. The load-
bearing trick — the same one T0.2 used for slot residency — is that a `Permit`
is **moved into the blocking/compute closure**, so "the work has returned" is
*definitionally* "the permit was dropped." That is what makes permit-held-until-
return fall out of Rust's ownership for free, with no watchdog and no join-blocking
of the run loop.

## Context

`docs/arch.md` makes two execution-core commitments that meet at this decision
and cannot each pick their own answer without failing to compose:

- **C12 · Admission controller.** "The permit is held for the whole attempt and
  released on every terminal outcome — with one honest exception. Work that was
  timed out or cancelled but has not actually returned (a blocking closure cannot
  be killed — C14) is **abandoned-but-running**: it still occupies its thread and
  its memory, so its cost remains counted against the pools until the closure
  actually returns. The capacity invariant counts zombies, because a ledger that
  releases what is still running is a ledger that lies, and the container's OOM
  killer audits it." Its acceptance criteria demand: combined declared cost of
  executing nodes *including abandoned-but-running work* never exceeds capacity;
  permits release "for timeout and abandonment only when the underlying work has
  actually returned"; and declared-vs-measured cost side by side in the run
  artifact.
- **C14 · Attempt runner.** "**Timeout semantics differ by class, honestly.** An
  await-bound attempt that exceeds its timeout is truly cancelled — its future is
  dropped, and its permit releases immediately. A blocking or compute attempt
  cannot be killed: on timeout the attempt is *marked* timed out immediately (the
  event is emitted, the node's fate is decided), but the thread keeps running as
  abandoned-but-running work whose permit is held until the closure actually
  returns (C12). A retry of that node is deferred until the previous attempt's
  closure has returned — the alternative is the same task instance running
  concurrently with its own zombie, which violates C1's exclusivity. An abandoned
  attempt can never fill the output slot and can never write scratch; whatever it
  computes after its timeout is discarded. A node's terminal state is decided
  exactly once — a blocking timeout is and stays `timed-out`; `abandoned` arises
  only on the cancellation path (C16), never as a second state after `timed-out`.
  If the process is exiting while leftover threads are still running, they die
  with the process, each recorded as a *zombie-at-exit event* in the stream
  (C19), which changes no node's state."

The normative **Vocabulary** pins the terminal states this ADR reasons over:
`timed-out` ("The final attempt exceeded its per-attempt timeout") and
`abandoned` ("Was asked to cancel and never returned within the grace period; its
thread was left behind. **A *timed-out* attempt whose thread never returns stays
`timed-out`** — the leftover thread is recorded as a zombie event in the stream
(C19), never as a second terminal state"). `timed-out` and `abandoned` are both
*failure-like*. The readiness tracker (C11) adds the run-end rule: "an
abandoned-but-running closure is decided, so it does not hold the run open
indefinitely — at natural run end the process waits up to the grace period (C16)
for zombies to return, then exits, recording a zombie-at-exit event for each."

This ADR is the load-bearing contract for three consumers, so it must leave no
timeout/permit-lifecycle question open. **There are no open questions** — the
ticket's `## Open questions` says "None," and the `docs/tasks.md` T0.3 entry
carries no `Q:` items (it states the deliverable: "future-drop for await-bound;
zombie-thread bookkeeping, permit-held-until-return, and deferred retry for
blocking/compute; zombie-cost reporting shape"). This ADR records each of those,
so the entry's descriptive scope is fully resolved below.

## Decision

### 1. Per-class timeout semantics — two shapes, decided by *class shape*, not class name

The distinction is **synchronous vs asynchronous**, i.e. driven by the *shape* of
the work (does it advance by being polled, or by occupying a thread), not by the
class label. Await-bound is the one shape Rust can actually cancel; blocking and
compute share the other shape and therefore share the abandonment path — the
spike proved compute behaves *identically* to blocking precisely because both are
synchronous, unkillable closures.

**Await-bound: future-drop, permit releases immediately.** An await-bound attempt
that exceeds its timeout has its **future dropped** (in the runner this is a
`select!` of the task future against a `tokio::time` timeout / the attempt's
`CancellationToken::cancelled()`; the timeout branch drops the losing future).
The permit-shaped guard is **moved into the future**, so dropping the future
drops the permit, and the cost returns to the ledger **immediately** — no zombie
is ever recorded on this path. *Spike evidence:*

> `EVIDENCE await-release: future dropped on timeout, permit released in
> 51.280458ms (<=timeout+ε), ledger mem back to 0, zombies=0`

The future never completed (a `std::future::pending`), yet a 50 ms
`tokio::time::timeout` dropped it and the ledger returned to its pre-admission
level within the timeout plus a small scheduling bound (~1 ms), with zero zombies
— exactly C14's "cancelled and its permit released immediately" and C12's
immediate release.

**Blocking and compute: marked timed-out immediately, permit held until return.**
A blocking (`spawn_blocking`) or compute (`rayon`) attempt **cannot be killed** —
dropping its `JoinHandle` does not stop the thread. On timeout the attempt is
**marked timed out immediately**: the node's fate is decided, the `timed-out`
event is emittable at once, and the late-result barrier goes up (§4). But the
thread runs on as **abandoned-but-running** work whose permit **is held until the
closure actually returns**. *Spike evidence (blocking, then compute — identical):*

> `EVIDENCE blocking-hold: at timeout mark, state=TimedOut, ledger still counts
> 200 mem (1 zombie), permit NOT released`
> `EVIDENCE blocking-hold: after closure returned, ledger mem back to 0,
> zombies=0; late fill refused=true, late scratch refused=true; state
> TimedOut->TimedOut (decided once)`
> `EVIDENCE compute-hold: at timeout mark, state=TimedOut, ledger still counts
> 200 mem (1 zombie), permit NOT released`
> `EVIDENCE compute-hold: after closure returned, ledger mem back to 0,
> zombies=0; late fill refused=true, late scratch refused=true; state
> TimedOut->TimedOut (decided once)`

The two classes produce byte-for-byte the same accounting behaviour, proving the
abandonment path is **class-shape-driven (synchronous)**, not blocking-specific.
This is C14's per-class timeout semantics and C12's "cost remains counted … until
the closure actually returns," verified against real `spawn_blocking` and real
`rayon` threads.

### 2. The zombie ledger model — cost stays counted; the return is observed by permit drop, never by blocking the run

The admission controller keeps abandoned-but-running cost counted against **every
pool it drew from** by the simplest honest mechanism: the permit is **never
released at the mark**. Release is tied to the permit's `Drop`, and the permit
lives inside the closure, so the cost is counted for exactly as long as the thread
occupies its slot. A **zombie record** — `{ node, per-pool pinned cost }` — is
registered in the ledger's zombie set when the runner marks the attempt, so the
ledger can *report* the abandoned cost (§5) independently of the raw counted-cost
totals; the record is removed when the permit drops.

**How the ledger observes the return without blocking forward progress.** The
join/handoff is the closure's own completion: `spawn_blocking` yields a
`tokio::task::JoinHandle` and `rayon::spawn` hands its result back over a channel;
the runner **`await`s that handle on the framework/task runtime** (never blocking
the run loop), and the permit-drop that fires when the closure body ends is what
the ledger sees. Because the wait is an `await`, a live zombie does **not** hold
the run open: the readiness tracker (C11) already treats an abandoned-but-running
closure as *decided*, so at natural run end the process waits up to the grace
period (C16) for zombies to return, then exits, emitting a zombie-at-exit event
(C19) for any still running — the ledger's zombie set is exactly the source of
those events.

### 3. The capacity invariant — counted cost (including zombies) never exceeds capacity; permits are all-or-nothing across pools

Admission is **all-or-nothing across every pool a node needs** (C12: "acquisition
is all-or-nothing across pools … no permit is held while waiting for another"),
and the counted total — **including every live zombie** — never exceeds pinned
capacity at any instant. *Spike evidence* (pools pinned so total capacity admits
exactly one node of the declared cost):

> `EVIDENCE capacity: zombie pins 1000 mem == capacity 1000; second same-cost
> node REFUSED admission; invariant holds`
> `EVIDENCE capacity: after zombie closure returned, mem back to 0, second node
> ADMITTED; invariant never exceeded capacity at any instant`

A node was admitted, timed out into a live zombie, and a second ready node of the
same cost was presented while the zombie ran: the second node was **not admitted**
until the zombie's closure returned, and the sum of counted cost (zombie included)
never exceeded capacity — C12's "combined declared cost of executing nodes,
*including abandoned-but-running work*, never exceeds pool capacity." The ledger
mechanism that guarantees it is the drop-tied permit of §2: because release can
only happen by the closure returning, the ledger structurally cannot release what
is still running.

### 4. The late-result barrier — an abandoned closure's late result never fills a slot and never writes scratch

The instant the attempt is marked timed out, a **barrier** goes up on its output
slot and scratch handle. Any `fill()` or scratch write the abandoned closure
attempts *after* the mark is **refused** — whatever it computes after its timeout
has no observable effect. In the spike the slot's `fill`/`write_scratch` both
consult a `timed_out` flag raised at the mark; the closure ran on past its
timeout, produced a value, and tried to write both:

> (from the `blocking-hold`/`compute-hold` evidence above)
> `… late fill refused=true, late scratch refused=true …`

The output slot is never filled by that value and no scratch is written by that
attempt — C14's "an abandoned attempt can never fill the output slot and can never
write scratch." This composes with T0.2's slot (008): the slot is once-fillable
and its value is released only when every consumer's work has actually returned,
so the barrier here is the *producer-side* mirror of that *consumer-side* rule.

### 5. Deferred retry — a timed-out node's retry waits for its previous closure to return (C1 exclusivity)

A retry of a timed-out blocking/compute node is **deferred until the previous
attempt's closure has returned**. The rule is enforced by the zombie set: while a
node has a live zombie for its instance, no next attempt is dispatched — the
alternative is the same task instance running concurrently with its own zombie,
which violates C1's `&mut self` exclusivity (a value/`&mut self` in the running
zombie and in a fresh attempt at once). *Spike evidence:*

> `EVIDENCE deferred-retry: attempt-1 zombie live; attempt-2 NOT dispatched
> (would be concurrent with its own zombie)`
> `EVIDENCE deferred-retry: after attempt-1 closure returned, attempt-2
> dispatched — never concurrent with its own zombie`

Timeout is retry-eligible by default (C14), so the retry *does* eventually run —
it is only **deferred**, not cancelled, and it dispatches the moment the previous
closure's permit drops.

### 6. Terminal state is decided exactly once — a blocking timeout is and stays `timed-out`; `abandoned` is a C16-only state

A node's terminal state is decided **exactly once**. A blocking/compute timeout is
`timed-out` at the mark and **stays `timed-out`** after the closure returns — the
return is a permit/ledger event, **not** a state transition. There is **no
`abandoned` transition on the timeout path**: per the normative Vocabulary, "a
*timed-out* attempt whose thread never returns stays `timed-out`," and `abandoned`
arises **only on the cancellation path** (C16) — a node that was asked to *cancel*
and did not return within grace. *Spike evidence* (state observed at the mark and
again after return):

> (from the `blocking-hold`/`compute-hold` evidence above)
> `… state TimedOut->TimedOut (decided once)`

If leftover threads are still running when the process exits, they die with the
process and each is recorded as a **zombie-at-exit event** in the stream (C19),
which changes no node's state — the state was already `timed-out`.

### 7. Zombie-cost reporting shape — count of live zombies and the per-pool cost each pins

The ledger exposes a **zombie-cost report**: a list of live-zombie records, each
`{ node, per-pool pinned cost }`, from which the count of live zombies and the
total per-pool cost they pin are derivable. *Spike evidence* (two blocking nodes
timed out and still running, each with a distinct per-pool declared cost):

> `EVIDENCE zombie-report: 2 live zombies pinning per-pool cost:
> [zombie-alpha{mem=300,threads=1}, zombie-beta{mem=500,threads=2}]`

This is the **shape** the ADR fixes, not its final wiring. Its two downstream
consumers:

- **C19 · Event stream** folds each record into a **zombie-at-exit event** (the
  node identity and the cost it still pinned at exit); the readiness-tracker /
  cancellation run-end path emits one per still-running zombie.
- **C23 · Node metrics / C22 run artifact** fold the per-pool pinned cost into
  the **declared-vs-measured juxtaposition** so a run's zombie-pinned capacity and
  time are visible next to declared cost (the run artifact "juxtaposes declared
  against measured cost per node"). The reporting fields are: `live_zombie_count`
  and, per zombie, `node` + per-pool `pinned` cost.

### 8. The cancellation-token seam is consistent with T2, and the two paths are distinct

The primitive is T2's `tokio_util::sync::CancellationToken` with a per-attempt
`child_token()`. A blocking/compute closure is handed its child token and can
**observe** it cooperatively, but the framework can only **mark** it — never kill
it — so the *same* permit-held-until-return accounting governs the cancellation
path too. *Spike evidence:*

> `EVIDENCE cancel-seam: run.cancel() marks child; blocking closure
> cooperative-only, permit held at mark and released only on return;
> child.cancel() leaves parent uncancelled`

The **difference between the two paths is the terminal state, not the accounting**:
timeout → `timed-out` (this ADR); cooperative-cancel-then-return-within-grace →
`cancelled`, no-return-within-grace → `abandoned` (C16, out of scope here). Both
paths hold the permit until the closure returns; only C16 decides `cancelled`
vs `abandoned`. Recording this keeps the ADR honest that the *mechanism* it
proves is shared, while the *classification* on the cancellation path is C16's to
own.

### 9. The seams the three consumers implement against

- **T21 (C14 per-attempt timeout) — the timeout-classification points.** The
  per-class fork of §1: await-bound → `select!`-drop the future (permit releases
  immediately); blocking/compute → mark timed-out immediately + raise the
  late-result barrier (§4) + register a zombie (§2) + hold the permit until the
  `JoinHandle`/channel resolves; retry deferred until that return (§5); terminal
  state `timed-out`, decided once (§6). Timeout is retry-eligible by default.
- **T31 (C12 admission pools and permit lifecycle) — the permit-ledger
  operations.** `try_admit(node, cost) -> Option<Permit>` (all-or-nothing across
  pools); a `Permit` whose `Drop` returns cost to every pool it drew from;
  `mark_zombie(&permit)` registering a `{node, per-pool cost}` record without
  releasing; `zombie_report()`; and the invariant that counted cost (zombies
  included) ≤ capacity at every instant (§3). The permit is held for the whole
  attempt and released on every terminal outcome except that timeout/abandonment
  release only on actual closure return.
- **T37 (C12 permit-release outcome-matrix tests) — the outcome/capacity
  assertions.** For each induced outcome (success, permanent failure, retryable
  failure, timeout in *both* classes, panic, cooperative cancellation,
  abandonment): assert the ledger released exactly when it should (immediately for
  await-bound timeout; only on closure return for blocking/compute timeout and
  abandonment), that the capacity invariant never exceeded pinned capacity while a
  zombie was live, and that slot residency pinned by a zombie consumer stays
  counted until its closure returns (composing with T0.2/T17).

## Consequences

- **T21, T31, and T37 inherit a closed contract.** T21 forks timeout by class
  shape against §1/§4/§5/§6; T31 builds the permit ledger against §2/§3 and the
  operations in §9; T37 asserts the outcome/capacity matrix in §9 — none re-opens
  a decision here.
- **Permit-held-until-return is structural, not policed.** Because release is tied
  to `Drop` and the permit lives in the closure, the ledger *cannot* release what
  is still running — there is no watchdog to get wrong and no join that blocks the
  run loop (the return is observed by `await`ing the `JoinHandle`/channel).
- **The capacity invariant is honest under abandonment.** Counted cost includes
  zombies, so the container's OOM killer never finds the ledger lying — the C12
  acceptance criterion holds by construction.
- **The two abandonment paths share one mechanism.** Timeout (this ADR) and
  cooperative cancellation (C16) both hold the permit until the closure returns;
  they differ only in the terminal state (`timed-out` vs `cancelled`/`abandoned`),
  which keeps C16's classification cleanly separable from this ADR's accounting.
- **The zombie-cost report has a fixed shape** (`live_zombie_count`; per zombie
  `node` + per-pool `pinned` cost) that C19 folds into zombie-at-exit events and
  C23/C22 fold into the declared-vs-measured juxtaposition — the *shape* is fixed
  here, the *wiring* is C19/C23's.
- **Spike is throwaway.** The `/tmp/dagr-t0.3-spike` project that proved all of
  the above against real `spawn_blocking`, `rayon`, and tokio threads is deleted;
  nothing was promoted into a shipping crate, `Cargo.lock` is unchanged, and
  `cargo audit` has no new surface. The surviving record is this ADR's quoted
  evidence plus
  [`scripts/check-timeout-and-permit-adr.sh`](../../scripts/check-timeout-and-permit-adr.sh).

### Scope — what this ADR deliberately does NOT decide

- **Cancellation-path `abandoned` semantics (C16).** This ADR covers the
  *timeout* path; the `abandoned` terminal state, the grace period, and the
  cooperative-cancellation fan-out belong to C16 (implemented in T35). This ADR
  only records that the *permit-held-until-return mechanism* is shared, and that
  `abandoned` never arises on the timeout path.
- **Container limit detection and pool sizing (T32, C12 bootstrap).** Prototype
  pools are pinned to make capacity deterministic; deriving pool sizes from
  cgroup v2/v1/host with a headroom fraction is T32's detection logic, out of
  scope here.
- **Final event-stream and run-artifact wiring (C19, C23).** This ADR fixes the
  *shape* of the zombie-cost report, not its emission into the durable event
  stream (C19) or its fold into the run artifact (C22/C23).
- **The production ledger, timeout runner, and outcome-matrix suite (T31, T21,
  T37).** This ADR ships only the decision and the throwaway prototype; the real
  permit ledger, per-attempt timeout machinery, and exhaustive release matrix are
  those consumers' deliverables.
- **Killing threads, a zombie-reaping watchdog, or cross-process capacity
  coordination.** Permanently out of dagr's scope (arch.md non-goals): the honest
  answer to an unkillable thread is to keep counting its cost until it returns,
  not to invent a way to terminate it.

## Rejected alternatives

- **Release the permit at the timeout mark (optimistic release).** Rejected: it is
  the exact "ledger that lies" C12 forbids. The thread and its memory are still
  occupied, so releasing at the mark lets the controller admit a second node whose
  cost the machine cannot actually satisfy — the container's OOM killer audits the
  ledger, and this alternative fails that audit. The spike's capacity scenario
  shows admission correctly *refused* while the zombie was live; optimistic
  release would have wrongly admitted it.
- **Kill the abandoned thread (abort it, or drop the `JoinHandle` and pretend).**
  Rejected: Rust cannot kill a running thread, and dropping a `spawn_blocking`
  `JoinHandle` does not stop the closure — the thread runs on regardless. Any
  design that assumes the thread stops on drop is unsound; a "watchdog that reaps
  zombies" is a scheduler concern outside dagr's permanent scope. The honest model
  is to count the cost until the closure returns.
- **A single shape for all classes (cancel everything by future-drop).** Rejected:
  blocking and compute work is synchronous and unkillable, so there is no future
  to drop that would stop the thread — forcing them through the await-bound path
  would either block an async worker (starving the runtime, which T2 forbids) or
  lie about having cancelled them. The spike shows the compute path behaving
  identically to blocking *because* both are synchronous, which is exactly why
  they need the abandonment path and await-bound does not.
- **Allow the retry to start while the previous closure still runs (concurrent
  retry).** Rejected: it puts the same task instance on two threads at once,
  violating C1's `&mut self` exclusivity (the task value cannot be `&mut`-borrowed
  by both). Deferring the retry until the previous closure returns is the only way
  to keep a task instance from running concurrently with its own zombie; the spike
  proves the retry still dispatches, just after the return.
- **Emit a second terminal state (`abandoned`) when a timed-out thread never
  returns.** Rejected: it contradicts the normative Vocabulary ("a *timed-out*
  attempt whose thread never returns stays `timed-out`") and would give a node two
  terminal states. The leftover thread is a **zombie-at-exit event** in the stream
  (C19), not a state change — the state was decided exactly once, at the timeout
  mark.
- **Track zombie cost only in aggregate (a single counter), not per-node
  records.** Rejected: C19's zombie-at-exit event needs the *identity* of each
  zombie and C22/C23 need the *per-pool* cost each pins next to that node's
  declared cost. A single aggregate counter cannot produce a per-node event or a
  per-node declared-vs-measured row, so the report shape must be a list of
  `{node, per-pool cost}` records, which the spike's `zombie_report` demonstrates.
