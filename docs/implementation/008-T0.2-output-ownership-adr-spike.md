# 008 · T0.2 — ADR + spike: output ownership and sharing model

> **Milestone:** M0 · **Size:** M · **Type:** decision (spike) · **Components:** C1, C3, C10
> **Branch:** `adr/t0.2-output-ownership-adr-spike` · **Depends on:** — · **Blocks:** T5, T9, T11, T17, T26

## Why / context
Every value that flows between nodes crosses an ownership boundary, and the spec has *chosen* a three-part model — sole-consumer-owns, multi-consumer-shared-read, and per-edge clone-on-read opt-in — that splits type checking (compile time, C3) from mode checking (assembly time, C1) and constrains delivery so an owned value moved into a failed attempt can never need re-forming (C1, C10). This is the single highest-risk feasibility item in the plan: the model is only defensible if it survives contact with Rust's borrow checker across a non-`Clone` output, a retry after a consumer has already read, and the author-visible trait bounds it forces onto task and output types. This ticket exists to *prove* that with a throwaway prototype and then lock the result as an Architecture Decision Record before the handle/binding spike (T5), the task abstraction (T9), the data-dependency binding (T11), the output slot (T17), and the metrics work (T26) are written against it. It governs C1's "Ownership of inputs" rules, C3's type-vs-mode split, and C10's slot-lifetime and residency accounting. If the model does not survive the borrow checker here, the spec decision reopens — that reopen-or-confirm outcome is the deliverable.

## Objective
Prototype the spec's chosen ownership model to destruction, then record the surviving design as a committed ADR that leaves no ownership choice open to T5, T9, T11, T17, or T26. Concretely:

- **Prove the delivery mechanism.** Establish, in disposable code, the representation by which a producer's single output slot (C10) hands its value to consumers in each of the three modes — moved out for a sole owner, shared read access for the duration of an attempt for multiple consumers, and a fresh per-attempt value for a clone-on-read edge — and confirm each compiles and each *conflict* is rejectable.
- **Break it on a non-`Clone` type.** Drive the prototype with an output type that is deliberately not `Clone` and show that sole-consumer-owns and multi-consumer-shared-read both work on it, while demanding clone-on-read on that type is caught (a compile error for the missing `Clone` bound, or a curated assembly error if the mode is chosen dynamically) — this is the case that most often collapses a naive design.
- **Break it on retry-after-consumer-failure.** Show that a shared-access consumer that reads the value and then fails a retry-eligible attempt finds the value intact on its next attempt (the slot still holds it), and that an owned-input edge into a node with a nonzero retry count is *rejected* unless the edge opts into clone-on-read — because a value moved into a failed attempt cannot be re-formed.
- **Pin the author-visible bounds.** Fix and document the trait bounds the model forces on authors: task values `Send + 'static` with the work taking `&mut self`; output values `Send + Sync + 'static`; clone-on-read edges additionally requiring the output type be `Clone`. Include the worked first-hour error (capturing a non-`Send` value in a task) as narrative, not shipping code.
- **Decide the compile-time / assembly-time partition.** Record exactly which violations are compile errors (type mismatch, missing `Clone` for a statically-clone-on-read edge, missing `Send`/`Sync`) and which are assembly errors (owned demand on a multiply-consumed value naming both consumers; owned edge into a retrying node without clone-on-read; a second consumer registered against a value whose sole consumer takes ownership), so C1, C3, T11, and T17 inherit one boundary instead of re-litigating it.
- **Record it.** Write the ADR at the ticket `path` with status, context, decision, consequences, and rejected alternatives, and quarantine or delete the prototype so it never becomes shipping surface.

## Test plan (write these first — TDD)
These are the evidence checks that validate the decision. Each pairs a throwaway prototype (kept out of the shipping crates, or gated so it never becomes production surface, and discarded or quarantined once the ADR is blessed) or a documentation-record assertion with a concrete, independently checkable outcome.

- **Sole-consumer-owns delivery evidence.** Setup: a prototype producer slot holding one value and exactly one registered consumer declared to receive by ownership. Action: run the consumer's single attempt. Expected outcome: the consumer receives the value by move, the slot is left with no copy afterward, and the code compiles — validating C1's "a sole consumer can receive its input by value."
- **Multi-consumer-shared-read delivery evidence.** Setup: a prototype producer slot with two consumers, both declared to receive by shared read access. Action: run both attempts, each reading the value concurrently. ## Context

`docs/arch.md` has *already chosen* a three-part ownership model in its C1
"Ownership of inputs" rules and its Decisions changelog ("Input ownership:
sole-consumer-owns / multi-consumer-shared-read / per-edge clone-on-read opt-in;
`&mut self` task exclusivity; `Send + Sync + 'static` outputs"). Three arch.md
components meet at this decision and cannot each pick their own answer without
failing to compose:

- **C1 · Task** fixes the receive-mode rules: a multiply-consumed value is
  delivered by shared read access; demanding ownership of it fails assembly
  *naming both consumers* unless the edge opts into clone-on-read (requiring
  `Clone`); an owned-input edge into a node with a nonzero retry count fails
  assembly unless it opts into clone-on-read, "because a value moved into a
  failed attempt cannot be re-formed"; task values are `Send + 'static` with work
  taking `&mut self`; output values are `Send + Sync + 'static`.
- **C3 · Data dependency** fixes the type-vs-mode split: the *value types* of
  bound handles must exactly match (a mismatch is a compile error), but the
  *receive mode* "is not part of type matching" — mode conflicts are whole-graph
  facts rejected at assembly. Type checking and mode checking stay on separate
  rails.
- **C10 · Output slot** fixes the residency rules: each node owns exactly one
  slot; the value is released only when **every consumer is terminal and every
  consumer's work has actually returned** (so a shared-access consumer that read
  the value and then failed a retry-eligible attempt finds it intact next time,
  and a zombie consumer pins the value until its closure returns); the value is
  counted **once** against the memory pool, not once per consumer.

This is the plan's single highest-risk feasibility item: the model is defensible
only if it survives contact with the borrow checker across a **non-`Clone`**
output, a **retry-after-a-consumer-has-read**, and the **author-visible trait
bounds** it forces. The spike existed to prove exactly that, and the reopen
condition (below) was the honest escape hatch had it failed.

## Decision

### 1. The three delivery modes, and the representation that carries them

A produced value lives in a **once-fillable slot** that Arc-wraps the value once
(the dagx §3 "wrap once, hand out cheap clones" pattern). Delivery to a consumer
is one of three modes; the mode is a property of the **edge**, decided at
assembly, and is *not* part of the value's type:

- **sole-consumer-owns** — the value is **moved out** of the slot. Legal only
  when the value has exactly one consumer that demanded ownership. After the
  move the framework has no copy left. Representation: `Arc::try_unwrap` (the
  slot is provably the sole strong holder, so it succeeds) → the owned `T`.
- **multi-consumer-shared-read** — each consumer receives an `Arc<T>` clone
  (O(1)), giving `&T` read access for the duration of its attempt; no consumer
  can move or mutate the value.
- **per-edge clone-on-read** — the edge receives a **fresh** `T` per attempt via
  `T::clone`; requires `T: Clone`. This is the only mode that demands `Clone`.

*Sole-owns evidence:* `EVIDENCE sole-owns: consumer received 64 bytes by move;
slot consumed, no copy left`. *Shared-read evidence:* `EVIDENCE shared-read: two
consumers read the same value (len 64 and 64), strong_count=3 (slot + 2 readers),
read-only`, and across threads `EVIDENCE concurrent-shared: 4 threads
concurrently read the same output, lens [256, 256, 256, 256]` — proving the
`Send + Sync` output bound is what makes concurrent shared reads sound.

### 2. Non-`Clone` outputs flow through the two common modes; only clone-on-read demands `Clone`

The spike drove a deliberately non-`Clone` output type (`NonCloneOutput`, which
*is* `Send + Sync + 'static` but has **no** `Clone` impl) through both common
modes. Both compiled and ran with no `Clone` bound demanded:
`EVIDENCE nonclone-modes: NonCloneOutput (no Clone impl) flowed through shared
(len 128) and owned (len 128) with no Clone bound demanded`. This is the case
that most
often collapses a naive design (a design that reaches for `Clone` everywhere);
here it does not, because shared read is `Arc` clone (of the pointer, never the
value) and owned is a move.

**Demanding clone-on-read on a non-`Clone` type is rejected, never silently, on
both paths:**

- **Static edge → compile error.** When the mode is expressed statically, the
  `fresh()` method (bounded `impl<T: Clone> Slot<T>`) is simply *not callable* on
  a non-`Clone` slot. The real error is
  `error[E0599]: the method `fresh` exists for struct `Slot<NonCloneOutput>`, but
  its trait bounds were not satisfied`, with the note `trait bound
  `NonCloneOutput: Clone` was not satisfied` and the actionable
  `help: consider annotating `NonCloneOutput` with `#[derive(Clone)]``. This is a
  checked-in compile-failure test per C28/C3 conventions in T12.
- **Dynamic edge → curated assembly error.** When the mode is chosen at
  registration, the missing `Clone` cannot be a compile error, so assembly emits
  a curated message naming the type: `EVIDENCE dyn-clone-on-read: assembly error:
  edge into value `payload` requests clone-on-read, but its output type
  `NonCloneOutput` is not Clone; clone-on-read requires the output type to
  implement Clone. Either make `NonCloneOutput` Clone, or receive `payload` by
  shared access.` — never a wall of raw trait errors.

### 3. The compile-time / assembly-time partition (the load-bearing table)

Type checking and mode checking are on **separate rails**. The **receive mode is
part of the task's signature** (visible in its declaration — a consumer declares
whether it takes `T` by ownership, `&T`/shared, or a fresh clone) and is **not
part of C3 type matching** (C3 matches only the *value type* `T`). This is why
mode conflicts cannot be compile errors: they are whole-graph facts (consumer
counts, retry policy) that exist only once every registration is in.

| Violation | Rejected at | Why |
|---|---|---|
| **Type mismatch** (bound handle's `T` ≠ consumer's declared input `T`) | **compile time** (C3) | a per-edge fact the compiler already has; message contains both type names (C3/T5/T8) |
| **Missing `Clone` on a statically-declared clone-on-read edge** | **compile time** | the `fresh()`/clone bound is unsatisfiable — real `E0599`, with the `#[derive(Clone)]` help |
| **Missing `Send` / `Sync` bound** (task not `Send`, output not `Send + Sync`) | **compile time** | an auto-trait fact the compiler proves at the bound — real `E0277` |
| **Owned demand on a multiply-consumed value** | **assembly time** | consumer count is a whole-graph fact; the error **names both consumers** of the shared value |
| **Second consumer registered against a sole-owner value** | **assembly time**, at the new registration | attributed to the *new* registration, not a silent change of the first consumer's mode |
| **Owned-input edge into a node with a nonzero retry count** (no clone-on-read) | **assembly time** | a value moved into a failed attempt cannot be re-formed; retry budget is policy, a whole-graph fact |

The mode-in-signature-not-in-type-matching split means **C1, C3, T11, and T17
inherit one boundary** and never re-litigate it: C3 does type matching only; the
assembly mode-checker (built in T14, against slot references from T17) owns every
row in the "assembly time" half.

### 4. Assembly rejects the three mode conflicts, precisely and by name

The spike's assembly checker reproduced each assembly-time rejection with a
curated, complete message:

- **Owned demand on a multiply-consumed value fails, naming both.**
  `EVIDENCE owned-on-shared: assembly error: node `reducer` demands ownership of
  value `stats`, but `stats` is also consumed by node `auditor`; a multiply-consumed
  value is delivered by shared read access. Either give `reducer` shared access,
  or opt this edge into clone-on-read (requires the output type to be Clone).`
  This is a *whole-graph* fact — it does not (and cannot) fail earlier at compile
  time, validating C1's "demanding ownership of a shared value fails at assembly
  (naming both consumers)."
- **Second consumer against a sole-owner value fails at the new registration.**
  `EVIDENCE second-consumer: assembly error (at registration of `auditor`): value
  `stats` already has a sole consumer `reducer` that takes ownership; registering
  a second consumer `auditor` would change `reducer`'s delivery silently. Make
  `reducer` shared, or opt into clone-on-read.` The conflict is attributed to the
  *new* registration, confirming it is an assembly error at that point rather than
  a silent change of the first consumer's delivery mode.
- **Owned edge into a retrying node is rejected without clone-on-read.**
  `EVIDENCE owned-into-retry: assembly error: node `loader` takes owned input
  from value `payload` and declares 3 retries; a value moved into a failed
  attempt cannot be re-formed. Either drop retries on `loader`, receive `payload`
  by shared access, or opt this edge into clone-on-read (requires Clone).`

### 5. Retry survives: shared readers find the value intact; clone-on-read gives each attempt a fresh value

- **Shared-access retry finds the value intact.** A shared consumer read the
  value, "failed" a retry-eligible attempt (dropped its `Arc`), and read again on
  the next attempt: `EVIDENCE shared-retry: attempt 1 saw len 64, failed; attempt
  2 found the value STILL present (len 64) in the same slot`. The slot still holds
  the value because release waits for terminal state, not last read (C10) — see §6.
- **Clone-on-read makes a retrying owned edge legal and fresh per attempt.** With
  clone-on-read stated on the edge and a `Clone` output type, assembly passes and
  each attempt gets an independent clone: `EVIDENCE clone-on-read: assembly
  passed=true; attempt 1 clone counter=100 but attempt 2 fresh clone counter=0
  (mutation did not leak)` — validating C1's "each attempt receives a fresh
  clone."

### 6. Author-visible bounds — fixed and documented

The model forces exactly these bounds, and no more:

- **Task values: `Send + 'static`, work takes `&mut self`.** `Send + 'static`
  lets a task value move to another worker thread and forbids borrowed data;
  `&mut self` makes sequential attempts safe **without any synchronization by the
  author** (attempt *n+1* never overlaps attempt *n*, per C1). *Evidence:*
  `EVIDENCE bounds-good: GoodTask: Send+'static and NonCloneOutput/CloneOutput:
  Send+Sync+'static all satisfied; &mut self work returned 15`.
- **Output values: `Send + Sync + 'static`.** `Sync` is what makes the concurrent
  shared reads of §1 sound; `Send + 'static` lets the value live in the slot and
  move between threads. This bound is demanded **only** by the shared-read mode's
  concurrency; it is not a `Clone` demand.
- **Clone-on-read edges additionally require `Clone`** — and nothing else does.

**The worked first-hour error (capturing a non-`Send` value in a task).** The
single most common first-hour mistake is capturing a non-`Send` value (an `Rc`, a
`RefCell`-of-non-`Send`, a raw pointer, a `MutexGuard`) in a task. The compiler
catches it at registration with a real, actionable error — the spike captured it
verbatim:

```
error[E0277]: `Rc<u32>` cannot be sent between threads safely
  |
6 |     let task = move |x: u32| -> u32 { x + *shared };
  |                -------------------- within this `{closure@…}`
7 |     assert_task(task);
  |     ----------- ^^^^ `Rc<u32>` cannot be sent between threads safely
  = help: within `{closure@…}`, the trait `Send` is not implemented for `Rc<u32>`
note: required by a bound in `assert_task`
```

The fix, documented in the cookbook (arch.md "Documentation": "the non-`Send`
capture error and its fixes"): construct the non-`Send` value **inside** the
task's work rather than capturing it, or wrap a genuinely shared resource in the
C9 registry (which holds `Send + Sync` clients) rather than in the task value.
The symmetric output-bound error — an output that is `Send` but not `Sync`
(e.g. a `Cell`) — is likewise a compile error: `error[E0277]: `Cell<u32>` cannot
be shared between threads safely`. And a shared-access consumer that tries to
move the value out of its `Arc<T>` is rejected: `error[E0507]: cannot move out
of an `Arc``.

### 7. The slot representation can express C10's zombie-aware release and single-count residency

Release fires only when **every consumer is terminal AND every consumer's work
has actually returned** — the second condition is what keeps a zombie
(abandoned-but-running, C14) consumer from freeing bytes its leftover thread
still pins. The spike modelled the shape with two independent counters (terminal
count, returned count) and the rule that release requires both to reach the
consumer total: `EVIDENCE residency-shape: releasable at terminal-mark = false,
releasable after closure-return = true`. Marking a consumer terminal while its
closure has not yet returned does **not** make the value releasable; only the
closure returning does. The value is counted **once** against the memory pool
(the slot holds one `Arc<T>`, not one copy per consumer), so residency is honest
and single-count. This is a *shape* check only: the full memory ledger,
consumer-count bookkeeping, and residency accounting are T17's implementation.

## Consequences

- **T5, T9, T11, T17, and T26 inherit a closed decision.** T5 encodes the typed
  handle/binding against three receive modes that are part of the task signature
  and orthogonal to type matching; T9 declares tasks with the fixed bounds
  (`Send + 'static`, `&mut self` work) and outputs (`Send + Sync + 'static`),
  clone-on-read adding `Clone`; T11 binds data dependencies doing *type* matching
  only, leaving every mode conflict to assembly; T17 builds the once-fillable
  Arc-wrapping slot with the terminal-AND-returned release rule and single-count
  residency; T26 records the measured single-count peak residency the slot
  exposes. **No ownership or receive-mode choice is left open to any of them.**
- **The compile/assembly partition is fixed for T14.** T14's assembly validation
  owns the three assembly-time rows (owned-on-multiply-consumed naming both,
  second-consumer-on-sole-owner at the new registration, owned-into-retrying
  without clone-on-read); the compiler owns the three compile-time rows (type
  mismatch, missing `Clone` on a static clone-on-read edge, missing
  `Send`/`Sync`). T12's compile-failure suite checks the compile-time rows;
  T14's assembly tests check the assembly-time rows.
- **`Clone` is an opt-in, not a tax.** The two common modes never demand `Clone`,
  so authors of large or unique outputs pay nothing; only an author who *chooses*
  clone-on-read (to make a retrying node's owned input re-formable, or to isolate
  attempts) accepts the memory multiplication, and does so per edge, visibly.
- **The cookbook carries the non-`Send` first-hour error and its fixes** (arch.md
  "Documentation"), quoting the E0277 above and the registry/construct-inside
  remedies.
- **Spike is throwaway.** The `/tmp/dagr-t0.2-spike` project that proved all of
  the above is deleted; nothing was promoted into a shipping crate, `Cargo.lock`
  is unchanged, and `cargo audit` has no new surface. The surviving record is
  this ADR's quoted evidence plus
  [`scripts/check-output-ownership-adr.sh`](../../scripts/check-output-ownership-adr.sh).

## Rejected alternatives

- **Always `Arc`-wrap everything (deliver every input as `Arc<T>`, never move,
  never own).** Rejected. It would force `Send + Sync` on *every* output
  regardless of whether it is ever shared, tax a sole consumer that could have
  taken its input by value (and thereby lost the ability to *consume* it — e.g.
  `into_bytes()`), and — worst — it makes the residency ledger dishonest by
  hiding true single-consumer moves behind a refcount. The chosen model reserves
  `Arc` for the shared-read mode where it is exactly right, and lets a sole
  consumer *own* its input, which the spike proved works on a non-`Clone` type.
- **Always `Clone` per consumer (hand each consumer a fresh clone).** Rejected. It
  demands `Clone` on every output — precisely the bound the spike showed the two
  common modes do **not** need — multiplies memory by the consumer count (defeating
  C10's honest single-count residency and C12's capacity ledger), and turns a
  cheap fan-out (N `Arc` pointer-clones) into N deep copies. Clone-on-read exists
  as a deliberate *per-edge opt-in* for the narrow cases that need a fresh value
  (a retrying owned edge; attempt isolation), not as the default.
- **Put the receive mode into the type (e.g. distinct `Owned<T>` / `Shared<T>`
  input types) so mode conflicts become compile errors.** Rejected. Mode
  conflicts depend on *whole-graph* facts (how many consumers a value has, a
  node's retry budget) that do not exist until every registration is in, so they
  cannot be sound compile errors; forcing them into the type would either miss
  real conflicts or reject legal graphs, and it would entangle C3 type matching
  with mode checking, defeating the separate-rails split. The mode lives in the
  task *signature* (visible, declared) and is checked at assembly.

**Reopen condition (did not fire).** If any prototype above could not be made to
compile, or could not be made to reject as specified — in particular if the
borrow checker could not carry a non-`Clone` output through both common modes, or
if the retry-after-read case could not find its value intact, or if the
author-visible bounds proved unworkable — then the spec's ownership decision
**reopens** rather than being papered over. Every prototype compiled or rejected
exactly as required, so the decision is **confirmed**.
