# 008 · T0.2 — ADR + spike: output ownership and sharing model

> **Milestone:** M0 · **Size:** M · **Type:** decision (spike) · **Components:** C1, C3, C10
> **Branch:** `adr/t0.2-output-ownership-adr-spike` · **Depends on:** — · **Blocks:** T5, T9, T11, T17, T26

## Why / context
Every value that flows between nodes crosses an ownership boundary, and the spec has *chosen* a three-part model — sole-consumer-owns, multi-consumer-shared-read, and per-edge clone-on-read opt-in — that splits type checking (compile time, C3) from mode checking (assembly time, C1) and constrains delivery so an owned value moved into a failed attempt can never need re-forming (C1, C10). This is the single highest-risk feasibility item in the plan: the model is only defensible if it survives contact with Rust's borrow checker across a non-`Clone` output, a retry after a consumer has already read, and the author-visible trait bounds it forces onto task and output types. This ticket exists to *prove* that with a throwaway prototype and then lock the result as an Architecture Decision Record before the handle/binding spike (T5), the task abstraction (T9), the data-dependency binding (T11), the output slot (T17), and the metrics work (T26) are written against it. It governs C1's "Ownership of inputs" rules, C3's type-vs-mode split, and C10's slot-lifetime and residency accounting. If the model does not survive the borrow checker here, the spec decision reopens — that reopen-or-confirm outcome is the deliverable.

## Objective
Prototype the spec's chosen ownership model to destruction, then record the surviving design as a committed ADR that leaves no ownership choice open to T5, T9, T11, T17, or T26. Concretely:

- **Prove the delivery mechanism.** Establish, in disposable code, the representation by which a producer's single output slot (C10) hands its value to consumers in each of the three modes — moved out for a sole owner, shared read access for the duration of an attempt for multiple consumers, and a fresh per-attempt value for a clone-on-read edge — and confirm each compiles and each *conflict* is rejectable.
- **Break it on a non-`Clone` type.** Drive the prototype with an output type that is deliberately not `Clone` and show that sole-consumer-owns and multi-consumer-shared-read both work on it, while demanding clone-on-read on that type is caught (a compile error for the missing `Clone` bound, or a curated assembly error if the mode is chosen dynamically) — this is the case that most often collapses a naive design.
- **Break it on retry-after-consumer-failure.** Show that a shared-access consumer that reads the value and then fails a retry-eligible attempt finds the value intact on its next attempt (the slot still holds it), and that an owned-input edge into a node with a nonzero retry count is *rejected* unless the edge opts into clone-on-read — because a value moved into a failed attempt cannot be re-formed.
- **Pin the author-visible bounds.** Fix and document the trait bounds the model forces on authors: task values `Send + 'static` with the work taking `&mut self`; output values `Send + Sync + 'static`; clone-on-read edges additionally requiring the output type be `Clone`. Include the worked first-hour error (capturing a non-`Send` value in a task) as narrative, not shipping code.
- **Decide the compile-time / assembly-time partition.** Record exactly which violations are compile errors (type mismatch, missing `Clone` for a statically-clone-on-read edge, missing `Send`/`Sync`) and which are assembly errors (owned demand on a multiply-consumed value naming both consumers; owned edge into a retrying node without clone-on-read; a second consumer registered against a value whose sole consumer takes ownership), so C1, C3, T11, and T17 inherit one boundary instead of re-litigating it.
- **Record it.** Write the ADR at the ticket `path` with status, context, decision, consequences, and rejected alternatives, and quarantine or delete the prototype so it never becomes shipping surface.

## Test plan (write these first — TDD)
These are the evidence checks that validate the decision. Each pairs a throwaway prototype (kept out of the shipping crates, or gated so it never becomes production surface, and discarded or quarantined once the ADR is blessed) or a documentation-record assertion with a concrete, independently checkable outcome.

- **Sole-consumer-owns delivery evidence.** Setup: a prototype producer slot holding one value and exactly one registered consumer declared to receive by ownership. Action: run the consumer's single attempt. Expected outcome: the consumer receives the value by move, the slot is left with no copy afterward, and the code compiles — validating C1's "a sole consumer can receive its input by value."
- **Multi-consumer-shared-read delivery evidence.** Setup: a prototype producer slot with two consumers, both declared to receive by shared read access. Action: run both attempts, each reading the value concurrently. Expected outcome: both read the same value through shared access, neither can mutate or move it, and the code compiles — validating C1's "a multi-consumer input is received as shared access."
- **Non-`Clone` output survives owned and shared modes.** Setup: a prototype output type that deliberately does not implement `Clone`. Action: route it through the sole-consumer-owns path and, separately, through the multi-consumer-shared-read path. Expected outcome: both compile and run; the non-`Clone`-ness never blocks either mode — proving the common modes do not secretly demand `Clone`.
- **Clone-on-read on a non-`Clone` type is rejected.** Setup: the same non-`Clone` output type, with one edge demanding clone-on-read. Action: attempt to build it. Expected outcome: the demand is rejected — a compile-failure test (checked in per C28 conventions) if the mode is expressed statically, or a curated assembly error naming the type's missing `Clone` if the mode is chosen at registration — never a silent success and never a wall of raw trait errors.
- **Owned demand on a multiply-consumed value fails assembly, naming both.** Setup: a prototype value with two consumers where one demands ownership. Action: run assembly. Expected outcome: assembly fails with an error that names *both* consumers of the shared value, and does not fail earlier at compile time (the mode conflict is a whole-graph fact) — validating C1's "demanding ownership of a shared value fails at assembly (naming both consumers)."
- **Second consumer against a sole-owner value fails at the new registration.** Setup: a value whose already-registered sole consumer takes ownership. Action: register a second consumer for that same value. Expected outcome: assembly reports the conflict attributed to the *new* registration, confirming it is an assembly error at that point rather than a silent change of the first consumer's delivery mode.
- **Shared-access retry finds the value intact.** Setup: a prototype shared-access consumer over a filled slot, scripted to fail its first (retry-eligible) attempt after reading and to inspect the value again on its second attempt. Action: run the two attempts. Expected outcome: the value is still present and readable on the second attempt — validating C10's "a shared-access consumer that reads a value and then retries finds the value still available on its next attempt."
- **Owned edge into a retrying node is rejected without clone-on-read.** Setup: a prototype owned-input edge into a node whose policy declares a nonzero retry count, with no clone-on-read on that edge. Action: run assembly. Expected outcome: assembly fails, because a value moved into a failed attempt cannot be re-formed — validating C1's "an owned-input edge into a node with retries fails assembly unless the edge opts into clone-on-read."
- **Clone-on-read makes a retrying owned edge legal and gives each attempt a fresh value.** Setup: the same owned-into-retrying-node shape but with clone-on-read stated on the edge and a `Clone` output type. Action: run assembly, then two attempts. Expected outcome: assembly passes and each attempt receives a fresh clone (the first attempt's mutation or consumption does not affect the second) — validating C1's "each attempt receives a fresh clone."
- **Author-visible bounds are enforced and documented.** Setup: a prototype task capturing a non-`Send` value, and an output type that is `Send` but not `Sync`. Action: attempt to register each. Expected outcome: each fails to compile, and the ADR carries the worked narrative of the non-`Send`-capture first-hour error — validating C1's bound documentation acceptance criterion (`Send + 'static` tasks, `Send + Sync + 'static` outputs).
- **Slot residency accounting shape is expressible.** Setup: a prototype slot with a consumer count and the rule that release waits until every consumer is terminal *and* every consumer's work has returned. Action: mark a consumer terminal while its closure has not yet returned, then let it return. Expected outcome: the value is not releasable at the terminal mark and becomes releasable only after the closure returns — a *shape* check that the chosen representation can express C10's zombie-aware release (the full memory ledger is C10's implementation in T17, not built here).
- **Compile-time / assembly-time partition record check.** Setup: the ADR document. Action: read the Decision section's partition table. Expected outcome: type mismatch, missing `Clone` on a statically-clone-on-read edge, and missing `Send`/`Sync` are listed as compile errors; owned-on-multiply-consumed (naming both), owned-into-retrying-without-clone, and second-consumer-on-sole-owner are listed as assembly errors — an assertable presence check against the committed file.
- **Rejected-alternatives record check.** Setup: the ADR document. Action: read the rejected-alternatives section. Expected outcome: at least the always-`Arc`-wrap-everything and always-`Clone`-per-consumer alternatives are named with the reason each was rejected (needless `Send + Sync`/`Clone` burden on authors; memory multiplication; loss of the honest single-count residency of C10) — proving the decision was made against considered options.

## Definition of done
- [ ] The ADR records the chosen three-mode model — sole-consumer-owns, multi-consumer-shared-read, per-edge clone-on-read opt-in — as the locked design, traceable to C1's "Ownership of inputs" rules.
- [ ] A prototype demonstrates a sole consumer receiving its input by value and a multi-consumer input received as shared read access, both compiling (C1 acceptance).
- [ ] A prototype demonstrates that a non-`Clone` output type flows through both the owned and the shared modes, and that demanding clone-on-read on a non-`Clone` type is rejected (a compile error for a static clone-on-read edge, or a curated assembly error), never silently (C1, C3).
- [ ] The compile-time versus assembly-time partition is decided and recorded: type mismatch, missing `Clone` for a statically-declared clone-on-read edge, and missing `Send`/`Sync` bounds are compile errors; mode conflicts — owned demand on a multiply-consumed value (naming both consumers), a second consumer registered against a sole-owner value, and an owned edge into a retrying node without clone-on-read — are assembly errors (C1, C3).
- [ ] A prototype demonstrates that demanding ownership of a shared value fails at assembly and the error names both consumers (C1 acceptance).
- [ ] A prototype demonstrates that an owned-input edge into a node with a nonzero retry count fails assembly unless the edge opts into clone-on-read, and that a shared-access consumer with retries finds its input intact on every attempt (C1, C10 acceptance).
- [ ] A prototype demonstrates that a clone-on-read edge into a retrying node passes assembly and delivers a fresh value to each attempt (C1 acceptance).
- [ ] The author-visible trait bounds are fixed and documented on the model: task values `Send + 'static` with the work taking `&mut self`; output values `Send + Sync + 'static`; clone-on-read edges additionally require `Clone` — with the worked first-hour error (capturing a non-`Send` value in a task) written up in the ADR (C1 acceptance).
- [ ] The receive mode is confirmed to be part of the task's *signature* (visible in its declaration) and *not* part of C3 type matching, so type checking and mode checking stay on separate rails (C1, C3).
- [ ] The slot-delivery representation is shown to express C10's release rule — value released only when every consumer is terminal *and* every consumer's work has returned — at least as a shape check, so T17 inherits a representation that supports zombie-aware release and single-count residency (C10).
- [ ] The ADR names the rejected alternatives (at minimum always-`Arc` and always-`Clone`-per-consumer) with the reason each was rejected, and states the reopen condition: if any prototype above cannot be made to compile or to reject as specified, the spec's ownership decision reopens rather than being papered over.
- [ ] All throwaway prototype code is quarantined from shipping crates (or removed), and the ADR references what each prototype proved.
- [ ] The ADR is committed at exactly `/Users/athvin/github.com/athvin/dagr/docs/implementation/008-T0.2-output-ownership-adr-spike.md` with status, context, decision, consequences, and rejected-alternatives sections, leaving no ownership or receive-mode choice open to T5, T9, T11, T17, or T26.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- The full handle and typed-dependency-binding API (C2/C3) — that is the T5 spike and the T11 implementation; this ticket fixes only the ownership/receive-mode semantics they will bind against, not the binding surface itself.
- The task abstraction and error classification (C1) — implemented in T9; here we only prove the ownership bounds T9 must honor.
- The output slot's actual implementation — its consumer-count bookkeeping, zombie-aware release, and the full allocator-level memory ledger and residency accounting (C10) — that is T17; this ticket confirms only that the chosen representation *can* express those rules.
- The declared-cost vector, working-memory-versus-output-residency split, and admission-pool charging (C5, C12) — referenced only insofar as the slot representation must not preclude C10's single-count residency; not built here.
- Metrics recording of measured peak slot residency (C23/C26) — the T26 consumer of this decision; not implemented here.
- Any temptation to make the graph shape or a consumer's delivery mode change at runtime, or to introduce a shared mutable metadata store to track ownership across runs — dagr's graph shape never changes at runtime and it is never a metadata store; that boundary is named so the ownership model cannot smuggle it in.
