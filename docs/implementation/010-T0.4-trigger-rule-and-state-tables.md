# 010 · T0.4 — Trigger-rule and terminal-state reference tables

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C11, C15
> **Branch:** `adr/t0.4-trigger-rule-and-state-tables` · **Depends on:** — · **Blocks:** T3, T18, T29, T34, T50, T52

## Why / context
The whole run engine turns on two closed sets — the terminal states and the trigger rules — and every component that mentions a state or a rule (C11 readiness, C15 failure/propagation, C17 teardown, C22/C24 artifacts and diagrams, C26 CLI) must mean *exactly* the same thing by them. This ticket lifts arch.md's normative **Vocabulary — terminal states and trigger rules** section into the project's binding internal reference: the terminal-state enum, the four state classes that make trigger rules total over that taxonomy, and a per-rule fires / can-never-fire decision table with the resulting propagated state for every can-never-fire case. It is a decision record — no runtime, no engine — that later tickets implement against: T3 (error taxonomy) needs the failure-like states settled before it maps errors to them, T18 (readiness tracker) and T34 (failure policy) implement the table directly, and T29 (node policy), T50 (ordering edges), and T52 (teardown) each depend on the rule set and the consume-nothing restriction being fixed. Landing it wrong here means every downstream component disagrees about what a run's outcome even is, so the table must be reviewed against every component that references it.

## Objective
Produce a committed ADR that turns the Vocabulary section into normative internal definitions and a reviewed decision table, unambiguous enough that T18 and T34 can implement it without re-deriving anything. Concretely:

- **Terminal-state enum:** enumerate the nine terminal states — `succeeded`, `failed`, `timed-out`, `skipped`, `upstream-skipped`, `upstream-failed`, `cancelled`, `abandoned`, `satisfied-from-prior` — each with its one-line meaning, the payload it carries (`upstream-skipped` and `upstream-failed` carry the originating node identity; `satisfied-from-prior` carries the originating run identity), and the invariant that every node ends in exactly one of these, exactly once.
- **`not-requested` boundary:** record explicitly that `not-requested` is an **artifact marking** for single-node replay (C26), *not* a terminal state, and therefore is not a variant of the terminal-state enum — so no downstream ticket adds it as one.
- **State classes:** define the four classes as a total partition of the nine states — *success-like* (`succeeded`, `satisfied-from-prior`), *skip-like* (`skipped`, `upstream-skipped`), *failure-like* (`failed`, `timed-out`, `abandoned`, `upstream-failed`), *stop-like* (`cancelled`) — and state that trigger rules are defined over classes precisely so they are total over the taxonomy.
- **Trigger-rule set:** enumerate the closed set — `all-succeeded` (default), `all-terminal`, `any-failed` — and record the invariant that a rule is evaluated for a node only once *every* upstream is terminal (rules never fire early on a partial result).
- **Per-rule fires / can-never-fire decision table:** for each rule, record when it fires and, when it can never fire, the exact propagated terminal state as a function of the upstream state classes — including the corner cases below — so the table is total over every combination of upstream classes.
- **Consume-nothing restriction:** record that non-default rules (`all-terminal`, `any-failed`) are expressible only on nodes that consume nothing (C4), that data-dependent nodes always use `all-succeeded` (C3), and that this restriction is enforced at compile time via the builder's typestate, not at runtime — naming the seam so T29/T50 honor it.
- **Downstream hand-off:** state, per blocked ticket, exactly what T3, T18, T29, T34, T50, and T52 each consume from this table, and note that the terminal-state and event vocabulary feeds T19's event stream.

The decision-table corner cases that MUST be pinned:

- `all-succeeded` fires when every upstream is success-like; once any upstream is non-success it can never fire and the node is marked `upstream-skipped` when every non-success upstream is skip-like, `cancelled` when every non-success upstream is stop-like, and `upstream-failed` otherwise (i.e. any failure-like upstream, or a mix of classes).
- `all-terminal` fires whenever every upstream is terminal, regardless of class; it *never* propagates failure and has no can-never-fire case — this is what un-deadens cleanup nodes downstream of a failure.
- `any-failed` fires when every upstream is terminal and at least one is failure-like — a transitively `upstream-failed` upstream counts as failure-like — and is marked `skipped` (a contingency that never arose) when no upstream is failure-like.
- `satisfied-from-prior` upstreams count as success-like everywhere (so a resumed prior success satisfies a downstream `all-succeeded`).

## Decision

### 1. Terminal states — the nine-state enum

**Every node ends a run in exactly one of these nine states, exactly once.** This
is the canonical terminal-state enum; downstream code (T3's outcome taxonomy,
T18's assignment, the C22 run artifact, the C24 renderers) names these variants
and no others.

| # | State | Meaning | Payload |
|---|---|---|---|
| 1 | `succeeded` | The task returned a value; the slot was filled. | — |
| 2 | `failed` | Permanent failure, retries exhausted, or a caught panic. | — |
| 3 | `timed-out` | The final attempt exceeded its per-attempt timeout. | — |
| 4 | `skipped` | The task itself returned a deliberate skip (an *originated* skip). | — |
| 5 | `upstream-skipped` | Never ran because an upstream skip propagated to it. | **originating node identity** |
| 6 | `upstream-failed` | Never ran because its trigger rule can no longer be satisfied due to an upstream failure. | **originating node identity** |
| 7 | `cancelled` | Observed the cancellation signal and returned promptly, or was never admitted after cancellation began. | — |
| 8 | `abandoned` | Was asked to cancel and never returned within the grace period; its thread was left behind. | — |
| 9 | `satisfied-from-prior` | Not executed in this run; resume (C27) carried its prior success forward — its durable output was rehydrated, or its value was never demanded. | **originating run identity** |

**Payloads (normative).** `upstream-skipped` and `upstream-failed` each carry the
**identity of the originating node** — the node that decided to skip, or the node
whose failure made the rule unsatisfiable — so artifacts and diagrams (C22, C24)
can always distinguish the node that *decided* from the nodes affected *as a
consequence*. `satisfied-from-prior` carries the **originating run identity** —
the prior run whose success is being carried forward by resume (C27). The other
six states carry no propagation payload.

**Invariant — exactly one, exactly once.** Every node ends a run in exactly one
of the nine states, and that state is decided exactly once. The originated and
propagated forms are **distinct** states (`skipped` vs `upstream-skipped`,
`failed`/`timed-out`/`abandoned` vs `upstream-failed`), never collapsed. Per
T0.3 (009), a blocking `timed-out` attempt whose thread never returns **stays**
`timed-out` — the leftover thread is a zombie *event* (C19), not a second
terminal state; and `abandoned` arises only on the cancellation path (C16), never
as a second state after a timeout.

### 2. The `not-requested` boundary — not a terminal state

`not-requested` is an **artifact marking** for single-node replay (C26): in a
single-node replay artifact, nodes *outside* the replay request are marked
`not-requested`. It is **not** a terminal state, is **not** a variant of the
nine-state enum above, and no downstream ticket may add it as one. It is recorded
here precisely so the boundary is fixed in one place: the enum has exactly nine
variants; `not-requested` lives only in the C26 artifact layer.

### 3. State classes — the four-class total partition

Trigger rules are defined over **classes**, so they are **total over the
taxonomy**: every one of the nine states belongs to **exactly one** class, no
state is unclassified, and no state is in two classes. This is the partition that
makes the decision table in §5 total.

| Class | Members | Count |
|---|---|---|
| *success-like* | `succeeded`, `satisfied-from-prior` | 2 |
| *skip-like* | `skipped`, `upstream-skipped` | 2 |
| *failure-like* | `failed`, `timed-out`, `abandoned`, `upstream-failed` | 4 |
| *stop-like* | `cancelled` | 1 |

2 + 2 + 4 + 1 = 9 — a total partition of the nine-state enum.

**`satisfied-from-prior` is success-like.** A resumed prior success is a success:
it counts as success-like *everywhere*, so a downstream `all-succeeded` node whose
only "non-run" upstream is `satisfied-from-prior` sees a success-like upstream and
its rule **fires** (the node becomes ready). This is exactly C11's "with
`satisfied-from-prior` upstreams covered explicitly (they count as success-like)"
criterion, and it is what makes the cleanup-after-publish shape resume correctly
(C27): a satisfied publish presents as a success-like upstream to cleanup.

### 4. Trigger rules — the closed three-rule set, all-terminal-gated

The set is **closed** — three rules, no more, extensible only by spec amendment
(never a runtime knob). A rule is **evaluated for a node only once *every*
upstream of that node is terminal**; a rule **never fires early on a partial
result**. Until the last upstream reaches a terminal state, the node is neither
ready nor propagated — it simply waits.

| Rule | Default? | Fires when | Can-never-fire? |
|---|---|---|---|
| `all-succeeded` | **yes** (default) | every upstream is success-like | yes — see §5 |
| `all-terminal` | no | every upstream is terminal, regardless of class | **no** — always fires; never propagates failure |
| `any-failed` | no | every upstream is terminal and at least one is failure-like | it "can never fire" only in the degenerate sense that no upstream is failure-like → marks `skipped` (see §5) |

### 5. Per-rule decision table — total over upstream state classes

For each rule, and for **every** combination of upstream state classes (the input
is the *multiset of classes* over the node's upstreams, evaluated only once all
upstreams are terminal), the table records either **fires** (the node becomes
ready and executes) or a **single named propagated terminal state** (the node is
immediately assigned that state, without executing). The table is **total**: no
class combination is left undefined for any rule.

#### 5a. `all-succeeded` (default)

Let the upstreams' classes be partitioned into *success-like* and *non-success*
(anything not success-like). Then:

| Upstream class picture | Outcome |
|---|---|
| every upstream is success-like | **fires** → node becomes ready |
| some upstream non-success, and **every** non-success upstream is *skip-like* | **`upstream-skipped`** (carries originating node) |
| some upstream non-success, and **every** non-success upstream is *stop-like* | **`cancelled`** |
| some upstream non-success, **otherwise** — i.e. any *failure-like* upstream, **or** a cross-class mix of non-success classes (skip-like + stop-like, etc.) | **`upstream-failed`** (carries originating node) |

The three non-fires rows are mutually exclusive and jointly exhaustive over "at
least one non-success upstream": *all-skip-like* → `upstream-skipped`,
*all-stop-like* → `cancelled`, *everything else* (any failure-like, or any mix of
distinct non-success classes) → `upstream-failed`. Together with the fires row
(all-success-like) they cover every possible class multiset — the rule is total.
Because `satisfied-from-prior` is success-like, a resumed prior success is in the
fires branch.

#### 5b. `all-terminal`

| Upstream class picture | Outcome |
|---|---|
| every upstream is terminal (any classes at all) | **fires** → node becomes ready |

`all-terminal` has **no can-never-fire case** and **never propagates failure**:
once every upstream is terminal — which is the evaluation precondition for every
rule — it fires regardless of class. This is the rule that **un-deadens a cleanup
node downstream of a failure**: an `all-terminal` cleanup or notify node still
runs after its upstreams fail/skip/cancel, which is the entire reason non-default
rules exist. Under stop-on-first-failure (C15), such a consume-nothing node whose
`all-terminal` rule fires on the resulting terminal picture is precisely the work
a failure is supposed to trigger, so stop mode still admits it.

#### 5c. `any-failed`

| Upstream class picture | Outcome |
|---|---|
| every upstream terminal, and **at least one** is failure-like | **fires** → node becomes ready |
| every upstream terminal, and **none** is failure-like | **`skipped`** (a contingency that never arose) |

A **transitively `upstream-failed` upstream counts as failure-like** — because
the guarded work did not complete either way, an `any-failed` contingency should
still fire on it. When no upstream is failure-like, the guarded contingency never
arose, so the node is marked **`skipped`**: it uses the `skipped` state per the
Vocabulary, distinct *in intent* from a task-originated skip (this is a
deliberate-contingency-that-never-arose, not a task deciding "nothing to do") but
using the same state name. These two rows are exhaustive over "every upstream
terminal," so `any-failed` is total.

**Totality summary.** For `all-succeeded` the input space is every class multiset
with ≥1 upstream; the four §5a rows partition it. For `all-terminal` there is one
row and it fires on all of them. For `any-failed` the two §5c rows partition
"every upstream terminal" on the failure-like predicate. No combination is left
undefined for any rule — the decision table is total over the taxonomy.

### 6. Consume-nothing restriction — compile-time typestate

Non-default rules (`all-terminal`, `any-failed`) are **expressible only on nodes
that consume nothing** (C4): a rule firing without an upstream *value* can never
leave a hole where an input was expected. **Data-dependent nodes always use
`all-succeeded`** (C3). This restriction is **enforced at compile time via the
builder's typestate** — a **compile error**, not a runtime check. The seam is the
builder typestate that gates which trigger rule a node may declare based on
whether it has data dependencies (C3) versus ordering-only edges (C4); **T29**
(node policy struct) and **T50** (ordering-edge attach rules) must **preserve**
this seam — neither may make a non-default rule attachable to a data-dependent
node. An ordering edge (C4) may attach to *any* node, including one with data
dependencies; it is only the *non-default trigger rule* that is restricted to
consume-nothing nodes.

### 7. Downstream hand-off — what each blocked ticket consumes

This ADR is the single source the blocked tickets reference; none re-derives the
table:

- **T3** (error taxonomy) consumes the **failure-like state set** — `failed`,
  `timed-out`, `abandoned`, `upstream-failed` — to map caught errors and panics
  onto. The task-facing error enum stays three-valued (retryable / permanent /
  skip); the runner's classification is the superset that produces these terminal
  states.
- **T18** (readiness tracker, C11) consumes the **fires / can-never-fire
  evaluation** (§5) and the **immediate propagated-state assignment**: a node
  whose rule fires becomes ready; a node whose rule can never fire is immediately
  assigned `upstream-failed`, `upstream-skipped`, or (for `any-failed`) `skipped`
  without executing. M1 ships `all-succeeded` only, but against this final rule
  interface.
- **T29** (node policy, C5) consumes the **closed trigger-rule set** (§4) for the
  policy struct's trigger-rule field, plus the §6 restriction the struct's
  typestate must honor.
- **T34** (failure policy and trigger-rule runtime, C15) consumes the **runtime
  propagation semantics**: propagation only when a rule is unsatisfiable;
  propagated-state selection by upstream class (§5a); `all-terminal` firing after
  an upstream failure; `any-failed` firing on a failure-like upstream and marking
  `skipped` when the contingency never arose; and stop-mode contingency admission
  (consume-nothing non-default-rule nodes admitted on the final picture). Pending
  unrelated default-rule nodes under stop mode end `cancelled`.
- **T50** (ordering edges, C4 / T0.9) consumes the **consume-nothing restriction**
  (§6) and the ordering-edge attach rules it must enforce at compile time.
- **T52** (teardown nodes, C17) consumes the **"runs after any terminal state"**
  relationship — teardown fires when all covered nodes are terminal, regardless
  of their class, which composes with the `all-terminal` semantics of §5b.
- **T19** (event stream, C19) consumes the **terminal-state and state-class
  vocabulary** the events carry — this ADR supplies the state names those events
  reference, not the event record shapes (that is T19/T42).

### 8. C11 and C15 hand-off, stated explicitly

**C11 hand-off.** A node whose trigger rule **fires** becomes **ready** (eligible
to be admitted and executed). A node whose rule **can never fire** is
**immediately assigned its propagated terminal state without executing** —
`upstream-failed`, `upstream-skipped`, or `skipped` (the last only for an
`any-failed` contingency that never arose). Evaluation happens only once all
upstreams are terminal; there is no wave batching — a node becomes ready the
instant its own dependencies allow.

**C15 hand-off.** Propagation is **governed by trigger rules**: a node is marked
`upstream-failed` **only when its trigger rule can no longer be satisfied** — so
an `all-terminal` cleanup node downstream of a failure still runs. Deliberate
skips propagate as **`upstream-skipped`**, carrying the **originating node**
identity, and a run whose only non-success outcomes were skips reports **overall
success**.

## Consequences

- **A single closed vocabulary.** Every downstream component (T3, T18, T29, T34,
  T50, T52, T19) names the same nine states, four classes, and three rules. A new
  state or rule is a **spec amendment**, never a runtime knob — the state set and
  rule set are closed by construction, and the graph shape never changes at
  runtime.
- **The decision table is total**, so T18 and T34 can implement §5 without
  re-deriving any corner case: every upstream-class combination maps to exactly
  one of "fires" or a single named propagated state.
- **`satisfied-from-prior` = success-like everywhere** makes resume compose:
  resumed prior successes satisfy downstream `all-succeeded` rules, so
  cleanup-after-publish resumes correctly.
- **Non-default rules stay honest.** Because they are compile-time restricted to
  consume-nothing nodes, a firing rule can never leave a hole where a value was
  expected — the typestate seam T29/T50 must preserve is named here so they do
  not weaken it.
- **Consistency with T0.2/T0.3 is load-bearing.** The "terminal" predicate this
  ADR's readiness gate reads is the same one T0.2's slot-release logic reads
  (a zombie consumer is *decided*), and the failure-like class agrees with T0.3
  that `timed-out` and `abandoned` are distinct, both failure-like, and never a
  second terminal state.
- **No runtime, no engine.** This ticket produces the reference tables only; the
  readiness tracker (T18), failure policy (T34), error taxonomy (T3), node policy
  (T29), ordering-edge mechanics (T50), and teardown (T52) implement *against*
  them and own their own tests. Shipping no covering test, this ADR makes **no
  change to `docs/coverage-matrix.md`** — the C11/C15 machine criteria remain
  `unmapped`, owed by their implementing tickets (T18/T25 for C11, T34/T37 for
  C15), which flip them as they land their tests.

## Rejected alternatives

- **A tenth terminal state (e.g. promoting `not-requested` into the enum).**
  Rejected: `not-requested` is a C26 single-node-replay *artifact marking* for
  nodes outside the request, not an outcome a node reaches during a run. Adding
  it would break the "exactly one of nine, exactly once" invariant. It is
  recorded in §2 precisely so no downstream ticket re-adds it.
- **Collapsing originated and propagated forms** (one `skipped` covering both
  task-skip and upstream-skip; one `failed` covering both direct and propagated
  failure). Rejected: artifacts and diagrams (C22, C24) must always distinguish
  the node that *decided* from the nodes affected *as a consequence*, and the
  propagated forms carry the originating-node payload that the originated forms
  do not.
- **Rules over raw states instead of over classes.** Rejected: defining rules
  over the nine states directly would require enumerating dozens of cases and
  would not be visibly total. Defining them over the four-class partition makes
  each rule a small, obviously-total table (§5) — the classes exist *so that* the
  rules are total over the taxonomy.
- **A runtime-extensible / pluggable rule or state set.** Rejected as a permanent
  non-goal: dagr is not a DSL or a scheduler, the graph shape never changes at
  runtime, and a new rule or state is a spec amendment. The sets are closed by
  construction.
- **Making `all-terminal` propagate failure, or giving it a can-never-fire
  case.** Rejected: that would re-deaden the cleanup-after-failure node that is
  the entire reason non-default rules exist. `all-terminal` fires whenever every
  upstream is terminal and never propagates failure (§5b).
- **A separate `contingency-not-triggered` state for the `any-failed`
  no-failure case.** Rejected: the Vocabulary reuses `skipped` for it. It is a
  contingency that never arose, semantically a skip; a distinct state would add a
  tenth variant for no downstream benefit. The *intent* distinction is documented
  (§5c) without a new state.
- **Allowing non-default rules on data-dependent nodes with a runtime guard.**
  Rejected: the restriction is a **compile-time typestate** error, not a runtime
  check — a data-dependent node with a non-default rule would leave a hole where
  a value was expected, and that must be unrepresentable, not merely diagnosed at
  run time (C3/C4).
