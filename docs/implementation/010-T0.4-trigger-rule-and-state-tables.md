# 010 · T0.4 — Trigger-rule and terminal-state reference tables

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C11, C15
> **Branch:** `adr/t0.4-trigger-rule-and-state-tables` · **Depends on:** — · **Blocks:** T3, T18, T29, T34, T50, T52

## Why / context
The whole run engine turns on two closed sets — the terminal states and the trigger rules — and every component that mentions a state or a rule (C11 readiness, C15 failure/propagation, C17 teardown, C22/C24 artifacts and diagrams, C26 CLI) must mean *exactly* the same thing by them. This ticket lifts arch.md's normative **Vocabulary — terminal states and trigger rules** section into the project's binding internal reference: the terminal-state enum, the four state classes that make trigger rules total over that taxonomy, and a per-rule fires / can-never-fire decision table with the resulting propagated state for every can-never-fire case. It is a decision record — no runtime, no engine — that later tickets implement against: T3 (error taxonomy) needs the failure-like states settled before it maps errors to them, T18 (readiness tracker) and T34 (failure policy) implement the table directly, and T29 (node policy), T50 (ordering edges), and T52 (teardown) each depend on the rule set and the consume-nothing restriction being fixed. Landing it wrong here means every downstream component disagrees about what a run's outcome even is, so the table must be reviewed against every component that references it.

## Objective
Produce a committed ADR that turns the Vocabulary section into normative internal definitions and a reviewed decision table, unambiguous enough that T18 and T34 can implement it without re-deriving anything. Concretely:

- **Terminal-state enum:** enumerate the nine terminal states — `succeeded`, `failed`, `timed-out`, `skipped`, `upstream-skipped`, `upstream-failed`, `cancelled`, `abandoned`, `satisfied-from-prior` — each with its one-line meaning, the payload it carries (`upstream-skipped` and `upstream-failed` carry the originating node identity; `satisfied-from-prior` carries the originating run identity), and the invariant that every node ends in exactly one of these, exactly once.
- **`not-requested` boundary:** record explicitly that `not-requested` is an **artifact marking** for single-node replay (C26), *not* a terminal state, and therefore is not a variant of the terminal-state enum — so no downstream ticket adds it as one.
- **State classes:** define the four classes as a total partition of the nine states — *success-like* (`succeeded`, `satisfied-from-prior`), *skip-like* (`skipped`, `upstream-skipped`), *failure-like* (`failed`, `timed-out`, `abandoned`, `upstream-failed`), *stop-like* (`cancelled`) — and state that trigger rules are defined over classes precisely so they are total over the taxonomy.
- **Trigger-rule set:** enumerate the closed set — `all-succeeded` (default), `all-terminal`, `any-failed` — and record the invariant that a rule is evaluated for a node only once *every* upstream is terminal (rules never fire early on a partial result).
- **Per-rule fires / can-never-fire decision table:** for each rule, record when it fires and, when it can never fire, the exact propagated terminal state as a function of the upstream state classes — including the corner cases below — so the table is total over every combination of upstream classes.
- **Consume-nothing restriction:** record that non-default rules (`all-terminal`, `any-failed`) are expressible only on nodes that consume nothing (C4), that data-dependent nodes always use `all-succeeded` (C3), and that this restriction is enforced at compile time via the builder's typestate, not at runtime — naming the seam so T29/T50 honor it.
- **Downstream hand-off:** state, per blocked ticket, exactly what T3, T18, T29, T34, T50, and T52 each consume from this table, and note that the terminal-state and event vocabulary feeds T19's event stream.

The decision-table corner cases that MUST be pinned:

- `all-succeeded` fires when every upstream is success-like; once any upstream is non-success it can never fire and the node is marked `upstream-skipped` when every non-success upstream is skip-like, `cancelled` when every non-success upstream is stop-like, and `upstream-failed` otherwise (i.e. any failure-like upstream, or a mix of classes).
- `all-terminal` fires whenever every upstream is terminal, regardless of class; it *never* propagates failure and has no can-never-fire case — this is what un-deadens cleanup nodes downstream of a failure.
- `any-failed` fires when every upstream is terminal and at least one is failure-like — a transitively `upstream-failed` upstream counts as failure-like — and is marked `skipped` (a contingency that never arose) when no upstream is failure-like.
- `satisfied-from-prior` upstreams count as success-like everywhere (so a resumed prior success satisfies a downstream `all-succeeded`).

## Test plan (write these first — TDD)
This is a decision ticket, so the "tests" are validation checks against the recorded ADR and its decision table; each is concrete and independently checkable, and each corresponds to an acceptance criterion that T18 or T34 will later turn into executable tests against the same table.

- **State enum is complete and singular.** Setup: the ADR's terminal-state list and arch.md's Vocabulary terminal-states table. Action: cross-reference the two. Expected: exactly the nine states appear, each with its meaning and its carried payload, and the invariant "every node ends in exactly one, exactly once" is stated. No tenth state and no missing state.

- **`not-requested` is excluded from the enum.** Setup: the ADR. Action: look up `not-requested`. Expected: it is recorded as a C26 artifact marking for nodes outside a single-node replay request, explicitly *not* a terminal state and *not* a variant of the terminal-state enum.

- **State classes partition the taxonomy totally.** Setup: the ADR's four state classes and the nine states. Action: assign every state to a class and check coverage. Expected: every one of the nine states belongs to exactly one class; success-like = {`succeeded`, `satisfied-from-prior`}, skip-like = {`skipped`, `upstream-skipped`}, failure-like = {`failed`, `timed-out`, `abandoned`, `upstream-failed`}, stop-like = {`cancelled`}; no state is unclassified and none is in two classes.

- **`satisfied-from-prior` is success-like.** Setup: the class table. Action: look up `satisfied-from-prior`. Expected: it is classed success-like, with a note that a resumed prior success therefore satisfies a downstream `all-succeeded` — matching C11's "covered explicitly" criterion.

- **Trigger-rule set is closed and evaluation is all-terminal-gated.** Setup: the ADR's rule list. Action: read it. Expected: exactly three rules — `all-succeeded` (default), `all-terminal`, `any-failed` — and the stated invariant that a rule is evaluated only once every upstream is terminal (no early fire on partial results).

- **`all-succeeded` fires case.** Setup: the decision table, an `all-succeeded` node with all upstreams success-like (including one `satisfied-from-prior`). Action: read the outcome. Expected: the rule fires and the node becomes ready.

- **`all-succeeded` can-never-fire → `upstream-skipped`.** Setup: an `all-succeeded` node whose non-success upstreams are all skip-like. Action: read the outcome. Expected: marked `upstream-skipped`, carrying the originating node identity, without executing.

- **`all-succeeded` can-never-fire → `cancelled`.** Setup: an `all-succeeded` node whose non-success upstreams are all stop-like. Action: read the outcome. Expected: marked `cancelled` without executing.

- **`all-succeeded` can-never-fire → `upstream-failed`.** Setup: an `all-succeeded` node with at least one failure-like upstream, or a mix of skip-like and stop-like and failure-like non-success upstreams. Action: read the outcome. Expected: marked `upstream-failed` in every case that is neither all-skip-like nor all-stop-like, carrying the originating node identity.

- **`all-terminal` always fires, never propagates failure.** Setup: an `all-terminal` node with upstreams spanning all four classes (a success, a skip, a failure, a cancellation). Action: read the outcome. Expected: the rule fires and the node executes; the table records no can-never-fire case for `all-terminal`. This is the cleanup-after-failure case that must run.

- **`any-failed` fires on a failure-like upstream.** Setup: an `any-failed` node with all upstreams terminal and at least one failure-like (including the transitive case where the failure-like upstream is itself `upstream-failed`). Action: read the outcome. Expected: the rule fires and the node executes.

- **`any-failed` contingency-never-arose → `skipped`.** Setup: an `any-failed` node with every upstream terminal and none failure-like. Action: read the outcome. Expected: marked `skipped` (a deliberate-contingency-that-never-arose), distinct from a task-originated skip in intent but using the `skipped` state per the Vocabulary.

- **Decision table is total over upstream class combinations.** Setup: the table and the set of possible upstream-class multisets. Action: for each rule, enumerate the class combinations and check that every combination maps to either "fires" or a single named propagated state. Expected: no combination is left undefined for any rule; the table is total.

- **Non-default rules are compile-time restricted to consume-nothing nodes.** Setup: the ADR's restriction clause. Action: read it. Expected: it states that `all-terminal` and `any-failed` are expressible only on nodes with no data dependencies, that data-dependent nodes always use `all-succeeded`, and that the restriction is enforced by the builder typestate at compile time (a compile error, not a runtime check) — matching C3/C4, and naming the seam T29/T50 must preserve.

- **Downstream hand-off is explicit.** Setup: the ADR's hand-off section. Action: read it. Expected: it states per blocked ticket what each consumes — T3 the failure-like state set to map errors onto, T18 the fires/can-never-fire evaluation and propagated-state assignment, T29 the closed trigger-rule set for the policy struct, T34 the runtime propagation and stop-mode contingency-admission semantics, T50 the ordering-edge attach rules and consume-nothing restriction, T52 the teardown "runs after any terminal state" relationship — so no downstream ticket re-derives the table.

- **ADR is committed in the accepted state.** Setup: the repository. Action: locate the ADR file and read its status. Expected: the ADR exists at its recorded in-repo path, is in an accepted (not draft/proposed) status, and is the single source the blocked tickets reference.

## Definition of done
- [ ] The ADR enumerates the nine terminal states with meanings, matching arch.md's Vocabulary table exactly, and records the carried payloads: `upstream-skipped`/`upstream-failed` carry the originating node identity, `satisfied-from-prior` carries the originating run identity (Vocabulary).
- [ ] The ADR records the invariant that every node ends a run in exactly one terminal state from the normative taxonomy, exactly once (Vocabulary, C11, C15).
- [ ] The ADR records that `not-requested` is a C26 single-node-replay artifact marking, not a terminal state, and is excluded from the terminal-state enum (Vocabulary).
- [ ] The ADR defines the four state classes as a total partition of the nine states — success-like {`succeeded`, `satisfied-from-prior`}, skip-like {`skipped`, `upstream-skipped`}, failure-like {`failed`, `timed-out`, `abandoned`, `upstream-failed`}, stop-like {`cancelled`} — with every state in exactly one class (Vocabulary).
- [ ] The ADR enumerates the closed trigger-rule set — `all-succeeded` (default), `all-terminal`, `any-failed` — and records that a rule is evaluated only once every upstream is terminal, never on a partial result (Vocabulary, C11).
- [ ] The ADR contains a per-rule decision table that, for every combination of upstream state classes, records either "fires" or a single named propagated terminal state, and is total over the taxonomy (Vocabulary, C11).
- [ ] The table records `all-succeeded`: fires when every upstream is success-like; otherwise `upstream-skipped` (all non-success upstreams skip-like), `cancelled` (all non-success upstreams stop-like), or `upstream-failed` (all other cases, i.e. any failure-like upstream or a cross-class mix) (Vocabulary, C11, C15).
- [ ] The table records `all-terminal`: fires whenever every upstream is terminal regardless of class, with no can-never-fire case, and never propagates failure — the criterion that keeps a cleanup node downstream of a failure running (Vocabulary, C11, C15).
- [ ] The table records `any-failed`: fires when every upstream is terminal and at least one is failure-like (a transitively `upstream-failed` upstream counts); marked `skipped` when the contingency never arose (Vocabulary, C11, C15).
- [ ] The ADR records that `satisfied-from-prior` upstreams count as success-like everywhere, so C11's "satisfied-from-prior upstreams covered explicitly" case resolves to the fires branch of `all-succeeded` (Vocabulary, C11).
- [ ] The ADR records that non-default rules are expressible only on consume-nothing nodes, that data-dependent nodes always use `all-succeeded`, and that this is a compile-time typestate restriction — naming the seam T29/T50 must preserve (Vocabulary, C3, C4).
- [ ] The ADR records the C11 hand-off: a node whose rule fires becomes ready; a node whose rule can never fire is immediately assigned its propagated terminal state without executing (`upstream-failed`, `upstream-skipped`, or `skipped`) (C11).
- [ ] The ADR records the C15 hand-off: propagation is governed by trigger rules — a node is marked `upstream-failed` only when its rule can no longer be satisfied — and deliberate skips propagate as `upstream-skipped` carrying the originating node, with a skip-only run reporting overall success (C15).
- [ ] The ADR names the per-ticket downstream hand-off (T3, T18, T29, T34, T50, T52) and notes the terminal-state/event vocabulary feed into T19, so no blocked ticket re-derives the table (task dependency map).
- [ ] The ADR is committed at its recorded in-repo path in an accepted status and is the single source referenced by the blocked tickets.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- Implementing the readiness tracker itself — the remaining-dependency countdown, the decrement-on-terminal mechanics, the no-wave-batching diamond behavior, and the deadlock-freedom property test are T18; this ticket only fixes the table T18 evaluates against.
- Implementing failure policy at runtime — stop-on-first-failure vs continue-independent, in-flight drain, stop-mode contingency admission, and `cancelled`-on-pending-unrelated behavior are T34; this ticket only fixes the propagation semantics T34 applies.
- Designing the error taxonomy that maps caught errors and panics onto the failure-like states — that is T3, which depends on this ticket's failure-like set being settled first.
- Designing the node policy struct, execution-class override validation, cost vector, or retention/durability flags — that is T29; this ticket contributes only the closed trigger-rule set it embeds.
- Designing ordering-edge mechanics and the compile-time attach rules — that is T50 (and T0.9); this ticket only states the consume-nothing restriction on non-default rules that T50 must enforce.
- Designing teardown nodes and their fresh-signal / cost-zero / covered-state-exposure behavior — that is T52; this ticket only states the "runs after any terminal state" relationship teardown builds on.
- Defining the event-stream records, schema, or the folding function — that is T19/T42; this ticket supplies the state vocabulary those events carry, not the record shapes.
- Any temptation to make the state set or rule set runtime-extensible, pluggable, or configurable — the set is closed by construction and the graph shape never changes at runtime; dagr is not a DSL or a scheduler, and a new rule or state is a spec amendment, never a runtime knob.
