# 013 · T0.7 — ADR: stable-name trait and fingerprint composition

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C20, C21
> **Branch:** `adr/t0.7-stable-name-and-fingerprint-adr` · **Depends on:** — · **Blocks:** T13, T40, T41, T58, T61

## Why / context
Node identity and pipeline-shape identity are baked in at the very first builder ticket (T13), long before the graph artifact (C20, T40) and the two fingerprints (C21, T41) actually ship in M3. If T13 wires nodes against a placeholder notion of "name" and the real stable-name trait arrives later, every node identity, every artifact, and every fingerprint has to be re-derived — so the trait, the field lists that each hash covers, the canonicalization rules, and what is deliberately excluded must all be decided now, in M0. This ADR is governed by arch.md **C20 · Graph artifact** and **C21 · Graph fingerprint**, and it sits downstream of the algorithm-versioning and "cannot compare" semantics already fixed in T0.10 — this ticket designs the composition and canonicalization that those version identifiers wrap, it does not re-decide the versioning story. It locks: the author-declared stable-name trait/derive for task and payload types; the exact split of fields between the structural fingerprint and the policy hash; the canonical ordering and encoding both hashes feed on; and the rule that group labels and environmental values are in neither hash. Get this wrong and T13, T40, T41, T58, and T61 all disagree about what a node *is* and whether two builds are the same pipeline.

## Objective
Produce a committed ADR that fixes the stable-name contract and the composition of both fingerprints precisely enough that T13 can bake node identity in, and T40/T41 can emit the artifact and hashes without re-deriving anything. Concretely, the ADR must decide and record:

- **Stable-name trait/derive.** A trait carrying an author-declared stable name as a constant, implemented by both task types and payload (input/output) types, with a derive that makes the common case one line. Record that the name is the *declared* name the author writes — never `std::any::type_name`, which is explicitly unstable across compiler versions and may appear only as an informational debug field, never as identity and never in either hash. State the well-formedness rules for a stable name (non-empty, a fixed character set / shape) and that uniqueness of declared task and payload names is a *whole-pipeline* assembly check, not a compile-time one, with the duplicate error naming both declarations.
- **What C20 records per node and per edge.** The field list the graph artifact carries for every node — name, group, stable task name, stable input and output type names, execution class, complete effective policy (including defaulted values written out), declared resource requirements, dependency lists — and for every edge whether it carries data or only ordering, plus for data edges the stable name of the carried type. This is the raw material both hashes are computed from; the ADR fixes which of these fields feed which hash.
- **Structural-fingerprint field list.** Exactly the shape-determining inputs: the node set (by stable name), the edge set (with carried-type stable names and the data-vs-ordering edge kind), and the per-node trigger rule. These are "the things that determine whether a prior run's outputs mean anything to this binary" — the resume gate (C27).
- **Policy-hash field list.** The remaining effective-policy values — retry count/backoff, per-attempt timeout, declared resource cost, execution class, retention flag, durability flag — and the rule that a node with no stated policy hashes *identically* to one with every default written out (C5). A policy divergence is a proceed-with-diff at resume, never a refusal.
- **Exclusions.** Group labels are in *neither* hash (C6) and are not part of node identity; a group rename shows up only in the structure diff (C28), never breaks resume. Both hashes exclude timestamps, hostnames, compiler/tool versions, generation time, git commit, lockfile hash, and everything else environmental, so two builds of unchanged source on different toolchains hash identically.
- **Canonicalization.** The single canonical ordering and byte-encoding both hashes and the artifact's byte-identity comparison rely on: how the node set and edge set are deterministically ordered (independent of registration order), how each field is serialized into the hash input, and the hash function itself. This is what makes "assemble twice → byte-identical artifact (generation-time field aside)" and "two toolchains → same fingerprint" achievable, and it is the input over which the T0.10 algorithm-version identifier is defined.
- **Algorithm-version binding.** Record that each hash carries the versioned algorithm identifier fixed in T0.10, that changing this canonicalization or field split is an algorithm-version bump, and that a version mismatch reads as "cannot compare" (distinct from "topology differs") — pointing at T0.10 as the owner of that decision rather than re-deciding it.
- **Point-of-use documentation.** Record the C21 internal-logic limitation — changing a task's internal logic without changing its interface does *not* change the fingerprint — and require it be documented where developers meet it: on the resume verb (C27) and the structure-assertion API (C28), with the hand-maintained task version marker named as the honest manual answer. Not buried in the ADR.
- **Downstream hand-off.** State, per blocked ticket, exactly what each consumes: T13 the stable-name trait/derive and the whole-pipeline uniqueness rule for node identity; T40 the per-node/per-edge C20 field list and the byte-identity/canonicalization rules; T41 the two field lists, canonical ordering, group exclusion, and the defaulted-values-hash-identical and cross-toolchain properties; T58 the fingerprint hashes it records into run artifacts; T61 the structural fingerprint the structure-snapshot assertion diffs against.

## Test plan (write these first — TDD)
This is a decision ticket, so the "tests" are validation checks against the recorded ADR; each is concrete and independently checkable, and each corresponds to an acceptance criterion that T40, T41, or T61 will later turn into an executable test against these same rules.

- **Stable name is author-declared, never `type_name`.** Setup: the ADR's stable-name section. Action: read how a name is obtained. Expected: it is an author-declared constant via the trait/derive; `std::any::type_name` is recorded as unstable-across-compilers and permitted only as an informational debug field, never as identity and never in either hash.

- **Trait covers both task and payload types, derive is one line.** Setup: the ADR. Action: look up which types implement the trait and how. Expected: task types and payload (input/output) types both carry a stable name, the derive makes the common case one line, and the recorded name matches the *declared* name of the compiler-enforced type.

- **Stable-name well-formedness and uniqueness scope.** Setup: the ADR. Action: read the name rules. Expected: a stated well-formedness rule (non-empty, fixed shape/character set) and the rule that duplicate declared task/payload names are a whole-pipeline *assembly* failure naming both declarations — not a compile error, not scoped to a group.

- **C20 per-node field list is complete.** Setup: the ADR's artifact field list and arch.md C20. Action: cross-reference. Expected: every node records name, group, stable task name, stable input/output type names, execution class, complete effective policy *including defaulted values*, declared resource requirements, and dependency lists — nothing the running pipeline would use is missing.

- **C20 per-edge field list captures kind and carried type.** Setup: the ADR. Action: read the edge fields. Expected: every edge records whether it carries data or only ordering, and every data edge records the stable name of the type it carries.

- **Structural fingerprint covers exactly shape inputs.** Setup: the ADR's structural-fingerprint field list. Action: enumerate its inputs. Expected: node set (stable names), edge set (carried-type names + edge kind), and trigger rules — and nothing else; no policy value, no group, no environmental value.

- **Policy hash covers exactly the residual policy.** Setup: the ADR's policy-hash field list. Action: enumerate its inputs. Expected: retries/backoff, timeout, declared cost, execution class, retention, durability — the effective policy values that are *not* in the structural fingerprint, and no shape input.

- **Defaulted policy hashes identically to written-out defaults.** Setup: two node descriptions, one with no stated policy and one with every default written out explicitly. Action: apply the policy-hash rule to both. Expected: identical policy-hash inputs and therefore an identical hash (C5) — the ADR states this as a required property with a matching T41 test named.

- **Group labels are in neither hash and not in identity.** Setup: the ADR. Action: look up where group labels feed. Expected: they feed artifact organization and diagram clustering only; a group rename changes neither the structural fingerprint nor the policy hash nor node identity, and surfaces solely in the C28 structure diff.

- **Both hashes exclude everything environmental.** Setup: the ADR's exclusion list. Action: read it. Expected: timestamps, hostnames, compiler/tool versions, generation time, git commit, and lockfile hash are all excluded from both hashes, so unchanged source on a different machine/toolchain hashes identically.

- **Canonicalization is deterministic and registration-order-independent.** Setup: the ADR's canonicalization section, and two mental orderings of the same node/edge set. Action: apply the canonical ordering to both. Expected: identical ordered inputs regardless of registration order, a single fixed byte-encoding per field, and a single named hash function — enough to make "assemble twice → byte-identical artifact (generation-time aside)" and "two toolchains → same fingerprint" achievable.

- **Structural change moves the structural fingerprint; policy-only change moves only the policy hash.** Setup: the ADR's change matrix. Action: walk each change class. Expected: a node add/remove/rename, a rewire, a trigger-rule change, or a carried-type change moves the structural fingerprint; a retries/timeout/cost/class/retention/durability change moves only the policy hash; a group rename moves neither — mapping directly to the T41 change/no-change matrix.

- **Algorithm-version binding points at T0.10.** Setup: the ADR. Action: read the versioning note. Expected: each hash carries the T0.10 versioned algorithm identifier, changing this canonicalization/field split is an algorithm-version bump, a mismatch reads as "cannot compare" (distinct from "topology differs"), and the ADR defers the versioning *policy* to T0.10 rather than re-deciding it.

- **Emission needs no environment.** Setup: the ADR's emission constraint. Action: read it. Expected: producing the artifact and computing the fingerprints requires no credentials, no network, and no database — the constraint that lets C20 run in CI on every pull request, consistent with C7's pure-assembly rule.

- **Internal-logic limitation is documented at point of use.** Setup: the ADR. Action: read the documentation-placement decision. Expected: the "internal logic change does not move the fingerprint" limitation is placed on the resume verb (C27) and the structure-assertion API (C28), the hand-maintained task version marker is named as the honest manual answer, and it is *not* left buried only in the ADR.

- **Downstream hand-off is explicit.** Setup: the ADR's hand-off section. Action: read it. Expected: it states per blocked ticket what each consumes — T13 the trait/derive and uniqueness rule, T40 the field list and byte-identity/canonicalization rules, T41 the two field lists plus ordering/exclusion/properties, T58 the recorded hashes, T61 the structural fingerprint to diff — so no downstream ticket re-derives the composition.

- **ADR is committed in the accepted state.** Setup: the repository. Action: locate the ADR file and read its status. Expected: the ADR exists at its recorded in-repo path, is in an accepted (not draft/proposed) status, and is the single source the blocked tickets reference.

## Definition of done
- [ ] The ADR defines a stable-name trait carried as an author-declared constant on both task and payload (input/output) types, with a one-line derive for the common case, and records that the stored name is the *declared* name of the compiler-enforced type (C20).
- [ ] The ADR records that `std::any::type_name` is unstable across compiler versions and may appear only as an informational debug field — never in either hash and never as identity (C20).
- [ ] The ADR states stable-name well-formedness rules and records that duplicate declared task/payload names fail *assembly* (not compilation), whole-pipeline scoped, naming both declarations (C20, C7).
- [ ] The ADR records the C20 per-node field list — name, group, stable task name, stable input/output type names, execution class, complete effective policy including defaulted values, declared resource requirements, dependency lists — such that every node and edge the running pipeline would use appears in the artifact (C20).
- [ ] The ADR records the C20 per-edge field list — data-vs-ordering edge kind, and for data edges the stable name of the carried type (C20).
- [ ] The ADR fixes the structural-fingerprint field list as exactly the node set (stable names), edge set (carried-type names + edge kind), and trigger rules, and nothing else (C21).
- [ ] The ADR fixes the policy-hash field list as exactly the residual effective-policy values (retries/backoff, timeout, declared cost, execution class, retention, durability), and records that a node with no stated policy hashes identically to one with every default written out (C21, C5).
- [ ] The ADR records that group labels are in neither hash and are not part of node identity, so a group rename changes neither hash and surfaces only in the C28 structure diff (C21, C6).
- [ ] The ADR records that both hashes exclude timestamps, hostnames, compiler/tool versions, generation time, git commit, and lockfile hash, so two builds of unchanged source on different toolchains produce the same structural fingerprint and policy hash (C21, C20).
- [ ] The ADR fixes a deterministic, registration-order-independent canonical ordering of the node and edge sets and a single byte-encoding and hash function, sufficient to make assembling the same pipeline twice produce byte-identical artifacts outside the generation-time field (C20, C21, C7).
- [ ] The ADR records the change matrix: any structural change (node add/remove/rename, rewire, trigger-rule change, carried-type change) moves the structural fingerprint; any policy-only change moves only the policy hash; a group rename moves neither — the matrix T41 implements (C21).
- [ ] The ADR records that each hash carries the T0.10 versioned algorithm identifier, that changing this composition/canonicalization is an algorithm-version bump, and that a version mismatch reads as "cannot compare" distinct from "topology differs" — deferring the versioning policy to T0.10 (C21).
- [ ] The ADR records that both hashes and the algorithm version appear in the graph artifact and in every run artifact (C21).
- [ ] The ADR records that emitting the artifact and computing the fingerprints requires no credentials, no network, and no database, consistent with pure assembly, so it runs in CI in an empty environment (C20, C7).
- [ ] The ADR requires the C21 internal-logic limitation to be documented at the point of use — the resume verb (C27) and the structure-assertion API (C28) — naming the hand-maintained task version marker as the honest manual answer, rather than only in the ADR (C21).
- [ ] The ADR names the per-ticket downstream hand-off (T13, T40, T41, T58, T61) so no blocked ticket re-derives the stable-name contract or the fingerprint composition (task dependency map).
- [ ] The ADR is committed at its recorded in-repo path in an accepted status and is the single source referenced by the blocked tickets.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- Implementing the stable-name trait, its derive macro, or the builder that bakes node identity in — that is T13; this ticket only fixes the contract T13 implements against.
- Emitting the graph artifact, its schema, or the byte-identity comparison harness — that is T40 (and the schema-versioning ADR T4); this ticket fixes the field lists and canonicalization T40 serializes, not the emission code or wire format.
- Computing the fingerprints, the defaulted-values-hash-identical test, the cross-toolchain two-toolchain CI job, or the change/no-change matrix as executable tests — that is T41; this ticket fixes the composition those tests exercise.
- Deciding the fingerprint-algorithm *versioning policy*, the "cannot compare" wording, MSRV, or the additive-only schema-evolution rules — those are owned by T0.10; this ticket consumes the version identifier and defers to it.
- Recording fingerprints into run artifacts, or the resume structural/policy gate itself — those are T58 and the resume verb (C27); this ticket supplies the hashes and the point-of-use documentation requirement, not the gate logic.
- The structure-snapshot assertion API and its diff rendering — that is T61 (C28); this ticket supplies the structural fingerprint it diffs against.
- Assembly validation beyond name uniqueness — the empty-pipeline check, execution-class-override validation, the durable-without-contract check, the zero-consumer warning, and precomputation — that is T14; this ticket fixes only the stable-name uniqueness rule assembly enforces.
- Any temptation to make node identity depend on task internal logic via automatic content hashing, or to make the field split, canonicalization, or hash set runtime-configurable — the composition is fixed by construction and the graph shape never changes at runtime; dagr is not a metadata store, a DSL, or a scheduler, and the honest answer to node-level change detection is a manual task version marker, never an automatic under-detecting hash.
