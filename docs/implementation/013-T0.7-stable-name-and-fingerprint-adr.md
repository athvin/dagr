# 013 · T0.7 — ADR: stable-name trait and fingerprint composition

> **Milestone:** M0 · **Size:** S · **Type:** decision · **Components:** C20, C21
> **Branch:** `adr/t0.7-stable-name-and-fingerprint-adr` · **Depends on:** — · **Blocks:** T13, T40, T41, T58, T61

## Why / context
Node identity and pipeline-shape identity are baked in at the very first builder ticket (T13), long before the graph artifact (C20, T40) and the two fingerprints (C21, T41) actually ship in M3. If T13 wires nodes against a placeholder notion of "name" and the real stable-name trait arrives later, every node identity, every artifact, and every fingerprint has to be re-derived — so the trait, the field lists that each hash covers, the canonicalization rules, and what is deliberately excluded must all be decided now, in M0. This ADR is governed by arch.md **C20 · Graph artifact** and **C21 · Graph fingerprint**, and it sits downstream of the algorithm-versioning and "cannot compare" semantics already fixed in T0.10 — this ticket designs the composition and canonicalization that those version identifiers wrap, it does not re-decide the versioning story. It locks: the author-declared stable-name trait/derive for task and payload types; the exact split of fields between the structural fingerprint and the policy hash; the canonical ordering and encoding both hashes feed on; and the rule that group labels and environmental values are in neither hash. Get this wrong and T13, T40, T41, T58, and T61 all disagree about what a node *is* and whether two builds are the same pipeline.

## Objective
Produce a committed ADR that fixes the stable-name contract and the composition of both fingerprints precisely enough that T13 can bake node identity in, and T40/T41 can emit the artifact and hashes without re-deriving anything. Concretely, the ADR must decide and record:

- **Stable-name trait/derive.** A trait carrying an author-declared stable name as a constant, implemented by both task types and payload (input/output) types, with a derive that makes the common case one line. Record that the name is the *declared* name the author writes — never `std::any::type_name`, which is explicitly unstable across compiler versions and may appear only as an informational debug field, never as identity and never in either hash. State the well-formedness rules for a stable name (non-empty, a fixed character set / shape) and that uniqueness of declared task and payload names is a *whole-pipeline* assembly check, not a compile-time one, with the duplicate error naming both declarations.
- **What C20 records per node and per edge.** The field list the graph artifact carries for every node — name, group, stable task name, stable input and output type names, execution class, complete effective policy (including defaulted values written out), declared resource requirements, dependency lists — and for every edge whether it carries data or only ordering, plus for data edges the stable name of the carried type. This is the raw material both hashes are computed from; the ADR fixes which of these fields feed which hash.
- **Structural-fingerprint field list.** Exactly the shape-determining inputs: the node set (by stable name), the edge set (with carried-type stable names and the data-vs-ordering edge kind), and the per-node trigger rule. These are "the things that determine whether a prior run's outputs mean anything to this binary" — the resume gate (C27).
- **Policy-hash field list.** The remaining effective-policy values — retry count/backoff, per-attempt timeout, declared resource cost, execution class, retention flag, durability flag — and the rule that a node with no stated policy hashes *identically* to one with every default written out (C5). A policy divergence is a proceed-with-diff at resume, never a refusal.
- **Exclusions.** Group labels are in *neither* hash (C6) and are not part of node identity; a group rename shows up only in the structure diff (C28), never breaks resume. Both hashes exclude timestamps, hostnames, compiler/tool versions, generation time, git commit, lockfile hash, and everything else environmental, so two builds of unchanged source on different toolchains hash identically.
- **Canonicalization.** The single canonical ordering and byte-encoding both hashes and the artifact's byte-identity comparison rely on: how the node set and edge set are deterministically ordered (independent of registration order), how each field is serialized into the hash input, and the hash function itself. This is what makes "assemble twice → byte-identical artifact (generation-time field aside)" and "two toolchains → same fingerprint" achievable, and it is the input over which the T0.10 algorithm-version identifier is defined.
- **Algorithm-version binding.** Record that each hash carries the versioned algorithm identifier fixed in T0.10, that changing this canonicalization or field split is an algorithm-version bump, and that a version mismatch reads as "cannot compare" (distinct from "topology differs") — pointing at T0.10 as the owner of that decision rather than re-deciding it.
- **Point-of-use documentation.** Record the C21 internal-logic limitation — changing a task's internal logic without changing its interface does *not* change the fingerprint — and require it be documented where developers meet it: on the resume verb (C27) and the structure-assertion API (C28), with the hand-maintained task version marker named as the honest manual answer. Not buried in the ADR.
- **Downstream hand-off.** State, per blocked ticket, exactly what each consumes: T13 the stable-name trait/derive and the whole-pipeline uniqueness rule for node identity; T40 the per-node/per-edge C20 field list and the byte-identity/canonicalization rules; T41 the two field lists, canonical ordering, group exclusion, and the defaulted-values-hash-identical and cross-toolchain properties; T58 the fingerprint hashes it records into run artifacts; T61 the structural fingerprint the structure-snapshot assertion diffs against.

## Decision

### 1. The stable-name trait — an author-declared constant on task and payload types

Node and pipeline-shape identity rest on **author-declared stable names**, not on
any compiler-derived string. The contract:

- A **stable-name trait** carries the name as an **associated constant**. It is
  implemented by **both task types and payload (input/output) value types** — the
  types a task consumes and produces. The stored name is the **declared name the
  author writes** for the compiler-enforced type, so the recorded name always
  matches a real, compiler-checked type.
- A **derive makes the common case one line.** The derive, absent an explicit
  override, uses the type's own declared identifier as the stable name, so a task
  or payload type opts into a stable identity by adding one attribute and writing
  nothing else. An author who needs the recorded name to survive a *Rust* rename
  (or to disambiguate two types that share a short name) writes the constant
  explicitly; that explicit name is then the stable identity and a later Rust
  rename does not move it.
- **`std::any::type_name` is never identity and never in either hash.** It is
  explicitly **unstable across compiler versions** (its output format is not
  guaranteed stable across toolchains), so it may appear **only as an
  informational debug field** in the artifact — never as identity, never in the
  structural fingerprint, and never in the policy hash. Recorded identity names
  are therefore the author-declared stable names, which are byte-stable across
  toolchains by construction.

Illustrative shape (not shipping code; T13 owns the real definition and derive):

```rust
/// An author-declared, toolchain-stable name. Implemented by task types and by
/// payload (input/output) value types. Never std::any::type_name.
pub trait StableName {
    /// The author-declared stable name. Defaults (via derive) to the type's own
    /// declared identifier; an explicit override survives a Rust rename.
    const STABLE_NAME: &'static str;
}

// The common case is one line:
#[derive(StableName)]
struct RowCount(u64);   // STABLE_NAME == "RowCount"
```

**Well-formedness.** A stable name is **non-empty** and drawn from a **fixed
character set / shape** — a documented restriction (ASCII letters, digits, and a
small punctuation set such as `_`, `-`, `.`, and `::` for namespacing) with no
whitespace and no control characters, so it round-trips through the artifact
encoding (T4), the run-store path segment where it is the `<pipeline>` component
(T0.6), and the DOT/Mermaid renderers (T46) without escaping surprises. A
malformed stable name is an **assembly** failure (below), not a silent
truncation.

**Uniqueness is a whole-pipeline assembly check, not a compile-time one.** Two
distinct types can carry the same declared stable name, and the compiler cannot
see across the whole pipeline to notice; uniqueness is therefore checked at
**assembly** (C7), over the **whole pipeline**, not scoped to a group. Duplicate
declared **task** names and duplicate declared **payload** names each **fail
assembly**, and the error **names both declarations** (which two types, at which
registration/declaration sites) so the fix is unambiguous — the same
name-both-declarations discipline C7 already applies to duplicate node names.
This is an assembly failure, never a compile error and never a runtime surprise.

### 2. What C20 records — per node and per edge

The graph artifact is the raw material both hashes are computed from. It records,
**for every node**:

| Field | Note |
|---|---|
| **name** | the explicit registration node name (node identity; C2/C7) |
| **group** | the presentation label, or none (C6) — recorded, but in **neither** hash (§5) |
| **stable task name** | the author-declared `STABLE_NAME` of the task type (§1) |
| **stable input type name(s)** | the author-declared `STABLE_NAME` of each consumed payload type |
| **stable output type name** | the author-declared `STABLE_NAME` of the produced payload type |
| **execution class** | await-bound / blocking / compute (the *effective* class after any C5 override) |
| **complete effective policy** | every C5 field, **including defaulted values written out** — retry count/backoff, per-attempt timeout, declared resource cost vector, trigger rule, execution-class override, group, retention flag, durability flag |
| **declared resource requirements** | the resource types the node declares it needs (C9) |
| **dependency lists** | the node's upstreams, split into data dependencies and ordering dependencies |

and **for every edge**:

| Field | Note |
|---|---|
| **edge kind** | whether it carries **data** or **only ordering** (C3 vs C4) |
| **carried type stable name** | for a **data** edge, the author-declared `STABLE_NAME` of the value type carried; an ordering edge carries no value and no carried type |

Every node and edge the running pipeline would use appears in the artifact; the
build-provenance header fields (tool version, git commit, lockfile hash) are part
of the C20 header but their **emission is deferred to T40** (see §7 and
Consequences) — they are excluded from both hashes regardless (§5).

### 3. The structural-fingerprint field list — exactly the shape inputs

The **structural fingerprint** covers **exactly** the shape-determining inputs
and **nothing else**:

1. **The node set, by stable name** — each node contributes its identity (node
   name) and its stable task and payload (input/output) type names.
2. **The edge set** — each edge contributes its **kind** (data vs ordering) and,
   for a data edge, the **stable name of the carried type**.
3. **The per-node trigger rule** — one of the closed T0.4 set (`all-succeeded`,
   `all-terminal`, `any-failed`).

These are "the things that determine whether a prior run's outputs mean anything
to this binary" — the resume gate (C27). **No policy value, no group label, and
no environmental value is in the structural fingerprint.** This is what makes
resume's structural gate mean *topology*, not *tuning*.

### 4. The policy-hash field list — exactly the residual effective policy

The **policy hash** covers **exactly** the remaining effective-policy values —
the ones **not** in the structural fingerprint:

- **retry count and backoff shape**
- **per-attempt timeout**
- **declared resource cost** (the per-pool cost vector)
- **execution class** (the effective class after any override)
- **retention flag** (whether the output is retained after consumers finish)
- **durability flag** (whether the output is durable)

and **no shape input** (no node set, no edge set, no trigger rule). Trigger rule
is in the *structural* fingerprint, not here, because it is shape (it decides
whether a node runs at all), not tuning.

**Defaulted policy hashes identically to written-out defaults (C5).** A node with
**no stated policy** produces **exactly the same policy-hash input** as a node
with **every default written out explicitly** — the hash is computed over the
*effective* policy (defaults resolved), so the two are byte-for-byte identical
and therefore hash identically. This is a **required property** T41 will assert
with a named test (`defaulted_policy_hashes_identically_to_written_out_defaults`).
A policy divergence at resume is a **proceed-with-diff**, never a refusal (C21,
C27): raising a timeout to resume an expensive half-finished run is the
*motivating* case for resume.

### 5. Exclusions — group labels and everything environmental

- **Group labels are in NEITHER hash and not part of node identity (C6).** A
  group is presentation metadata: it feeds **artifact organization and diagram
  clustering only**. A group rename (or removal) changes **neither** the
  structural fingerprint, **nor** the policy hash, **nor** node identity; it
  surfaces **solely** in the C28 structure diff (review-visible) and **never**
  breaks resume.
- **Both hashes exclude everything environmental.** Neither hash includes
  **timestamps, hostnames, compiler/tool versions, generation time, git commit,
  lockfile hash**, or anything else about the machine or the build. Two builds of
  **unchanged source** — on different machines, with different toolchains —
  therefore produce the **same structural fingerprint and the same policy hash**.
  This is testable precisely because every hashed input is author-declared (the
  stable names of §1), and it is the property T41's cross-toolchain (two
  toolchains in CI) job asserts.

### 6. Canonicalization — deterministic, registration-order-independent

Both hashes and the artifact's byte-identity comparison rely on a **single
canonical ordering and a single byte-encoding**, so that assembling the same
pipeline twice yields byte-identical artifacts and two toolchains yield the same
fingerprint:

- **Ordering is deterministic and independent of registration order.** The node
  set is ordered by a **total, stable key** derived from author-declared data —
  the node **name** (unique across the pipeline, C7) is the primary key. The edge
  set is ordered by the **(source node name, destination node name, edge kind)**
  tuple, again a total order over author-declared data. Because both orders are
  functions of names and kinds — never of the order registrations happened to be
  added — two assemblies of the same pipeline in different registration orders
  produce **identical** ordered inputs. (Node identity being name-derived and
  reorder-stable is already fixed by C2 (arch.md); this ADR reuses it as the
  sort key.)
- **A single fixed byte-encoding per field.** Each field is serialized into the
  hash input by a **fixed, length-prefixed (or otherwise unambiguously delimited)
  canonical encoding** — no map/set iteration order leaks in (sets are emitted in
  the canonical order above), no locale, no floating-point ambiguity (the cost
  vector's units are integers — bytes and thread counts, per T0.5). The encoding
  is chosen so that two distinct field structures can never serialize to the same
  bytes (unambiguous framing), which is what gives collision-resistance its
  footing.
- **A single named hash function.** Both hashes use **one** cryptographic hash
  function, named once here: **BLAKE3** — fast, modern, fixed 256-bit output,
  with no external C dependency (a pure-Rust implementation), which keeps the
  core crate's dependency set minimal (Stability, supply chain) and keeps
  emission dependency-light enough to run in CI. The *choice* of function is part
  of the versioned algorithm (§7): naming BLAKE3 here fixes v1; a later change to
  the function is an algorithm-version bump, not a silent swap. Collision
  resistance is the hash function's (BLAKE3's 256-bit output makes an accidental
  collision between two distinct canonical encodings vanishingly unlikely);
  determinism is the canonicalization's (identical author-declared inputs →
  identical bytes → identical digest).

Together these make **"assemble twice → byte-identical artifact (generation-time
field aside)"** (C7/C20) and **"two toolchains → same fingerprint"** (C21)
achievable, and the canonical byte input is exactly the surface over which the
T0.10 algorithm-version identifier is defined.

### 7. Algorithm-version binding — deferred to T0.10

Each hash **carries the versioned algorithm identifier fixed in T0.10**. This ADR
**consumes** that identifier and does **not** re-decide the versioning policy:

- **Changing this canonicalization or this field split is an algorithm-version
  bump.** Reordering the canonical sort, changing the byte-encoding, changing the
  hash function (§6), or moving a field between the two hashes is a **version
  bump** — it changes what the digest means, so the identifier that stamps the
  digest must change with it.
- **A version mismatch reads as "cannot compare," distinct from "topology
  differs."** When two fingerprints were computed by **different** algorithm
  versions, they are **not comparable** — resume reports **"cannot compare,"** a
  *distinct* outcome from a genuine **topology difference** (same algorithm, and
  the structural fingerprints simply differ). **T0.10 owns** the version-identifier
  scheme and the "cannot compare" wording and precedence; this ADR points at it
  rather than restating it.
- **Both hashes and the algorithm version appear in the graph artifact and in
  every run artifact.** The C20 graph artifact header carries the structural
  fingerprint, the policy hash, and the algorithm version; every C22 run artifact
  header carries the same three, so a run artifact can be matched to a graph
  artifact from the same build and resume can perform the version-comparability
  check (C27) before any topology comparison.

**Build-provenance coverage is deferred to T40.** The C20 build-provenance header
(tool version, git commit, lockfile hash, embedded at build time) is **not**
emitted or asserted here; its emission and its covering test are **T40's**, and
`docs/coverage-matrix.md` already folds that obligation into C20's row
(`Covered-by T40`). This ADR only fixes that those provenance fields are
**excluded from both hashes** (§5) — it does not fill the provenance seam, which
would contradict the coverage matrix.

### 8. Emission needs no environment (C7 pure-assembly)

Producing the graph artifact and computing both fingerprints requires **no
credentials, no network, and no database** — every hashed input is
author-declared and available from pure assembly (C7). This is the constraint
that lets C20 run in continuous integration on **every pull request**, in an
empty environment with no configuration present, consistent with C7's
pure-assembly rule and with T0.5's assembly→bootstrap seam (assembly sees no
parameter values and touches no machine).

### 9. The internal-logic limitation — documented at the point of use

**Changing a task's *internal logic* without changing its interface does NOT
change the fingerprint.** The fingerprint is composed from author-declared names,
edges, trigger rules, and policy (§§3–4) — never from the task's function body —
so a task whose stable name, input/output types, edges, trigger rule, and policy
are unchanged hashes identically even if its body was rewritten. This is a **real
limitation with no cheap fix in a compiled language**, and this ADR **requires it
be documented where developers meet it, not buried here**:

- **On the resume verb (C27)** — a resumed run may skip a node whose body changed
  but whose interface did not; the resume verb's documentation states this and
  points at the manual answer.
- **On the structure-assertion API (C28)** — a structure snapshot may pass across
  a body-only change; the structure-assertion API's documentation states the same
  limitation.

The **honest manual answer** — named at both points of use — is a
**hand-maintained task version marker**: an author who genuinely needs node-level
change detection bumps an explicit version constant on the task (visible,
reviewable, obviously manual), which *is* part of the task's declared interface
and therefore *does* move the structural fingerprint. This is deliberately not an
automatic content hash of the body (see **Rejected alternatives**): an automatic
hash silently under-detects across a compiled language's opaque boundaries and
lies about what it covers.

### 10. Scope boundary (permanent non-goals)

Everything above describes a **fixed-by-construction** identity and fingerprint
contract: author-declared names, a fixed two-way field split, one canonical
ordering, one byte-encoding, one hash function. It is **never
runtime-configurable** — the field split, the canonicalization, and the hash set
do not change at runtime, because **the graph's shape never changes at runtime**
and dagr is permanently **not a metadata store, a DSL, or a scheduler**. Node
identity **never** depends on task internal logic via automatic content hashing;
the honest answer to node-level change detection is the manual task version
marker (§9), never an automatic under-detecting hash. The **Rejected
alternatives** below name each temptation that would breach this boundary.

## Consequences

**Each blocked ticket inherits a named seam and reopens no question this ADR
closed:**

- **T13** (023 — flow builder and node identity, C7) binds against **the
  stable-name trait/derive (§1) and the whole-pipeline uniqueness rule (§1)**: it
  implements the trait and its one-line derive, bakes node identity in as the
  explicit registration name, and enforces duplicate-declared-name uniqueness at
  assembly naming both declarations. It uses the name-derived, reorder-stable sort
  key (§6) as the basis for the precomputed fingerprint *slot* (the computation
  itself is T41).
- **T40** (051 — graph artifact emission, C20) binds against **the per-node and
  per-edge field lists (§2) and the byte-identity / canonicalization rules
  (§6)**: it serializes exactly those fields under the canonical ordering and
  byte-encoding, emits the versioned header (with the build provenance whose
  emission this ADR defers to it, §7), and produces byte-identical artifacts
  outside the generation-time field in an empty environment.
- **T41** (052 — fingerprints, C21) binds against **the two field lists (§§3–4),
  the canonical ordering and group exclusion (§§5–6), and the
  defaulted-values-hash-identical and cross-toolchain properties (§§4–5)**: it
  computes the structural fingerprint and policy hash over exactly those inputs,
  stamps them with the T0.10 algorithm version (§7), and implements the
  change/no-change matrix (below), the defaulted-hashes-identically test, and the
  two-toolchain CI job.
- **T58** (070 — resume core, C27) binds against **the two hashes it records into
  run artifacts (§7)**: it records the structural fingerprint, policy hash, and
  algorithm version into every run artifact, gates resume on the **structural**
  fingerprint (printing a structural diff on mismatch), proceeds-with-diff on a
  **policy** divergence, and performs the T0.10 version-comparability ("cannot
  compare") check before any topology comparison.
- **T61** (074 — structure snapshot testing, C28) binds against **the structural
  fingerprint the structure-snapshot assertion diffs against (§3)** and the
  canonical, stably-ordered serialization (§6): its semantic comparison over node
  set, edge set, and effective policies rests on this canonicalization, and a
  group rename is review-visible here (C6) even though it never moves the
  fingerprint (§5).

**The change / no-change matrix (the matrix T41 implements).** Any **structural
change** (node add/remove/rename, rewire, trigger-rule change, carried-type
change) **moves the structural fingerprint**; any **policy-only change moves only
the policy hash**; a **group rename moves neither**:

| Change | Structural fingerprint | Policy hash |
|---|---|---|
| node add / remove / rename | **moves** | — |
| edge rewire (add/remove/redirect) | **moves** | — |
| trigger-rule change | **moves** | — |
| carried-type (data-edge type) change | **moves** | — |
| retries / backoff change | — | **moves** |
| per-attempt timeout change | — | **moves** |
| declared cost change | — | **moves** |
| execution-class change | — | **moves** |
| retention flag change | — | **moves** |
| durability flag change | — | **moves** |
| **group rename / removal** | — | — |
| task internal-logic change (interface unchanged) | — | — |
| different machine / toolchain (source unchanged) | — | — |

(A defaulted policy value equals its written-out default, §4, so writing a
default out explicitly is **not** a change and moves neither hash.)

**Coverage matrix: no change.** C20 and C21 remain as they stand in
`docs/coverage-matrix.md`: C20 is `machine`/`unmapped` deferred to T40 (with the
build-provenance obligation folded into its row, Covered-by T40), and C21 is
**`human`** (its governing class is documentation-at-point-of-use, criterion 8),
with its mechanical sub-criteria machine-tested under T41. A decision ticket owes
no covering test; the covering tests land with T40 (C20) and T41 (C21), which
will each edit the matrix per its per-ticket duty. This ADR makes **no edit** to
the coverage matrix and **no edit** to the criteria-matrix partition (T0.10's),
and it **agrees** with both — in particular it does not fill T40's
build-provenance seam and does not treat C21 as machine-owned here.

**Reopen condition.** If a downstream ticket cannot honor a seam as written — for
example, if the associated-constant trait cannot be implemented for a payload
type the design requires (an orphan-rule or coherence obstruction the derive
cannot route around), if the name-derived canonical ordering proves not total
over a legal pipeline, or if BLAKE3 proves unavailable under the pinned MSRV or
the supply-chain policy — **the contract reopens here**, in this ADR, rather than
being worked around locally. A local workaround that silently diverges from this
contract (a second identity scheme, a runtime-configurable split, an automatic
content hash) is a defect, not a fix.

## Rejected alternatives

- **`std::any::type_name` as identity** (derive the stable name from
  `type_name::<T>()` instead of an author-declared constant). **Rejected:**
  `type_name`'s output is **explicitly unstable across compiler versions** — its
  format is not a stability guarantee — so two toolchains would produce different
  identities and different fingerprints for unchanged source, breaking the
  cross-toolchain property (§5) and C21's whole premise. `type_name` is admitted
  **only** as an informational debug field (§1), never as identity and never in
  either hash.
- **An automatic content hash of task internal logic** (hash the task's function
  body / compiled bytes so a body change moves the fingerprint). **Rejected on
  the scope boundary and on honesty:** in a compiled language there is **no cheap,
  reliable way** to hash "the logic" — inlining, monomorphization, dependency
  bumps, and optimization levels all perturb the bytes without a semantic change,
  and a source hash misses behavioral changes in called libraries. Such a hash
  **silently under-detects and lies about what it covers**. The honest answer is a
  **hand-maintained task version marker** (§9): visible, reviewable, and obviously
  manual. Node identity **never** depends on task internal logic via automatic
  content hashing.
- **A single combined fingerprint** (one hash over topology *and* policy).
  **Rejected:** resume must gate on **topology** and **proceed-with-diff** on
  **policy** (C21, C27) — raising a timeout to resume an expensive half-finished
  run is the *motivating* case, and a single hash would refuse it as a spurious
  "topology" mismatch. The **two-hash split** (§§3–4) is exactly what lets the
  structural gate mean shape while a policy change diffs-and-proceeds.
- **A runtime-configurable field split, canonicalization, or hash set** (let the
  operator or the pipeline choose what feeds which hash, or swap the ordering/hash
  at runtime). **Rejected on the scope boundary:** the composition is **fixed by
  construction** and **never runtime-configurable** (§10) — the graph's shape
  never changes at runtime, and a configurable fingerprint is a metadata-store /
  DSL feature dagr will never grow. Changing the composition is an
  **algorithm-version bump** (§7), a deliberate, reviewed, versioned act — not a
  runtime knob.
- **A per-registration-order or map-iteration-order serialization** (hash the node
  and edge sets in whatever order they were registered or a hash map yields).
  **Rejected:** it would make the fingerprint depend on **registration order** and
  on **nondeterministic iteration order**, breaking C2's reorder-stability and
  C7's assemble-twice-byte-identical guarantee. The canonical ordering (§6) is a
  **total function of author-declared names and kinds**, independent of
  registration and iteration order, which is what makes the determinism testable.

*(Reopen condition stated in §Consequences: if a downstream ticket cannot honor a
seam as written, the contract reopens here rather than being worked around
locally.)*
