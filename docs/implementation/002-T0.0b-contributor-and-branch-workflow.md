# 002 · T0.0b — Contributor guide and branch-per-ticket workflow

> **Milestone:** M0 · **Size:** S · **Type:** setup · **Components:** system-level (Operational model, Stability)
> **Branch:** `chore/t0.0b-contributor-and-branch-workflow` · **Depends on:** T0.0a · **Blocks:** T7

## Why / context
Every subsequent dagr ticket assumes a shared process contract: one branch and one PR per implementation ticket, tests written first, and a fixed set of CI checks that must be green before merge. This ticket writes that contract down as version-controlled repository documents so the rule is enforced socially and mechanically, not by folklore. It builds directly on the T0.0a repo hygiene scaffolding and lands the human-readable half of the process that T7 will later automate with GitHub Actions and the acceptance-criteria coverage matrix. It is governed by the **Operational model**, **Stability**, and **Documentation** sections of arch.md — specifically the commitments that the release checklist and the criteria matrix are "version-controlled and reviewed like code," that supply-chain checks (`cargo audit`, `cargo deny`) run in CI, and that documentation deliverables are held to in CI where a machine can check them.

## Objective
Establish the written process contract that governs how every dagr ticket ships. Concretely:
- Author a `CONTRIBUTING.md` at the repo root describing the branch-per-ticket workflow: exactly one branch per implementation ticket, with the branch name taken verbatim from the ticket header's `Branch` field, and exactly one PR per branch.
- State the strict tests-first / TDD expectation as a hard rule: the Test plan and its failing tests land (or are demonstrably written first) before implementation, and every PR shows tests exercising the acceptance criteria it claims to satisfy.
- Enumerate the exact set of CI checks that must be green before any merge, matching arch.md's stated posture: `cargo fmt --check`, `cargo clippy` with warnings denied, the test suite, the rustdoc lint, and `cargo audit` / `cargo deny` where configured. Name these as the merge gate and reference that T7 wires them into GitHub Actions.
- Add a pull-request template under `.github/` that requires each PR to link its ticket by `tid` and file path, restate the acceptance criteria being satisfied, and carry a checklist mirroring the ticket's Definition of done (tests-first confirmed, CI checks named, docs/rustdoc updated).
- Add a `CODEOWNERS` file (in `.github/` or repo root) assigning review ownership so every PR requires review before merge.
- Cross-link these documents from the top-level `README` so a new contributor can find the workflow from the front door.
- Keep every document consistent with the permanent scope boundary and with the milestone/component vocabulary already used in `docs/tasks.md` and `docs/arch.md`.

## Definition of done

- [ ] `CONTRIBUTING.md` documents the one-PR-per-ticket rule and requires each PR to link its ticket by `tid` and by `docs/implementation/` path.

- [ ] A `CODEOWNERS` file exists at a GitHub-honoured location and assigns review ownership such that every PR requires review before merge (satisfying the arch.md commitment that the criteria matrix and release checklist are reviewed like code, and that core-crate dependency additions are reviewed as API decisions).

- [ ] Internal links in the new documents resolve and files sit at their committed paths.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Out of scope
- Implementing the GitHub Actions workflows, the acceptance-criteria coverage matrix, or the CI script that fails on unmapped machine criteria — that is **T7**. This ticket only writes the human-readable contract those automations enforce; it does not add `.github/workflows/` pipelines.
- The compile-failure / UI-test harness and pinned-toolchain policy for UI tests (**T8**) beyond naming that such checks join the merge gate when they exist.
- The repository hygiene scaffolding delivered by **T0.0a** (dependency of this ticket); do not re-create it here.
- Any crate layout, `Cargo` workspace, or code (**T1** and later) — this ticket ships process documents only.
- Branch-protection rules configured in the GitHub UI/API (an operator action outside the repo), and any bot or merge-queue automation.
- Any temptation to encode runtime graph behaviour, scheduling policy, distributed coordination, or a metadata service into contributor process — explicitly forbidden by the permanent scope boundary.
