# 053 · T42 — C22: event-stream folding into run artifact

> **Milestone:** M3 · **Size:** M · **Type:** feature · **Components:** C22
> **Branch:** `feat/t42-event-stream-folding` · **Depends on:** T19, T31, T39 · **Blocks:** T43, T44, T47, T48, T55, T57, T68

## Why / context
The run artifact (C22) is the outcome of one execution, joinable to the structure — and it is *derived*, never authored: every field comes from folding the append-only event stream (C19). This ticket builds the standalone fold function that turns a stream — whole or truncated by a crash — into a validated run artifact, with no access to the original run's live state (C19 says the fold "needs no access to the original run"). It builds on the event-stream writer and its vocabulary (T19), the permit/pool accounting that produces cost and zombie-pinning signals (T31), and the published artifact schemas (T39). It is the foundation for the summary/critical-path enrichment (T43), node metrics surfacing (T44), run-overlay rendering (T47), artifact-compatibility CI (T48), the CLI `fold` verb wiring (T55), durable-reference recording (T57), and the crashed-run finalize path (T68). The governing behaviour and acceptance criteria are in arch.md §"C22 · Run artifact" and §"C19 · Event stream", with supporting semantics in §"C10 · Output slot" (slot residency, zombie-pinned memory) and the terminal-state taxonomy in §Vocabulary.

## Objective
Build a pure, standalone folding function (and its supporting artifact types) that consumes an event stream and produces a run artifact, plus the fixture corpus and validation harness that keep it honest.

Concrete pieces of work:
- A fold entry point that takes a decoded (or decodable) event stream and returns a run artifact value, taking no run store, no live graph, and no network — callable both at natural run end and by a later invocation over a crashed run's stream.
- Stream ingestion that tolerates and discards at most one trailing partial record (per C19) and treats such a stream as truncated/interrupted.
- **Header** assembly from the run-started event: run identity, pipeline identity, both fingerprint hashes (structural + policy, C21), invocation parameters and data interval, resume lineage (immediate parent and lineage-root run identities when present), and the allowlisted captured environment values only.
- **Body** assembly: one record per *attempt* (not per node), each carrying node identity, attempt number, terminal status from the normative taxonomy, phase-broken elapsed time computed from monotonic offsets (phases sum exactly to the attempt total), which worker ran it, a message, structured error detail, the metrics the task reported, declared-versus-measured cost, and the durable-output reference when one exists. Satisfied-from-prior nodes carry their originating run identity.
- **Node coverage**: every node in the graph appears at least once — including never-ran nodes carrying their propagated terminal state (`upstream-failed`, `upstream-skipped`, originated `skipped`).
- **Summary**: total elapsed time, critical-path time, peak measured slot residency (C10), values still retained at run end, and the time *and* capacity pinned by abandoned-but-running (zombie) work, derived from zombie-at-exit and terminal-state events. (This ticket wires the summary fields the stream already supplies; the critical-path/structure-vs-resource enrichment and any ADR on wait-time-on-path belong to T43.)
- **Outcome variants**: full-run outcome; `interrupted` marking for a crash-truncated stream; and the pre-execution variants `assembly-failed` and `bootstrap-failed` (distinct — graph's fault vs. machine's), each with the complete error list, zero attempts, and the fingerprint present only when assembly succeeded.
- **Schema discipline**: the fold declares which stream schema versions it reads; the emitted artifact validates against its published schema (T39); missing fields default and unknown fields are ignored (additive-only within a version).
- Resolve the two open questions below and record the decision in the fold's rustdoc and in the artifact's phase/worker field definitions.

## Test plan (write these first — TDD)
Each scenario is independent and checks an observable output of the fold function or the produced artifact.

- **Phases sum exactly.** Setup: a hand-built stream for one attempt with node-ready, admitted, attempt-started, and attempt-succeeded events carrying known monotonic offsets. Action: fold it. Expected: the attempt record's named phase durations sum bit-exactly to the attempt's total elapsed, and the total equals the offset delta between attempt start and terminal event (no wall-clock arithmetic anywhere).
- **One record per attempt, not per node.** Setup: a stream where a node fails twice then succeeds on attempt 3 (three attempt-started/attempt-outcome pairs, three distinct backoff intervals). Action: fold. Expected: three attempt records for that node, with attempt numbers 1, 2, 3 and the correct per-attempt terminal statuses (two `failed`, one `succeeded`), not a single collapsed record.
- **Never-ran nodes covered.** Setup: a stream for a graph whose node B never ran because upstream A failed, and node C was skipped-propagated from an originated skip at node S. Action: fold with the graph's node set available. Expected: the artifact contains B with `upstream-failed`, C with `upstream-skipped` carrying S's identity, and S with originated `skipped` — every graph node appears at least once.
- **Structured error and message preserved.** Setup: a stream with a failed attempt carrying a structured error detail and a human message. Action: fold. Expected: the attempt record reproduces both the message and the structured error detail unmodified.
- **Metrics reach the artifact unmodified.** Setup: an attempt-outcome event carrying a set of task-reported metrics plus framework metrics. Action: fold. Expected: the attempt record's metrics equal the input metrics exactly (byte/value-identical), with no reordering-induced change to values.
- **Declared-vs-measured cost juxtaposed.** Setup: a stream whose attempt records a declared cost vector and a measured cost. Action: fold. Expected: both the declared and the measured cost appear on the attempt record, side by side.
- **Durable reference recorded.** Setup: a succeeded attempt whose node is durable and whose event carries a durable-output reference. Action: fold. Expected: the reference appears on that attempt record. (A non-durable node's record has no reference.)
- **Satisfied-from-prior carries originating run id.** Setup: a stream containing a `satisfied-from-prior` node with an originating run identity. Action: fold. Expected: the node's record carries `satisfied-from-prior` and the originating run identity.
- **Allowlist positive.** Setup: a run-started event whose header includes allowlisted environment values. Action: fold. Expected: exactly those allowlisted values appear in the artifact header.
- **Allowlist negative (planted sentinel).** Setup: a run-started event / stream carrying an environment value NOT on the declared allowlist (a planted sentinel string). Action: fold, then search the *entire* serialized artifact for the sentinel. Expected: the sentinel appears nowhere in the artifact — no environment value outside the declared allowlist survives the fold.
- **Header complete from run-started alone.** Setup: a stream containing only the run-started event and one following event. Action: fold. Expected: the header (identity, pipeline identity, both fingerprints, parameters, data interval, resume lineage, allowlisted env) is fully populated from the run-started event; only overall outcome and summary reflect the truncation.
- **Interrupted marking on truncation.** Setup: a stream that ends mid-run with no run-finished event, plus a trailing byte-truncated partial record. Action: fold. Expected: the fold discards exactly the one trailing partial, marks the artifact `interrupted`, and includes every attempt and transition up to the crash; no error is raised for the single partial.
- **Trailing-partial tolerance boundary.** Setup: a stream with two corrupt/partial trailing records. Action: fold. Expected: the fold's tolerance is limited to one trailing partial (per C19); a second corruption is surfaced as a fold error rather than silently dropped. (Records the contract boundary.)
- **Assembly-failed variant.** Setup: a stream with run-started then an assembly-failure record and its complete error list, no attempt events. Action: fold. Expected: outcome `assembly-failed`, zero attempt records, the full error list present, and no fingerprint in the header (assembly did not succeed).
- **Bootstrap-failed variant.** Setup: a stream with run-started (fingerprint present, assembly succeeded) then a bootstrap-failure record and its errors, no attempt events. Action: fold. Expected: outcome `bootstrap-failed` (distinct from assembly-failed), zero attempts, full error list, fingerprint present.
- **Summary: retained values.** Setup: a completed-run stream where node R is marked retained and its value is still held at run end while node U's value was released. Action: fold. Expected: the summary identifies R among values retained at run end and does not list U.
- **Summary: peak slot residency.** Setup: a stream whose events record measured slot residency peaking at a known value. Action: fold. Expected: the summary's peak measured slot residency equals that value.
- **Summary: zombie-pinned time and capacity.** Setup: a stream with a timed-out attempt whose thread never returned, a zombie-at-exit event, and the pinned capacity it held (C10/C14). Action: fold. Expected: the summary reports the pinned time and pinned capacity attributable to abandoned-but-running work; the zombie is recorded as an event, and the node's terminal state stays `timed-out` (no second terminal state).
- **Fingerprint joins to graph artifact.** Setup: fold a full-run stream and separately produce a graph artifact from the same build fixture. Action: compare structural fingerprints. Expected: the run artifact's structural fingerprint equals the graph artifact's.
- **No run access required.** Setup: fold a stream that references a run store path that does not exist on disk. Action: fold. Expected: success — the fold reads only the stream bytes it is given and never touches the store, network, or a live graph.
- **Schema version declaration + fixture corpus.** Setup: the checked-in fixture corpus with one artifact per released schema version, and one stream fixture per stream schema version the fold declares it reads. Action: fold each stream fixture and parse each artifact fixture with current tooling; validate every produced artifact against its published schema. Expected: every declared stream version folds successfully, every artifact validates against its schema, and every corpus artifact from a prior schema version remains parseable (unknown fields ignored, missing fields defaulted).
- **Determinism.** Setup: one stream fixture. Action: fold it twice. Expected: identical artifacts (modulo any explicitly non-deterministic field, of which there should be none in the fold's own output).

## Definition of done
- [ ] The fold is a standalone function that produces a run artifact from an event stream with no access to the original run — no run store, no live graph, no network (C19 fold criterion).
- [ ] Every node present in the graph appears at least once in the run artifact, including never-ran nodes with their propagated terminal states (`upstream-failed`, `upstream-skipped`, originated `skipped`).
- [ ] The body carries one record per attempt (not per node); retries appear as distinct attempt records with ascending attempt numbers and their own terminal statuses.
- [ ] Each attempt record carries: node identity, attempt number, terminal status from the normative taxonomy, phase-broken elapsed time, worker identity, message, structured error detail, task-reported metrics, declared-versus-measured cost, and the durable-output reference when present.
- [ ] Phase durations for every attempt sum exactly to that attempt's total, with both derived from monotonic offsets (never wall clocks).
- [ ] Satisfied-from-prior nodes carry their originating run identity in the artifact.
- [ ] The header carries run identity, pipeline identity, both fingerprint hashes, invocation parameters and data interval, resume lineage (immediate parent and lineage-root, when resumed), overall outcome, and only allowlisted captured environment values.
- [ ] No environment value outside the declared allowlist appears anywhere in any produced artifact — verified by a planted-sentinel negative test scanning the full serialized artifact.
- [ ] A crash-truncated stream folds into an artifact marked `interrupted`, containing everything up to the crash, tolerating and discarding at most one trailing partial record (C19); a second corruption is a fold error.
- [ ] An `assembly-failed` stream produces the assembly-failure variant: zero attempts, complete error list, no fingerprint in the header.
- [ ] A `bootstrap-failed` stream produces the bootstrap-failure variant (distinct from assembly-failed): zero attempts, complete error list, fingerprint present.
- [ ] The summary carries total elapsed time, critical-path time, peak measured slot residency (C10), values still retained at run end, and the time and capacity pinned by abandoned-but-running work.
- [ ] The run artifact's structural fingerprint matches a graph artifact produced from the same build.
- [ ] The produced artifact validates against its published schema (T39); the fold declares which stream schema versions it reads; readers ignore unknown fields and default missing ones (additive-only within a version).
- [ ] A checked-in fixture corpus contains one artifact per released schema version and is parsed by current tooling in CI; the CI job is wired so this holds forever after.
- [ ] The canonical phase list (open question 1) and the worker-identity representation (open question 2) are decided, documented in the fold's rustdoc and the artifact field docs, and reflected in the emitted artifact.
- [ ] The fold is deterministic: folding the same stream twice yields identical artifacts.
- [ ] Public items are documented with rustdoc; the module explains the fold's inputs, the "no run access" guarantee, and the schema-version contract.
- [ ] CI is green on the ticket branch (fmt, clippy with warnings denied, tests, rustdoc lint, and cargo-audit/deny where configured).

## Open questions
- Canonical phase list — proposal: `ready-wait`, `permit-wait`, `executing`, `backoff`; confirm during implementation and ensure the chosen set partitions attempt elapsed time so phases sum exactly (C22 §phase criterion). The list must align with the waiting/admission/dispatch/backoff phases the execution core records (C14).
- "Which worker ran an attempt" — thread id, pool name, or both. Decide and document; the artifact field's meaning must be unambiguous for capacity analysis.

## Out of scope
- The critical-path *definition* under retries and permit waits, the structure-limited-vs-resource-limited distinguishing test, and any ADR on whether wait time is on the critical path — that is **T43**; this ticket only assembles the summary fields the stream already provides.
- Defining the node-metrics API, the reserved `dagr.` prefix enforcement, metric caps/truncation, and allocator-attributed peak memory — that is **T44 (C23)**; the fold merely copies whatever metrics the stream recorded, unmodified.
- The durable-output *declaration and recording* mechanism (implementing the reference contract, assembly-time rejection, recording at production) — that is **T57 (C27)**; the fold only surfaces a reference that is already present in the stream.
- The CLI `fold` verb, argument parsing, and exit codes — that is **T55 (C26)**; this ticket delivers the library function the verb will call.
- Killing a live run and driving the crash clause of system criterion 3 end-to-end — that is **T68**; this ticket tests the fold over hand-built and captured truncated streams, not a real kill.
- Rendering artifacts to DOT/Mermaid or overlaying run state — that is **T47 (C24)**.
- Publishing or evolving the schemas themselves — that is **T39**; this ticket consumes the published schemas and adds fixtures against them.
- Anything crossing the permanent scope boundary: the fold reads a stream and produces a static artifact — it is not a metadata store, does not query or coordinate across runs, does not schedule or re-execute, and never mutates graph shape. Cross-run analysis stays "concatenate streams partitioned by run identity" (C19), not a queryable database.
